A senior data engineer's guide to folder layout, naming, ref()/source(), surrogate keys, and mart grain — with a real example tree.

The dbt project structure that scales is three layers: staging (one cleanup model per source table, materialized as views), intermediate (purpose-built models that join and reshape staging data), and marts (wide, business-facing fct_/dim_ tables built on a single, explicit grain). Models reference each other only through ref() and pull raw data only through source() — never hard-coded table names. Get those three layers, their naming conventions, and your surrogate keys right, and the project stays navigable from 20 models to 2,000.

I've watched dbt projects rot the same way twice. A team starts with a models/ folder, drops in customer_final_v2.sql, joins to a raw table by its fully-qualified name, and six months later nobody can answer "where does this number come from?" without reading 400 lines of nested CTEs. The fix is almost never a tool — it's a structure you commit to on day one and refuse to violate.

This is that structure. It's the layered approach dbt Labs documents in How we structure our dbt projects, plus the conventions I've found actually hold up under pressure on real teams, plus honest notes on where the official guidance and the common Kimball-style community practice diverge.

What is the standard dbt project structure?

Three transformation layers, each a folder under models/, each with one job:

Layer              Folder                       Prefix                             What it does                                                         What it must NOT do                            Materialization
----------------   --------------------------   --------------------------------   ------------------------------------------------------------------   --------------------------------------------   ------------------------
**Staging**        `models/staging/<source>/`   `stg_`                             Rename, recast, light cleanup. One model per source table.           No joins, no aggregations, no business logic   `view`
**Intermediate**   `models/intermediate/`       `int_`                             Joins, fan-outs, reusable business logic between staging and marts   Be exposed directly to BI tools                `ephemeral` or `view`
**Marts**          `models/marts/<domain>/`     `fct_` / `dim_` (or entity name)   Wide, denormalized, business-facing entities at a defined grain      Reference sources directly; mix grains         `table` or `incremental`

The shape of the DAG matters more than any single rule: many narrow models fan in, a few wide models come out. Dozens of raw tables become dozens of one-to-one staging views, a handful of intermediate models do the heavy lifting, and a small number of marts is what analysts actually query.

The three-layer dbt DAG: many raw sources fan into one-to-one staging views, optional intermediate models do the heavy joins, and marts emerge as a few wide fct_/dim_ tables.

What does the folder tree actually look like?

Here's a trimmed example for a fictional jaffle_shop plus a stripe payment source. This is the layout I'd open a PR with:

models/
├── staging/
│   ├── jaffle_shop/
│   │   ├── _jaffle_shop__sources.yml
│   │   ├── _jaffle_shop__models.yml
│   │   ├── stg_jaffle_shop__customers.sql
│   │   └── stg_jaffle_shop__orders.sql
│   └── stripe/
│       ├── _stripe__sources.yml
│       ├── _stripe__models.yml
│       └── stg_stripe__payments.sql
├── intermediate/
│   └── finance/
│       ├── _int_finance__models.yml
│       └── int_payments_pivoted_to_orders.sql
└── marts/
    ├── finance/
    │   ├── _finance__models.yml
    │   ├── fct_orders.sql
    │   └── dim_customers.sql
    └── marketing/
        ├── _marketing__models.yml
        └── fct_campaign_performance.sql

Three conventions are doing quiet work here:

  1. Staging is organized by source system (jaffle_shop/, stripe/), because at this layer you still think in terms of "where did this come from." Marts are organized by business domain (finance/, marketing/), because by then you think in terms of "who consumes this."
  2. YAML files are co-located and prefixed with _ so they sort to the top of each folder. One _<source>__sources.yml declaring sources, one _<area>__models.yml documenting and testing the models in that folder. (More on this below.)
  3. The double underscore in stg_stripe__payments separates the source (stripe) from the entity (payments). It reads cleanly and never collides.

How should you name staging models?

The dbt Labs convention is stg_[source]__[entity]s.sql — source-prefixed, double-underscore separator, plural entity. So stg_stripe__payments.sql, stg_jaffle_shop__orders.sql.

The hard rule of staging: one staging model per source table, a strict 1:1 relationship. Each source table gets exactly one entry point into your project. Every model is the same shape — import CTE, rename CTE, final select:

-- models/staging/stripe/stg_stripe__payments.sql
with source as (
    select * from {{ source('stripe', 'payment') }}
),

renamed as (
    select
        -- ids
        id            as payment_id,
        orderid       as order_id,

        -- strings
        paymentmethod as payment_method,

        -- numerics: stripe stores amounts in cents
        amount / 100.0 as amount,

        -- booleans
        case when status = 'successful' then true else false end
                      as is_completed_payment,

        -- timestamps
        created::timestamp as created_at
    from source
)

select * from renamed

What's allowed here: renaming, recasting types, simple unit conversions (cents to dollars), and basic categorization with case. What's not allowed: joins and aggregations. A join duplicates logic and confuses the grain; an aggregation throws away rows you'll need downstream. If you genuinely must join at this layer (e.g., unioning a soft-deletes table), the documented escape hatch is a base_ model in a base/ subfolder that the staging model then reads from — keep the join out of the staging model proper.

And notice the very first line: {{ source('stripe', 'payment') }}, not raw.stripe.payment. That's the only place in your entire project that a raw table name should appear.

How do ref() and source() actually work?

These two macros are the load-bearing wall of dbt. They do two jobs at once: build the dependency graph (the DAG) and abstract away physical database locations.

  • source('stripe', 'payment') — used only in staging models — points at a raw table declared in a _sources.yml. Change the schema once in YAML and every reference follows.
  • ref('stg_stripe__payments') — used everywhere else — points at another dbt model by name. dbt resolves it to the correct database/schema/table for the current environment (dev vs. prod) and infers that this model must run after the referenced one.

The rule that keeps the DAG clean: source() appears once per source table, in exactly one staging model. Everything downstream uses ref(). If you find source() in an intermediate or mart model, that's a bug — you've bypassed your own staging layer and tied a business model directly to a raw table's quirks.

-- declaring the source once
-- models/staging/stripe/_stripe__sources.yml
version: 2
sources:
  - name: stripe
    database: raw
    schema: stripe
    tables:
      - name: payment
        loaded_at_field: _loaded_at
        freshness:
          warn_after: {count: 12, period: hour}
          error_after: {count: 24, period: hour}

When should you add an intermediate layer?

This is the layer people most often get wrong — either skipping it and bloating their marts, or over-engineering it with a model for every trivial join.

Add an intermediate model when the logic is either (a) reused by more than one mart, or (b) a complex fan-out/aggregation that a mart needs before its final join. Otherwise, keep simple joins inline in the mart. Don't add a layer for its own sake.

Decision flow for when to add an intermediate model instead of stuffing logic into a mart.

The classic case: payments arrive one-row-per-payment, but fct_orders needs one-row-per-order with payment amounts pivoted into columns. That reshaping doesn't belong in staging (it's a join/aggregation) and doesn't belong inline in the mart (it muddies the final grain). It belongs in int_payments_pivoted_to_orders:

-- models/intermediate/finance/int_payments_pivoted_to_orders.sql
{{ config(materialized='ephemeral') }}

with payments as (
    select * from {{ ref('stg_stripe__payments') }}
)

select
    order_id,
    sum(case when payment_method = 'credit_card' then amount else 0 end)
        as credit_card_amount,
    sum(case when payment_method = 'bank_transfer' then amount else 0 end)
        as bank_transfer_amount,
    sum(amount) as total_amount
from payments
group by 1

Intermediate models are typically materialized ephemeral (inlined as a CTE into downstream models, leaving no object in the warehouse) or as views. They are plumbing — never point a dashboard at one. dbt Labs suggests a rough ceiling of around 4–6 entities being brought together before a single intermediate model is doing too much; past that, split it.

What is the mart grain, and why does it decide everything?

The grain is the answer to "what does one row in this table represent?" — and it's the single most important design decision in a mart. One row per order. One row per customer per day. One row per subscription. If you can't state the grain in one sentence, the mart isn't designed yet.

Grain governs everything downstream: which surrogate key is unique, which tests you write, whether two metrics can be safely joined. Mixing grains in one table is the bug behind most "why is revenue double-counted?" incidents — you fanned out a one-row-per-order fact to one-row-per-order-line somewhere and summed the wrong column.

Marts should be wide and denormalized. In a modern warehouse storage is cheap and compute is the constraint, so packing everything a consumer needs about an entity into one wide table — rather than forcing them to join five normalized tables every query — is the goal.

-- models/marts/finance/fct_orders.sql
{{ config(materialized='table') }}

with orders as (
    select * from {{ ref('stg_jaffle_shop__orders') }}
),

order_payments as (
    select * from {{ ref('int_payments_pivoted_to_orders') }}
),

final as (
    select
        -- surrogate key on the declared grain: one row per order
        {{ dbt_utils.generate_surrogate_key(['orders.order_id']) }}
            as order_key,
        orders.order_id,
        orders.customer_id,
        orders.ordered_at,
        order_payments.credit_card_amount,
        order_payments.bank_transfer_amount,
        order_payments.total_amount as amount
    from orders
    left join order_payments
        on orders.order_id = order_payments.order_id
)

select * from final

Should you use fct_/dim_ or just entity names?

Here's a genuine fork in the road, and you should know both sides:

  • dbt Labs' official guide names marts by plain entity — orders.sql, customers.sql — arguing that by the marts layer you're modeling business concepts, not warehouse mechanics.
  • The widely-adopted community convention borrows from Kimball dimensional modeling: fct_ for fact/event tables (things that happen — orders, payments, sessions) and dim_ for dimension tables (things that exist — customers, products). So fct_orders, dim_customers.

My take, on teams with analysts and BI developers who already think in stars and snowflakes: fct_/dim_ earns its keep. The prefix tells a consumer instantly whether a table is additive (facts, safe to sum) or descriptive (dimensions, safe to group by), which prevents real metric bugs. If your consumers don't speak dimensional modeling, the plain entity names are cleaner. Either way — pick one and enforce it. A project that mixes fct_orders, orders_fact, and final_orders is the actual problem.

How do surrogate keys fit in?

A surrogate key is a single hashed key column standing in for a natural or composite key, so every mart has one clean, unique, joinable identifier on its grain. The current macro is dbt_utils.generate_surrogate_key() — note it takes a list of columns:

{{ dbt_utils.generate_surrogate_key(['order_id', 'order_line_number']) }}
    as order_line_key

Two things to know, both documented in dbt-utils: the older dbt_utils.surrogate_key() (varargs, no brackets) was deprecated in dbt-utils v1.0 — if you see it, migrate. And generate_surrogate_key treats null differently from empty strings by default; if you're porting old keys and need the legacy behavior, set the surrogate_key_treat_nulls_as_empty_strings var to true. The surrogate key is also the natural column to put a unique and not_null test on — which is exactly how you assert your grain.

How should you organize YAML and exposures?

Don't keep one giant schema.yml. Co-locate small YAML files with the models they describe, prefixed with _ so they sort first:

  • _<source>__sources.yml — source definitions and freshness, one per source system.
  • _<area>__models.yml — model descriptions, column docs, and tests, one per folder.

Exposures are the often-skipped capstone. An exposure is a YAML node that declares a downstream consumer of your data — a dashboard, a notebook, an ML pipeline — and the models it depends on. Once declared, dbt run -s +exposure:weekly_finance_dashboard builds exactly what that dashboard needs, and dbt test covers its lineage. It closes the loop from raw source to the thing an executive actually looks at:

# models/marts/finance/_finance__exposures.yml
version: 2
exposures:
  - name: weekly_finance_dashboard
    type: dashboard
    maturity: high
    url: https://bi.internal/dashboards/finance-weekly
    owner:
      name: Finance Analytics
      email: finance-data@company.com
    depends_on:
      - ref('fct_orders')
      - ref('dim_customers')

When does this structure NOT pay off?

Honesty matters more than dogma. If you have 15 models feeding one dashboard for a five-person startup, three formal layers with ephemeral intermediates is ceremony you'll resent. Start with staging plus marts and add the intermediate layer the first time you copy-paste a join between two marts — that's the signal, not a calendar date. Likewise, surrogate keys on every model and exposures for every dashboard are investments that pay back at scale and feel like overhead at ten models.

The structure earns its complexity exactly when more than one person touches the project, when the same logic starts appearing in two places, or when someone outside the data team asks "can I trust this number?" If none of those are true yet, take the layers you need and leave the rest — but name things consistently from row one, because renaming 300 models later is the migration nobody volunteers for.

Once your marts are stable, the two structural follow-ups are testing them properly and making the big ones build incrementally — both deserve their own treatment.

FAQ

Do I always need an intermediate layer? No. The intermediate layer is optional and purpose-built. Add a model there only when logic is reused across multiple marts or when a single mart needs a complex fan-out/aggregation before its final join. For a simple one-to-one join, keep it inline in the mart.

Should staging models be views or tables? Views, by default. Staging is light cleanup and you want downstream models to always read the freshest source data without paying to persist a near-copy. Move to tables only if a specific staging view becomes a measurable query bottleneck.

What's the difference between source() and ref()? source() points at a raw external table declared in YAML and should appear only in staging models. ref() points at another dbt model and is used everywhere else. Both build the DAG; together they ensure no model ever hard-codes a database table name.

fct_/dim_ or plain entity names like orders.sql? dbt Labs' official guide uses plain entity names; the broader community widely uses Kimball-style fct_/dim_ prefixes. Use fct_/dim_ if your consumers think in dimensional models — it signals additive vs. descriptive tables. The only wrong answer is mixing conventions in one project.

What replaced dbt_utils.surrogate_key? dbt_utils.generate_surrogate_key(), which takes a list of columns in brackets. The old surrogate_key() was deprecated in dbt-utils v1.0. The new macro also treats nulls differently from empty strings; set surrogate_key_treat_nulls_as_empty_strings: true if you need the legacy hashing behavior.
