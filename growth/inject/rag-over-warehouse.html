Retrieve schema and metric definitions to power text-to-SQL — not raw rows. The patterns, the traps, and when a plain SQL agent wins.

TL;DR: RAG over a data warehouse almost never means embedding your rows. It means retrieving the right metadata — table and column docs, metric definitions from your semantic layer, a business glossary, and a few gold-standard query examples — and feeding that as context to a text-to-SQL model. The model writes SQL; the warehouse does the aggregation. Embedding raw rows is the trap: it's expensive, stale within hours, and worse at numeric questions than a single GROUP BY. Reserve row-level retrieval for genuinely unstructured columns (support tickets, contract text), and reach for a plain SQL agent over RAG whenever your schema is small enough to fit in the prompt.

Reference architecture: RAG retrieves metadata (schema chunks, metric definitions, glossary, gold queries) as context for a text-to-SQL LLM, which generates SQL that runs against the warehouse. Raw rows are never embedded.

I've watched three teams try to "put the warehouse in a vector database." All three started by embedding rows. All three ended up rebuilding around metadata retrieval. This article is the architecture I'd hand someone on day one to skip that detour — and an honest list of the places RAG buys you nothing.

What does "RAG over a data warehouse" actually mean?

There are two fundamentally different things people mean, and conflating them is the root of most failures.

Pattern A — RAG to help text-to-SQL (the default). You retrieve descriptions of the data — schema, metric definitions, glossary terms, example queries — and inject them into the prompt. The LLM then generates SQL, which runs against the live warehouse. The numbers come from the database engine, not the model. This is what you want ~90% of the time.

Pattern B — RAG to retrieve rows. You embed the actual contents of tables and retrieve rows by semantic similarity at query time. This is the right tool for a narrow case: columns that are genuinely unstructured natural language — support ticket bodies, product reviews, contract clauses, call transcripts. It is the wrong tool for "what was revenue in Q1?"

The distinction matters because they have opposite failure modes. Pattern A fails when the model lacks context about your schema. Pattern B fails when people use it for analytics it was never meant to do.

How do you build Pattern A? The reference architecture

The runtime path is short. The work is in what you index offline.

  1. Index your metadata (offline). Chunk and embed four corpora into a vector store: (a) per-table and per-column documentation, (b) metric/measure definitions from your semantic layer, (c) the business glossary, (d) a curated set of gold question→SQL pairs. Keep raw data out of this.
  2. Retrieve at query time. Embed the user's question, run similarity search, and pull the top-k chunks across those corpora. A small reranker noticeably improves which tables and metrics surface.
  3. Generate SQL. Send the question + retrieved context to the model. Ask it to emit SQL plus a one-line explanation of which tables and metrics it used.
  4. Guard and execute. Validate the SQL (parse it, enforce a read-only role, add a LIMIT, optionally dry-run for cost). Run it against the warehouse. (I covered this layer in depth in text-to-sql-guardrails — don't skip it; a generated query with a missing join key will happily return a confident, wrong number.)
  5. Return the answer and the SQL. Always show the query. It's the only way a user can sanity-check, and it's your cheapest evaluation signal.
  6. Close the loop. When a user confirms an answer was right, promote that question→SQL pair into your gold-examples corpus. This is the single highest-leverage thing you can do for accuracy over time.

The semantic layer is your secret weapon

The reason most text-to-SQL demos look great and most production systems struggle is that the model doesn't know your business. "Revenue" might mean booked, recognized, or net of refunds. "Active user" has a definition that took your analytics team a quarter to agree on.

If you have a semantic layer — dbt's semantic models / MetricFlow, Cube, LookML, or an equivalent — its metric definitions are the highest-value thing you can retrieve. Per the dbt MetricFlow documentation, metrics are defined once against your models and compiled to SQL, which means a definition you retrieve is the same definition your BI tool uses. Feeding active_user = ... into the prompt collapses an entire class of "the LLM invented its own definition" errors. If you maintain those definitions as dbt tests/contracts, see llm-dbt-tests for keeping them honest.

If you don't have a semantic layer, build a lightweight one first: a YAML file mapping canonical metric names to their SQL expressions and the tables they live in. It's a smaller project than the RAG pipeline and it does more for accuracy.

How should you chunk schema and the glossary?

Chunking strategy is where Pattern A is won or lost, because retrieval quality caps everything downstream.

  • One chunk per table, enriched. Include the table name, a plain-English description, the column list with types, and — critically — descriptions of what each column means and example values for low-cardinality enums (e.g., status ∈ {active, churned, trial}). A bare CREATE TABLE statement retrieves poorly; humans named columns dt, amt_usd_net, and cust_seg and the model can't guess.
  • One chunk per metric. Name, definition in words, the exact SQL expression, the grain it's valid at, and synonyms ("revenue" / "sales" / "turnover").
  • Glossary terms as their own chunks, each linking to the tables/metrics it touches.
  • Gold queries as Q→SQL chunks. Embed on the natural-language question; return the SQL. These are the best few-shot examples possible because they're yours.

Two practical notes. First, retrieve at the right granularity: if questions routinely span many tables, also index join paths ("orders joins customers on customer_id") so the model gets relationships, not just isolated tables. Second, watch your top-k. Stuffing 40 tables into context to be safe degrades generation and burns tokens; a tight top-8 with a reranker usually beats a loose top-40.

Why is embedding raw rows a trap?

This is the mistake worth dwelling on, because it's the intuitive first move and it's almost always wrong for analytics.

  • Math doesn't survive retrieval. "Total revenue by region last quarter" requires summing every qualifying row. Similarity search returns a sample of semantically similar rows. The model then sums what it sees and reports a confident, wrong total. A SELECT region, SUM(revenue) ... GROUP BY region is exact, and the warehouse is built to do it on billions of rows.
  • Freshness collapses. Warehouse tables change continuously. Embedded rows are a snapshot frozen at index time — wrong within hours for any live table, and you're now running a re-embedding pipeline that races your ETL.
  • Cost scales with row count, not value. You pay to embed and store every row, then pay again on every update. Metadata, by contrast, is tiny: thousands of chunks, not billions of rows. Embedding-model pricing is typically per-token and is cheap per call, but multiply "cheap per call" by hundreds of millions of rows and re-embedding on every load, and it's a real, recurring line item. (Check current per-token rates for your embedding provider — they change.)
  • Cardinality and joins break. A row in isolation loses its relationships. The interesting answer usually lives in a join the retrieved row can't express.

The legitimate exception, again, is unstructured text columns. If you have a ticket_body column and the question is "what are customers complaining about regarding billing?", semantic retrieval over those text values is exactly right — that's classic document RAG that happens to live in a warehouse column. The healthiest production systems are hybrid: Pattern A for structured analytics, Pattern B scoped to the text columns, and a router that decides which one (or both) a question needs.

How do you handle freshness and cost?

Freshness splits cleanly by corpus:

  • Metadata (schema, metrics, glossary) changes at the speed of schema migrations and metric-definition edits — slowly. Re-index on a schedule, or trigger re-indexing from your dbt/CI pipeline when models change. This is the right hook: your schema docs and your warehouse never drift.
  • The data itself is always fresh because you never cached it — you query it live. That's the whole point of Pattern A.

On cost, the dominant runtime expense is LLM generation tokens, not retrieval. Keep prompts lean (tight top-k, no dumping the full schema), and for warehouses billed per bytes scanned (BigQuery's on-demand model bills by bytes processed, per Google's pricing docs), make the guardrail layer enforce partition filters and LIMITs so a vague question can't trigger a full-table scan. Costs and pricing models change across BigQuery, Snowflake, and Databricks — design the guardrail, then check current pricing for your platform.

How do you evaluate this without fooling yourself?

You cannot ship text-to-SQL on vibes. Build an evaluation set of question→expected-result pairs (or question→canonical-SQL) before you tune anything.

  • Execution accuracy is the metric that matters: run the generated SQL and the reference SQL, compare result sets. This is more meaningful than string-matching the SQL, since many correct queries differ syntactically. The Spider and BIRD text-to-SQL benchmarks both center execution-based evaluation for exactly this reason.
  • Track retrieval separately from generation. If the answer is wrong, you need to know whether retrieval missed the right table or the model misused a table it did get. Log the retrieved chunk IDs.
  • Test refusal. A good system says "I don't have a table for that" instead of inventing a join. Put answerless questions in your eval set.
  • Re-run on every change — new gold examples, prompt edits, model upgrades. This is the regression suite that lets you upgrade models without fear.

When does a plain SQL agent beat RAG?

RAG is machinery. If you don't need it, it's pure overhead — more infra, more failure modes, more latency. Skip it when:

  • Your schema fits in the context window. Modern models take large prompts; if your relevant surface is a few dozen well-documented tables, just put the whole schema and your metric definitions in the system prompt. No vector store, no retrieval step, no retrieval bugs. This is the most common case people over-engineer past.
  • The query surface is narrow and known. A dashboard-style assistant over five tables doesn't need similarity search.
  • Latency is critical. Every retrieval hop adds time. A static-context SQL agent is faster.

Reach for RAG when the schema is genuinely too large to fit (hundreds or thousands of tables), when you have a rich semantic layer and glossary worth retrieving selectively, or when you're doing hybrid structured-plus-unstructured questions. The honest rule of thumb: start with a plain SQL agent and a hand-written context block. Add retrieval only when the context stops fitting or accuracy plateaus because the model can't see the right tables. Retrofitting RAG onto a working SQL agent is easy; ripping a row-embedding pipeline out of production is not.

FAQ

Should I embed my database rows for RAG? Almost never for analytics. Embed metadata — schema docs, metric definitions, glossary, and example queries — and let generated SQL compute the answer. Only embed actual rows for genuinely unstructured text columns (tickets, reviews, contracts), and even then, scope it tightly.

What's the difference between text-to-SQL and RAG over a warehouse? Text-to-SQL is the act of generating a query from a question. RAG over a warehouse (Pattern A) is a technique to make text-to-SQL better by retrieving relevant schema and metric context first. RAG is the retrieval layer; text-to-SQL is the generation layer. You usually want both.

Do I need a vector database for this? Not necessarily. If your schema and metric definitions fit in the context window, put them directly in the prompt and skip retrieval entirely. Add a vector store only when the relevant schema is too large to fit or retrieval measurably improves which tables the model picks.

How do I keep the retrieved context in sync with my warehouse? Trigger re-indexing of metadata from your schema-migration or dbt CI pipeline so docs and definitions never drift from the tables. The data itself stays fresh automatically because Pattern A queries it live rather than caching it.

How do I evaluate a text-to-SQL RAG system? Use execution accuracy: run both the generated and reference SQL and compare result sets, since correct queries can differ syntactically. Log retrieved chunk IDs so you can separate retrieval failures from generation failures, and include answerless questions to test that the system refuses instead of hallucinating.
