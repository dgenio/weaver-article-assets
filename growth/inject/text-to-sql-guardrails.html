A senior data engineer's field guide to the failure modes that produce confident, wrong numbers — and the guardrail stack that catches them.

Text-to-SQL is still lying to you because the dangerous failures are silent: the query runs, returns a clean number, and the number is wrong. LLMs reliably produce executable SQL, but executable is not the same as correct. The fixes are not better prompts — they are engineering guardrails: ground the model in a curated semantic layer, validate every query before it runs (dry-run/EXPLAIN, column allowlists, read-only roles, enforced LIMIT, row-level security), sanity-check results after, keep a human in the loop for anything that writes or feeds a decision, and regression-test the whole thing against a golden eval set.

If you take one thing from this article: the error you should fear is not the query that throws — it's the query that succeeds and quietly double-counts revenue.

Why is "it runs" the wrong bar for text-to-SQL?

Demos grade on whether the query executes and looks plausible. Production grades on whether the number is true and safe to act on. Those are different problems.

The benchmark literature makes the gap concrete. On the classic Spider 1.0 benchmark, top systems report execution accuracy around 91%. But Spider 2.0 — which reframes evaluation around realistic enterprise schemas, multiple SQL dialects, and multi-step workflows — sees agentic success rates collapse to roughly 21% (Spider 2.0 / Lei et al.; see also the VLDB CIDR 2026 analysis of benchmark annotation errors). The harder, more business-like BIRD benchmark reports execution accuracy around 73%. Treat all of these as documented benchmark figures, not your production numbers — your schema is messier than any benchmark. But the direction is unambiguous: the closer the task gets to a real warehouse, the more the accuracy falls.

A model that is 73% correct on read-only analytics is a model that hands your CFO a wrong number roughly one time in four, with no warning. That is the entire problem.

What are the most common ways LLM-generated SQL goes wrong?

These are the failure modes I see repeatedly, ordered by how silent (and therefore dangerous) they are.

  • Fan-out / double-counting. The model joins a parent table to a one-to-many child and then aggregates the parent's value, multiplying it by the number of child rows. SUM(orders.amount) after joining order_items is the canonical disaster. It runs, it returns a bigger-than-real number, and nothing errors.

Fan-out double-counting: a single one-to-many join silently inflates a SUM.

  • Silent wrong aggregations. COUNT(*) vs COUNT(DISTINCT user_id); AVG over a column that already contains pre-averaged values; summing across a currency or unit column without normalizing. The query is valid; the semantics are wrong.
  • Wrong joins / wrong grain. Joining on a column that looks like a key but isn't unique, or picking the wrong of two plausible foreign keys, changes the grain of the result set. Filters then apply at the wrong level.
  • Hallucinated columns and tables. The model invents customers.lifetime_value because it's a reasonable name. Sometimes this throws (good). Sometimes it matches a real-but-different column, or a stale view, and returns a confident wrong answer (bad).
  • No row-level security. A model with broad table access will happily write a query that returns every tenant's data, every employee's salary, or PII the asking user is not entitled to see. The SQL is "correct" — the authorization is the bug.
  • Date and timezone drift. "Last month" computed in UTC when the business reports in local time; BETWEEN boundaries that include or exclude the endpoint inconsistently; fiscal vs calendar periods.
  • Implicit NULL handling. NOT IN with a NULL in the subquery returns nothing; NULLs silently dropped from aggregates; = comparisons against NULL. Standard SQL footguns the model reproduces faithfully.

Notice the pattern: only one of these (hallucinated columns) reliably fails loudly. The rest succeed. That is why prompting alone never fixes text-to-SQL.

How do you actually guardrail LLM-generated SQL?

Think in layers — before generation, before execution, after execution, and around the whole loop. No single control is sufficient.

1. Ground the model in a semantic layer, not the raw schema. Don't hand the LLM information_schema. Hand it a curated, described subset: the tables that matter, human-readable column descriptions, the blessed join paths, and definitions of business metrics ("revenue = SUM(net_amount) from fct_orders at order grain"). This is the single highest-leverage move. Managed products have converged on exactly this: Snowflake Cortex Analyst generates SQL from a YAML semantic model rather than the bare schema, and its own guidance recommends keeping a semantic view small — on the order of ten tables and well under a hundred columns — because precision degrades as you widen the surface (Snowflake Cortex Analyst docs). The lesson generalizes whether or not you use Snowflake: less, well-described schema beats more raw schema.

2. Validate the query before it runs. Parse it. Check every referenced table and column against an allowlist derived from your semantic layer — reject anything not on it (this kills hallucinated columns deterministically). Then run a dialect-aware EXPLAIN / dry-run to confirm it's valid and to inspect the plan without returning rows. BigQuery's dry-run mode, for example, validates and estimates bytes scanned without executing — use it both as a correctness gate and a cost gate.

3. Run as a locked-down, read-only role. The execution identity should be a dedicated read-only role with SELECT-only grants, scoped to exactly the curated tables, with row-level security and column masking enforced at the database, keyed to the asking user. Never enforce authorization in the prompt — enforce it in the database, where the LLM cannot talk its way around it.

4. Force a LIMIT and a timeout. Wrap or rewrite the query to cap returned rows and set a statement timeout / scanned-byte ceiling. This bounds the blast radius of a bad query — both the runaway cartesian join and the accidental full-table scan.

5. Sanity-check the result. After execution, check the shape before trusting the number: Did a "single customer" query return one row or 40,000 (grain blew up)? Is a non-null metric suddenly 90% null? Is the magnitude inside a plausible range vs the prior period? These cheap post-hoc checks catch fan-out and grain errors that pass every pre-execution gate.

6. Show the SQL, and keep a human in the loop where it matters. Always surface the generated SQL and the assumptions next to the answer. For read-only exploration by analysts, that transparency may be enough. For anything that writes, feeds an automated decision, or goes in front of an executive, require explicit human approval of the SQL before it runs. Text-to-SQL should draft; a person should commit.

7. Regression-test with an eval set. Maintain a golden set of question → known-correct-SQL (or known-correct-result) pairs covering your nastiest cases — fan-out, distinct counts, timezone edges, RLS. Run it on every prompt change, model upgrade, and semantic-layer edit. Without this, every "improvement" is a coin flip you can't measure. (If you already test your dbt models, the same discipline applies — see my piece on using LLMs to write dbt tests.)

Which guardrails matter most, and what do they cost?

You won't build all seven on day one. Here's how I prioritize them by what they catch versus what they cost to run.

Guardrail                  Failure mode it stops                            Effort to build                Catches silent errors?                    Skip it when…
------------------------   ----------------------------------------------   ----------------------------   ---------------------------------------   -----------------------------------
Semantic-layer grounding   Wrong joins, wrong grain, hallucinated columns   High (curation is the work)    Yes                                       Never — this is the foundation
Column/table allowlist     Hallucinated columns/tables                      Low                            Partially (loud ones)                     Never; it's nearly free
Dry-run / EXPLAIN          Invalid SQL, runaway scans                       Low–Medium                     No (validity + cost only)                 The engine has no dry-run mode
Read-only role + RLS       Data exfiltration, PII leakage                   Medium (DB-side setup)         N/A (security, not correctness)           Single-tenant, all-public data
Enforced LIMIT + timeout   Cartesian blowups, cost runaways                 Low                            No (bounds blast radius)                  Never on shared warehouses
Result sanity checks       Fan-out, grain explosions, null spikes           Medium                         **Yes — best catcher of silent errors**   Throwaway internal tooling
Human-in-the-loop          Everything, at a cost of latency                 Low to add, high in friction   Yes (a person reads it)                   Pure read-only analyst self-service
Eval set / regression      Regressions from model/prompt/schema changes     Medium–High (ongoing)          Yes (offline)                             A one-off, never-maintained POC

The two with the best ratio of silent-error-coverage to effort are the allowlist (cheap, deterministic) and result sanity checks (the single best detector of fan-out and grain bugs). Semantic grounding is the most work but also the only thing that meaningfully raises baseline correctness rather than just catching errors after the fact.

What does a production-grade guardrail architecture look like?

Wire the controls into a pipeline so no query reaches your warehouse unguarded.

The text-to-SQL guardrail stack: where each control sits between the user's question and the returned answer.

The flow, end to end:

  1. Question in. Capture the asking user's identity — you'll need it for RLS.
  2. Retrieve semantic context. Pull the relevant curated tables, descriptions, blessed joins, and metric definitions for this question.
  3. Generate SQL. The LLM proposes a query against the semantic layer only.
  4. Pre-execution gate. Static parse → allowlist check → dry-run/EXPLAIN. Reject and (optionally) ask the model to repair on failure.
  5. Execute under the read-only role, against a read replica, with RLS/masking, a hard LIMIT, and a timeout.
  6. Post-execution gate. Run result sanity checks; attach a confidence signal.
  7. Return the answer with the SQL and assumptions shown; route low-confidence or write-adjacent cases to human review.
  8. Offline: the eval set runs in CI against steps 3–6 on every change.

Two architectural notes from experience. First, always read from a replica, never primary — a runaway generated query should never contend with production transactional load. Second, log the full triple — question, generated SQL, result — for every call. That log is simultaneously your debugging trail, your future eval-set source, and your audit record for "who queried what." It is the cheapest thing here and the one teams most often skip.

When should you NOT use text-to-SQL at all?

Honest answer: more often than vendors imply.

  • High-stakes, repeated reporting (board metrics, regulatory filings, anything financial) should be a reviewed, version-controlled SQL model, not generated per-question. Generate it once with the LLM as a drafting aid, have a human bless it, then run that exact query forever. Don't re-roll the dice every quarter. (My SQL patterns for data engineers post covers the patterns worth hand-writing.)
  • Untrusted end users on sensitive data without bulletproof DB-side RLS. If you can't enforce authorization in the database, don't expose text-to-SQL to those users — the prompt is not a security boundary.
  • Messy, undocumented schemas. If you don't have (or won't invest in) a semantic layer, text-to-SQL will inherit every ambiguity in your warehouse and amplify it. Fix the schema first; the cleanup pays off regardless.

Text-to-SQL is genuinely excellent for what it's good at: letting analysts and semi-technical users explore well-modeled, read-only data fast, with the SQL visible and a human reading the result. Keep it in that lane and the guardrails above make it trustworthy. Push it into unattended, high-stakes, write-capable territory and no amount of prompting will save you.

FAQ

Is LLM-generated SQL accurate enough to trust in production? For low-stakes, read-only exploration with the SQL shown and a human reading results — yes, with guardrails. For unattended high-stakes reporting — no. Documented benchmarks range from ~73% execution accuracy on business-like data (BIRD) down to ~21% on enterprise-realistic tasks (Spider 2.0), so assume a meaningful wrong-answer rate and design controls around it rather than trusting raw output.

What is the most dangerous text-to-SQL failure mode? Silent ones — especially fan-out double-counting, where a one-to-many join causes a SUM to multiply. The query runs and returns a wrong-but-plausible number with no error. Result sanity checks (row counts, magnitude vs expected range) are the best defense.

How do I stop text-to-SQL from leaking data between users or tenants? Enforce authorization in the database, not the prompt. Run generated SQL under a dedicated read-only role with row-level security and column masking keyed to the asking user. The LLM cannot bypass a constraint the database enforces.

Does a semantic layer actually improve text-to-SQL accuracy? Yes — it's the highest-leverage control. Giving the model a curated, well-described subset of tables with blessed join paths and metric definitions (as Snowflake Cortex Analyst does via its YAML semantic model) substantially outperforms exposing the raw schema, and keeping that surface small improves precision.

What's the difference between dry-run validation and result sanity checks? Dry-run/EXPLAIN runs before execution and only confirms the SQL is valid and bounds its cost — it cannot tell you the answer is correct. Result sanity checks run after execution and inspect the returned data's shape and magnitude, which is what actually catches silent semantic errors like fan-out. You need both.

Can I just prompt the model better instead of building guardrails? No. Better prompts reduce some errors but cannot make correctness verifiable or authorization enforceable. The failures that matter are silent and semantic; only execution-time validation, database-side security, and post-hoc result checks can catch them deterministically.
