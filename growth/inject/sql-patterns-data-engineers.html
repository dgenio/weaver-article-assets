Dedup, gaps-and-islands, running totals, MERGE, QUALIFY, sessionization — real-pipeline patterns with tight, runnable snippets.

The ten SQL patterns every data engineer should know by heart are: deduplication with ROW_NUMBER, gaps-and-islands, running totals, pivot/unpivot, anti-joins for missing rows, date-spine (calendar) joins, top-N per group, incremental upsert with MERGE, QUALIFY for filtering window results, and sessionization. Master these and you can express the vast majority of real pipeline logic — dedup, change tracking, time-series rollups, and incremental loads — without reaching for a procedural language. Almost all of them are built on window functions, which is why window functions are the single highest-leverage thing to learn in SQL. Every snippet below is runnable; dialect differences are flagged inline.

A quick note before the list: these are patterns, not silver bullets. I've called out where each one quietly falls apart at scale or hides a correctness bug. If you're newer to the window-function machinery underneath several of these, my deeper walkthrough of SQL window functions is a good companion read.

Now the patterns themselves.

1. Deduplication with ROW_NUMBER

The problem: an upstream system delivered the same logical record more than once and you need exactly one row per key — usually the newest.

SELECT *
FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY updated_at DESC, event_id DESC
    ) AS rn
  FROM raw_events
) t
WHERE rn = 1;

Two things people get wrong. First, always add a deterministic tie-breaker (event_id DESC above) — if two rows share updated_at, ROW_NUMBER will still pick one, but which one becomes non-deterministic across runs, which silently breaks reproducibility. Second, use ROW_NUMBER (not RANK) when you want exactly one row; RANK returns all tied rows.

When not to use it: if duplicates are byte-identical, a plain SELECT DISTINCT or GROUP BY is cheaper and clearer.

2. Gaps-and-islands

The problem: you have rows that should form continuous runs (consecutive login days, a device that was online for a stretch) and you want to collapse each run into one row with a start and end. This is the famous "gaps-and-islands" problem.

The trick: subtract a per-group ROW_NUMBER from the sequence value. Within a continuous run the difference is constant, so it becomes a group key.

WITH numbered AS (
  SELECT
    user_id,
    activity_date,
    activity_date
      - (ROW_NUMBER() OVER (
          PARTITION BY user_id ORDER BY activity_date
        ) * INTERVAL '1 day') AS grp
  FROM daily_activity
)
SELECT
  user_id,
  MIN(activity_date) AS streak_start,
  MAX(activity_date) AS streak_end,
  COUNT(*)           AS streak_length
FROM numbered
GROUP BY user_id, grp
ORDER BY user_id, streak_start;

Gaps-and-islands: how a difference-of-row-numbers trick collapses consecutive rows into groups (islands) separated by gaps.

Dialect note: the INTERVAL '1 day' arithmetic is Postgres-style. In BigQuery use DATE_SUB(activity_date, INTERVAL rn DAY); in SQL Server use DATEADD(day, -rn, activity_date). The idea is identical everywhere.

3. Running totals and rolling windows

The problem: cumulative revenue to date, or a 7-day moving average.

SELECT
  order_date,
  daily_revenue,
  SUM(daily_revenue) OVER (
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total,
  AVG(daily_revenue) OVER (
    ORDER BY order_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_7day_avg
FROM daily_sales;

The frame clause is the part everyone forgets. With an ORDER BY but no explicit frame, most engines default to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. The subtle bug: RANGE groups peer rows (ties on the order key) together, so duplicate dates get the same running total. If you want strict row-by-row accumulation, write ROWS explicitly. This single distinction causes more "my running total looks wrong" tickets than anything else.

Caveat: a 7-ROWS window is not the same as a 7-day window if dates are missing. For a true calendar window you need a date spine (pattern 6) first, or a RANGE ... INTERVAL frame where supported.

4. Pivot and unpivot

The problem: turn rows into columns (a monthly crosstab) or columns back into rows (wide source table → tidy long table).

The portable pivot is conditional aggregation — it works in every dialect:

SELECT
  product_id,
  SUM(CASE WHEN quarter = 'Q1' THEN revenue END) AS q1,
  SUM(CASE WHEN quarter = 'Q2' THEN revenue END) AS q2,
  SUM(CASE WHEN quarter = 'Q3' THEN revenue END) AS q3,
  SUM(CASE WHEN quarter = 'Q4' THEN revenue END) AS q4
FROM sales
GROUP BY product_id;

Unpivot (wide → long), also portable:

SELECT product_id, 'q1' AS quarter, q1 AS revenue FROM wide_sales
UNION ALL
SELECT product_id, 'q2', q2 FROM wide_sales
UNION ALL
SELECT product_id, 'q3', q3 FROM wide_sales
UNION ALL
SELECT product_id, 'q4', q4 FROM wide_sales;

Dialect note: SQL Server and Oracle have dedicated PIVOT/UNPIVOT operators; BigQuery and Snowflake support PIVOT too. They read nicely but require you to enumerate the pivot values — there's no truly dynamic pivot in standard SQL without generating the query string. When the column set is dynamic, do the pivot in your application or BI layer instead of fighting SQL.

5. Anti-join for missing rows

The problem: which customers in orders have no matching row in shipments? Reconciliation and data-quality checks live here.

-- LEFT JOIN / IS NULL form
SELECT o.order_id
FROM orders o
LEFT JOIN shipments s ON s.order_id = o.order_id
WHERE s.order_id IS NULL;

-- NOT EXISTS form (often the planner's favorite)
SELECT o.order_id
FROM orders o
WHERE NOT EXISTS (
  SELECT 1 FROM shipments s WHERE s.order_id = o.order_id
);

The trap to internalize: do not use NOT IN on a subquery that can return NULL. If any value in the NOT IN list is NULL, the entire predicate evaluates to unknown and you get zero rows back — silently. NOT EXISTS and the LEFT JOIN/IS NULL form are NULL-safe; prefer them. On most engines NOT EXISTS and the anti-join optimize to the same plan.

6. Date spine / calendar join

The problem: your report shows revenue per day, but days with no sales simply vanish — and a line chart with missing days lies. The fix is to generate a complete date range and LEFT JOIN your data onto it.

-- Postgres
WITH calendar AS (
  SELECT generate_series(
    DATE '2026-01-01', DATE '2026-01-31', INTERVAL '1 day'
  )::date AS d
)
SELECT
  c.d AS order_date,
  COALESCE(SUM(s.amount), 0) AS revenue
FROM calendar c
LEFT JOIN sales s ON s.order_date = c.d
GROUP BY c.d
ORDER BY c.d;

Dialect note: BigQuery uses GENERATE_DATE_ARRAY('2026-01-01','2026-01-31', INTERVAL 1 DAY) with UNNEST. Snowflake uses GENERATOR + DATEADD. In production warehouses, most teams keep a permanent dim_date table instead of generating the spine each query — it's reusable and lets you join on fiscal periods, holidays, and weekday flags for free.

7. Top-N per group

The problem: top 3 best-selling products within each category. A plain ORDER BY ... LIMIT gives you the global top 3, not per group.

SELECT category, product_id, revenue
FROM (
  SELECT
    category,
    product_id,
    revenue,
    ROW_NUMBER() OVER (
      PARTITION BY category ORDER BY revenue DESC
    ) AS rn
  FROM product_sales
) t
WHERE rn <= 3
ORDER BY category, rn;

Choose your ranking function on purpose: ROW_NUMBER gives exactly N rows (arbitrary tie-break), RANK keeps all ties (you might get 4+ rows on a tie), and DENSE_RANK keeps ties without skipping rank numbers. For "top 3 and I don't care which on a tie," ROW_NUMBER. For "everyone tied for 3rd counts," RANK.

8. Incremental upsert with MERGE

The problem: you have a batch of new and changed rows and you want to update existing keys and insert new ones in a single statement — the bread and butter of incremental loads.

MERGE INTO dim_customer AS tgt
USING staging_customer AS src
  ON tgt.customer_id = src.customer_id
WHEN MATCHED THEN
  UPDATE SET
    name       = src.name,
    email      = src.email,
    updated_at = src.updated_at
WHEN NOT MATCHED THEN
  INSERT (customer_id, name, email, updated_at)
  VALUES (src.customer_id, src.name, src.email, src.updated_at);

Incremental upsert flow: how MERGE routes each source row to UPDATE (matched) or INSERT (not matched) against the target table.

MERGE is in standard SQL and supported by SQL Server, Oracle, BigQuery, Snowflake, and Databricks. The landmine: if the source has more than one row per key, behavior is undefined or errors (SQL Server raises, BigQuery errors). Always deduplicate the source first — pattern 1 — so each key appears once. Postgres historically used INSERT ... ON CONFLICT DO UPDATE instead of MERGE (native MERGE arrived in Postgres 15); on MySQL the equivalent is INSERT ... ON DUPLICATE KEY UPDATE.

9. QUALIFY — filter on window function results

The problem: you can't put a window function in a WHERE clause (it's evaluated after WHERE), so deduplication needs a subquery wrapper. QUALIFY is the window-function equivalent of HAVING and removes the wrapper entirely.

SELECT
  user_id,
  event_id,
  updated_at
FROM raw_events
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY user_id ORDER BY updated_at DESC
) = 1;

That's pattern 1 in three fewer lines. The honest caveat: QUALIFY is not in the SQL standard. It works in Snowflake, BigQuery, Databricks, and DuckDB, but not in Postgres, MySQL, or SQL Server — there you must keep the subquery-and-WHERE form. If portability matters, stick with the wrapped subquery; if you live in a modern cloud warehouse, QUALIFY is one of the nicest quality-of-life features it offers. For more on what's safe to lean on in BigQuery specifically, see my notes on BigQuery antipatterns.

10. Sessionization

The problem: a stream of event timestamps per user, and you want to group them into sessions where a gap longer than (say) 30 minutes starts a new session. This is the canonical web-analytics pattern.

WITH gaps AS (
  SELECT
    user_id,
    event_ts,
    CASE
      WHEN event_ts - LAG(event_ts) OVER (
             PARTITION BY user_id ORDER BY event_ts
           ) > INTERVAL '30 minutes'
        OR LAG(event_ts) OVER (
             PARTITION BY user_id ORDER BY event_ts
           ) IS NULL
      THEN 1 ELSE 0
    END AS is_new_session
  FROM events
)
SELECT
  user_id,
  event_ts,
  SUM(is_new_session) OVER (
    PARTITION BY user_id ORDER BY event_ts
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS session_id
FROM gaps;

The mechanics: LAG gets the previous event's timestamp; if the gap exceeds the threshold (or it's the user's first event), flag a new session with 1; then a running SUM of that flag produces a monotonically increasing session_id per user. It's gaps-and-islands and running totals working together — which is exactly why those two are on this list. Dialect note: swap the INTERVAL arithmetic for TIMESTAMP_DIFF(event_ts, prev_ts, MINUTE) > 30 in BigQuery or DATEDIFF(minute, ...) in SQL Server.

How do these patterns relate to each other?

Notice how few primitives are actually in play. ROW_NUMBER, LAG/LEAD, and a windowed SUM cover deduplication, gaps-and-islands, top-N, running totals, and sessionization — five of the ten. Anti-joins, date spines, pivots, and MERGE are the non-window members. If you're going to invest study time, invest it in window functions and their frame clauses; everything else is composition.

FAQ

What is the difference between ROW_NUMBER, RANK, and DENSE_RANK? ROW_NUMBER assigns a unique sequential number even to tied rows (so exactly one row per number). RANK gives tied rows the same number and then skips (1, 1, 3). DENSE_RANK gives tied rows the same number without skipping (1, 1, 2). Use ROW_NUMBER for deduplication and strict top-N; use RANK/DENSE_RANK when ties should be kept.

Why does my NOT IN query return no rows? Almost certainly a NULL in the subquery. NOT IN (..., NULL, ...) evaluates to unknown for every row, so the result set is empty. Switch to NOT EXISTS or a LEFT JOIN ... WHERE x IS NULL, both of which handle NULL correctly.

Is QUALIFY standard SQL? No. QUALIFY is a vendor extension supported by Snowflake, BigQuery, Databricks, and DuckDB, among others. It is not available in PostgreSQL, MySQL, or SQL Server, where you must wrap the window function in a subquery and filter in the outer WHERE. Check your engine's current documentation before relying on it.

What is the difference between ROWS and RANGE in a window frame? ROWS counts physical rows, so each row gets its own frame. RANGE includes all rows that are peers on the ORDER BY value (ties), so duplicate sort keys share a frame and therefore share the computed value. For row-by-row running totals, specify ROWS explicitly — the default frame is usually RANGE, which can surprise you on duplicate dates.

When should I avoid MERGE? Avoid MERGE (or pre-deduplicate the source) when the source can contain multiple rows per matching key — behavior is undefined or errors across engines. Also reconsider it under high write concurrency, where some engines can deadlock or produce duplicate inserts; an explicit INSERT ... ON CONFLICT (Postgres) or a delete-then-insert pattern is sometimes safer. Always check your engine's documented MERGE semantics.
