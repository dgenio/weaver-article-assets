Calculate sample size for a conversion-rate A/B test with statsmodels — plus the power, alpha, and MDE decisions that drive the number.

To find the sample size for an A/B test on a conversion rate, you need four inputs: the baseline rate, the minimum detectable effect (MDE) you care about, the significance level (alpha, usually 0.05), and the statistical power (usually 0.80). Plug them into the two-proportion sample-size formula. In Python, convert the two rates to a standardized effect size with statsmodels' proportion_effectsize, then solve for n with NormalIndPower().solve_power. The result is the number of users per arm — and the single biggest lever is the MDE, because required n scales with roughly 1/MDE².

Most teams run A/B tests they were never statistically equipped to win. They ship a variant, watch the dashboard for a week, see "no significant difference," and conclude the change didn't work. Often the truth is that the test never had a chance: it was underpowered from the first user. Calculating sample size before you launch is the cheapest insurance in all of experimentation. This article walks through the formula, the runnable Python, a worked numeric example, a reference table, and the mistakes that quietly waste weeks of traffic.

What is "sample size" actually answering?

Sample size answers a deceptively simple question: how many users do I need in each group so that, if a real effect of a given size exists, my test will probably detect it?

That sentence hides four parameters, and you must choose all four before the math works:

  • Baseline conversion rate (p₁). Your current control rate — say 5% of visitors convert. Pull this from real historical data, not a guess.
  • Minimum detectable effect (MDE). The smallest improvement worth detecting. If you only care about lifts of 1 percentage point or more (5% → 6%), that 1pp is your MDE. Smaller MDE = much larger sample.
  • Significance level (alpha, α). Your tolerance for a false positive — concluding the variant works when it doesn't. Convention is 0.05.
  • Statistical power (1 − β). Your probability of detecting a real effect of MDE size. Convention is 0.80, meaning you accept a 20% chance of missing a true effect.

Get these four, and the required sample size is determined. There's no fifth degree of freedom to negotiate.

What is the two-proportion sample-size formula?

For comparing two conversion rates with a two-sided test, the classic approximation for the number of observations per group is:

n = (z_(1−α/2) + z_(1−β))² × [ p₁(1−p₁) + p₂(1−p₂) ] / (p₂ − p₁)²

Where:

  • z_(1−α/2) is the standard-normal critical value for your alpha (≈ 1.96 for α = 0.05, two-sided),
  • z_(1−β) is the critical value for your power (≈ 0.84 for 80% power),
  • p₁ is the baseline rate and p₂ = p₁ + MDE is the rate you want to be able to detect,
  • the denominator (p₂ − p₁)² is the effect size squared — which is why shrinking the MDE blows up n.

You can compute this by hand, but in practice you should let a library handle the edge cases. statsmodels uses a slightly more refined approach based on the arcsine-transformed effect size (Cohen's h), which behaves better for proportions near 0 or 1 and matches R's pwr package.

How do I calculate A/B test sample size in Python?

Here is the canonical, runnable approach with statsmodels. Two functions do the work: proportion_effectsize turns your two rates into Cohen's h, and NormalIndPower().solve_power solves the power equation for whichever argument you leave as None.

from statsmodels.stats.power import NormalIndPower
from statsmodels.stats.proportion import proportion_effectsize
import numpy as np

def sample_size_per_arm(baseline, mde, alpha=0.05, power=0.80, ratio=1.0):
    """Required sample size PER ARM for a two-proportion A/B test.

    baseline : current conversion rate, e.g. 0.05
    mde      : minimum detectable effect as an ABSOLUTE lift, e.g. 0.01 (1pp)
    alpha    : false-positive tolerance (two-sided)
    power    : probability of detecting a true effect of size `mde`
    ratio    : nobs2 / nobs1 (1.0 = equal-sized groups)
    """
    p1 = baseline
    p2 = baseline + mde
    # Cohen's h: 2*(arcsin(sqrt(p2)) - arcsin(sqrt(p1)))
    effect = proportion_effectsize(p2, p1)
    n1 = NormalIndPower().solve_power(
        effect_size=effect,
        alpha=alpha,
        power=power,
        ratio=ratio,
        alternative="two-sided",
    )
    return int(np.ceil(n1))

n = sample_size_per_arm(baseline=0.05, mde=0.01)
print(f"Need ~{n:,} users per arm  (~{2*n:,} total)")
# Need ~8,143 users per arm  (~16,286 total)

A few things worth knowing about this API:

  • solve_power solves for whatever you set to None. Leave power=None to find the power of a test you already ran; leave nobs1=None (the default when you don't pass it) to solve for sample size.
  • nobs1 is the size of group 1, and nobs2 = nobs1 * ratio. With ratio=1.0 the number returned is your per-arm size.
  • proportion_effectsize(p2, p1) returns Cohen's h via the arcsine transform: 2*(arcsin(sqrt(p2)) - arcsin(sqrt(p1))). Sign depends on argument order; sample size depends only on its magnitude.

A note on relative vs. absolute MDE. Product teams usually think in relative lift ("a 20% improvement"). A 20% relative lift on a 5% baseline is an absolute lift of 0.05 × 0.20 = 0.01 (1pp), so mde=0.01. Mixing these up is one of the most common ways to mis-size a test by a factor of several. Decide which one you mean and convert explicitly.

What actually drives the number?

Of the four inputs, they are not equal in influence:

  1. MDE dominates. Sample size scales with roughly 1/MDE². Halving the effect you want to detect quadruples your required traffic. This is the lever, and it's the one teams are most tempted to fudge.
  2. Baseline rate matters through variance. Rates near 50% have the most variance — p(1−p) is maximized at 0.5 — so they need more samples for the same absolute MDE. A 1pp lift is much "bigger" relative to a 2% baseline than to a 50% baseline.
  3. Power and alpha matter less, but they matter. Moving power from 0.80 to 0.90 increases n by roughly a third. Tightening alpha from 0.05 to 0.01 also raises it. These are real costs but rarely the thing that makes a test infeasible.

A worked numeric example

Suppose your checkout page converts at 5% and you want to detect at least a 1 percentage point absolute improvement (to 6%) — a sizable 20% relative lift. You use the conventional α = 0.05 and power = 0.80.

Running sample_size_per_arm(baseline=0.05, mde=0.01) returns ≈ 8,143 users per arm, or about 16,286 total.

Now suppose your real product traffic is 2,000 eligible users per week, split evenly. You'd need 16,286 / 2,000 ≈ 8.1 weeks of clean traffic. If the most you can stomach is a three-week test, you have three honest options: accept a larger MDE (only chase bigger wins), reduce variance (see CUPED below), or don't run the test. What you must not do is run it for three weeks anyway and treat a null result as evidence the change failed.

Why underpowered tests waste time (and mislead)

An underpowered test is worse than no test, because it manufactures false confidence in both directions.

  • It hides real wins. With low power, a genuinely good variant frequently produces a non-significant result. The team concludes "no effect," ships nothing, and the improvement is lost — sometimes permanently, because nobody re-tests a "failed" idea.
  • It inflates the wins it does report. This is the subtle one. When an underpowered test does hit significance, the observed effect is biased upward — a phenomenon often called the winner's curse or Type M (magnitude) error. You ship a "30% lift" that regresses to 8% in production, and trust in experimentation erodes.

The fix is not heroics after the fact; it's sizing before launch. If the math says you can't reach adequate power in a reasonable window, that is itself a finding — change the MDE, reduce variance, or pick a different test.

How does peeking break your sample-size plan?

Sample-size math assumes you decide on n up front and evaluate once, at the end. "Peeking" — checking the p-value repeatedly and stopping the moment it dips below 0.05 — quietly destroys this. Each look is another chance for noise to cross the threshold, so your real false-positive rate balloons well above the nominal 5%. With enough peeks, you can manufacture "significance" from pure noise a large fraction of the time.

If you genuinely need to monitor and stop early, don't eyeball a fixed-horizon test. Use a method designed for continuous evaluation — sequential testing with alpha-spending (e.g., O'Brien–Fleming group-sequential boundaries) or always-valid inference based on confidence sequences. These deliberately adjust the threshold over time so the error rate stays controlled. Peeking and pre-computed sample size are two halves of the same discipline; I cover the full set of traps in the companion piece on common A/B test pitfalls.

When can you get away with a smaller sample? (Variance reduction)

The sample-size formula has variance in its numerator. Cut the variance and you cut the required n — without touching alpha, power, or the effect you can detect. The most popular technique is CUPED (Controlled-experiment Using Pre-Existing Data), introduced by Microsoft.

The intuition: a user's behavior before the experiment predicts their behavior during it. CUPED uses a pre-period covariate (e.g., each user's purchases in the 4 weeks before the test) to remove the predictable part of the metric, shrinking variance by the square of the covariate's correlation with the outcome. In practice teams report variance reductions that translate to materially shorter tests for the same power. It changes the metric you analyze, not the randomization, so it composes cleanly with everything above. I walk through the math and a NumPy implementation in the dedicated CUPED article.

A quick comparison of the levers:

Variance reduction (CUPED)      Drops with covariate correlation²   Engineering effort; needs good pre-period data   When you have predictive pre-experiment data

When should you not bother with formal sample sizing?

Honesty matters here. Formal power analysis is overhead, and there are cases where it's the wrong tool:

  • You don't have the traffic, period. If the table tells you a meaningful test needs 16k users per arm and you get 200 visitors a week, no amount of sizing math fixes that. Consider a qualitative method, a bigger swing (larger MDE), or a different decision process entirely.
  • The change is a no-brainer or a bug fix. You don't need an experiment to justify fixing a broken checkout button.
  • You're measuring a guardrail, not chasing a win. Sometimes you ship and simply monitor for harm; that's a different statistical question.

Formal sizing earns its keep when the decision is genuinely uncertain, the effect is plausibly small, and the cost of being wrong is real. That's most product experiments — but not all of them.

FAQ

What sample size do I need for an A/B test? It depends on four inputs: baseline conversion rate, minimum detectable effect, alpha (typically 0.05), and power (typically 0.80). For a 5% baseline and a 1pp absolute MDE at standard settings, you need roughly 8,200 users per arm. Always compute it for your own baseline rather than copying a number — use the statsmodels snippet above.

What's the difference between absolute and relative MDE? Absolute MDE is the raw difference in rates (5% → 6% is 1pp). Relative MDE is the percentage change (5% → 6% is a 20% relative lift). The formula uses the absolute difference, so convert relative targets first: relative MDE × baseline = absolute MDE. Confusing the two is a leading cause of mis-sized tests.

Is the sample size per group or total? NormalIndPower().solve_power with ratio=1.0 returns the size of one group (per arm). Double it for the total users you need across control and variant.

Why does a smaller effect need so many more users? Required sample size scales with roughly 1/MDE². Detecting an effect half as large needs about four times the traffic, because you're trying to separate a smaller signal from the same noise. This is why MDE is the most consequential input you choose.

Can I just stop the test early once it's significant? Not with a fixed-horizon design — that's peeking, and it inflates your false-positive rate well beyond 5%. If you need to stop early, use sequential testing or always-valid inference, which adjust the significance threshold to keep error rates controlled. See the A/B test pitfalls piece for details.

Does CUPED reduce the sample size I need? Effectively yes. CUPED reduces the variance of your metric using pre-experiment data, and since variance is in the numerator of the sample-size formula, lower variance means fewer users for the same power. It doesn't change your randomization or alpha. See the CUPED walkthrough.
