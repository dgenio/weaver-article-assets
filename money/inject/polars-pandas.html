When the lazy API and query optimizer actually pay off, how to translate your groupby/join/window patterns, and the three cases where you should just stay on pandas.

You know the moment. A groupby().apply() that used to finish before your coffee cooled now takes four minutes. Your laptop fan spins up. htop shows one core pinned at 100% while the other eleven sit idle. And the dataset isn't even that big — maybe 30 million rows of event logs that swelled to 14 GB once pandas loaded every column as object dtype.

You try the usual tricks: category dtypes, chunking, dropping columns you don't need. It helps a little. But the real problem is structural: pandas runs your code line by line, on one thread, materializing every intermediate DataFrame in memory whether you need it or not.

Polars attacks exactly these three pain points — single-threaded execution, eager materialization, and a memory-hungry data model. This is a practical migration guide: the mental model, side-by-side translations of the patterns you actually write (groupby/agg, joins, windows), and — just as important — the cases where migrating is a waste of your time.

All code below runs on Polars 1.39 and pandas 2.3.

The mental model shift: eager vs lazy

pandas is eager. Every line executes the instant Python reaches it:

import pandas as pd

df = pd.read_csv("events.csv")          # reads ALL columns into RAM now
df = df[df["sales"] > 15]               # new DataFrame, materialized now
out = df.groupby("region")["sales"].sum()  # runs now

Polars supports the same eager style, but its superpower is the lazy API. You describe the whole computation as a plan, and nothing executes until you call .collect():

import polars as pl

out = (
    pl.scan_csv("events.csv")           # nothing read yet — just a plan node
    .filter(pl.col("sales") > 15)
    .group_by("region")
    .agg(pl.col("sales").sum())
    .collect()                          # NOW it runs, once, across all cores
)

The difference isn't cosmetic. Because Polars sees the entire plan before running anything, it can rearrange it.

Eager pandas vs. lazy Polars: pandas runs each line immediately on one thread and materializes every intermediate; Polars builds a plan, optimizes it (predicate + projection pushdown), then executes once across all cores.

The query optimizer is the real reason to switch

Here's the part that converts skeptics. Ask Polars to explain its plan instead of running it:

lf = (
    pl.scan_csv("events.csv")
    .filter(pl.col("sales") > 15)
    .group_by("region")
    .agg(pl.col("sales").sum())
)
print(lf.explain())

You'll see output shaped like this:

AGGREGATE[maintain_order: false]
  [col("sales").sum()] BY [col("region")]
  FROM
  FILTER [(col("sales")) > (15)]
  FROM
    Csv SCAN [events.csv]; PROJECT 2/4 COLUMNS; SELECTION: (col("sales")) > (15)

Read the bottom line carefully. Two things happened that you did not write:

  • Projection pushdown: PROJECT 2/4 COLUMNS. You referenced only sales and region, so Polars reads just those two columns off disk. The other two are never parsed. On a wide table this alone can cut I/O and memory by an order of magnitude.
  • Predicate pushdown: SELECTION: (col("sales")) > (15). The filter was pushed down into the CSV scan, so rows that fail the predicate are discarded as they're read, not loaded and then dropped.

In pandas you'd get these optimizations only by hand-writing usecols= and filtering during read. Polars derives them from the plan automatically, every time. This is the same query-planning idea behind columnar engines like DuckDB — if you're weighing engines for larger-than-memory analytics, my DuckDB vs Spark piece covers where each wins.

How the optimizer rewrites a naive query: the logical plan you wrote (top) is transformed so the filter and column selection are pushed down into the scan, so less data is ever read.

Expressions vs apply: the habit you have to break

The single biggest source of slow pandas code is apply with a Python function. It's a row-by-row Python loop wearing a method's clothing.

# pandas — Python-level loop, slow
df["band"] = df["x"].apply(lambda v: "hi" if v > 2 else "lo")

Polars wants you to express the what, not the how, using its expression API. The engine then runs it as vectorized, multi-threaded native code:

# Polars — vectorized expression, no Python per-row callback
df = df.with_columns(
    pl.when(pl.col("x") > 2)
      .then(pl.lit("hi"))
      .otherwise(pl.lit("lo"))
      .alias("band")
)

pl.when/then/otherwise is the workhorse replacement for np.where, apply, and chained .loc assignments. The mental reframe: in pandas you mutate columns; in Polars you build expressions and hand them to with_columns or select. Expressions compose, parallelize, and feed the optimizer. Python lambdas do none of that. (Polars does have map_elements for true escape-hatch cases, but treat it as a code smell, not a default.)

There is no index. That's a feature.

pandas has an index, and it is a constant low-grade tax: reset_index(), set_index(), MultiIndex ambiguity, silent alignment surprises when two Series with different indexes get added. Polars has no index. A DataFrame is just named columns. You select rows with filter, you reference columns by name, and operations never silently align on a hidden axis. For 90% of analytical work this removes a whole category of bugs.

The translation cheat sheet

Here are the patterns you write every day, in both dialects. These all run as-is.

Setup

import polars as pl
import pandas as pd

data = {
    "region":  ["west", "west", "east", "east", "east"],
    "product": ["a", "b", "a", "b", "a"],
    "sales":   [10, 20, 30, 40, 50],
    "qty":     [1, 2, 3, 4, 5],
}
pdf = pd.DataFrame(data)
df  = pl.DataFrame(data)

groupby / agg with multiple aggregations

# pandas
pdf.groupby("region").agg(
    total_sales=("sales", "sum"),
    avg_qty=("qty", "mean"),
    n=("sales", "size"),
).reset_index()

# Polars
df.group_by("region").agg(
    pl.col("sales").sum().alias("total_sales"),
    pl.col("qty").mean().alias("avg_qty"),
    pl.len().alias("n"),
).sort("region")

Note Polars takes a list of expressions in agg, so naming and computing each output is one fluent line — no named-tuple gymnastics, no trailing reset_index().

Joins

mgr = {"region": ["west", "east"], "manager": ["alice", "bob"]}
pmgr = pd.DataFrame(mgr)
dmgr = pl.DataFrame(mgr)

# pandas
pdf.merge(pmgr, on="region", how="left")

# Polars
df.join(dmgr, on="region", how="left")

Same join types you know (inner, left, right, full, semi, anti, cross). Note Polars uses how="full" for a full outer join — the older "outer" name was dropped. Polars adds genuinely useful ones pandas lacks natively — semi and anti joins let you filter by membership without the isin + boolean-mask dance.

Window functions (group-relative columns)

This is where pandas transform lives. You want a per-row value computed within its group — for example, each row's share of its region's total sales.

# pandas
pdf["sales_share"] = (
    pdf["sales"] / pdf.groupby("region")["sales"].transform("sum")
)

# Polars — .over() is the window operator
df.with_columns(
    (pl.col("sales") / pl.col("sales").sum().over("region")).alias("sales_share")
)

.over("region") is the entire concept of groupby().transform() compressed into one operator that you can drop into any expression. It composes with when/then, ranking, cumulative sums, anything.

Reading data lazily

# pandas — eager, reads everything
pdf = pd.read_csv("events.csv")

# Polars — lazy scan, optimizer trims columns/rows before reading
lf  = pl.scan_csv("events.csv")     # also: scan_parquet, scan_ndjson
result = lf.select("region", "sales").filter(pl.col("sales") > 100).collect()

Reach for scan_* over read_* whenever you can. The scan is what unlocks pushdown.

Memory, threads, and streaming

Two more reasons the migration pays off on big data:

Multi-threading is automatic. Polars is built in Rust on Apache Arrow and parallelizes across cores by default — no swifter, no multiprocessing boilerplate, no dask scheduler to babysit. The groupby, the join, the expression evaluation all fan out across your CPU.

Streaming handles larger-than-memory. If your dataset doesn't fit in RAM, run the lazy plan through the streaming engine, which processes data in batches:

result = lf.group_by("region").agg(pl.col("sales").sum()).collect(engine="streaming")

(Older tutorials show collect(streaming=True); the current API is engine="streaming".) Combined with Arrow's compact columnar memory layout — proper typed columns instead of pandas' object dtype for strings — you simply fit more data in the same machine.

When NOT to migrate

A senior take requires naming where this is the wrong call. Don't rewrite if:

  • Your data is small. Under a few hundred thousand rows, pandas is plenty fast and you'll never notice the difference. The migration cost is pure overhead, and print(df) ergonomics in notebooks are still slightly more familiar to most teams.
  • You have a mature, working pandas pipeline. Code that already runs nightly without complaint is not a performance problem. Rewriting it introduces regression risk for no business value. Migrate the slow new thing, not the boring working thing.
  • You depend on the pandas ecosystem. scikit-learn, statsmodels, most plotting libraries, and a long tail of domain packages speak pandas natively. If your DataFrame's destination is model.fit(X, y), the conversion tax may eat your gains. The good news: interop is cheap and lossless enough for most cases —
pdf = df.to_pandas()      # Polars -> pandas, e.g. before sklearn
df  = pl.from_pandas(pdf) # pandas -> Polars

A pragmatic pattern: do the heavy filtering, joining, and aggregation in Polars (where threads and the optimizer earn their keep), then .to_pandas() the small result for the last-mile library that needs it.

Gotchas to expect

  • null is not NaN. Polars has a true null type distinct from floating-point NaN. Use .is_null() for missing data and .is_nan() only for genuine not-a-number floats. They are not interchangeable like pandas often lets them be.
  • Column references need pl.col(...). You can't write df["a"] + df["b"] inside a lazy expression chain. It's pl.col("a") + pl.col("b"). This trips up everyone for the first hour.
  • Eager .collect() everywhere defeats the purpose. If you .collect() after every step, you're back to eager pandas with extra syntax. Keep the chain lazy and collect once at the end.
  • String methods live under .str, dates under .dt. pl.col("name").str.to_uppercase(), pl.col("ts").dt.year(). Same idea as pandas' .str/.dt accessors, slightly different method names.

Takeaways

  • The win is structural, not magic: lazy plan + query optimizer (predicate & projection pushdown) + multi-threading + Arrow memory layout. Run .explain() and see the pushdown before you believe the hype.
  • Replace apply with expressions. pl.when/then/otherwise and .over() cover the vast majority of what apply/np.where/transform did, vectorized and parallel.
  • No index means fewer alignment bugs. Columns and filter, nothing hidden.
  • Use scan_* + collect(), not read_*, to unlock the optimizer; add engine="streaming" for larger-than-memory.
  • Don't migrate small data, working pipelines, or the last mile into pandas-only libraries. Use Polars for the heavy lifting, to_pandas() for the handoff.

Migrate the pipeline that's actually hurting — the one with the pinned core and the spinning fan — and leave the rest alone. That's the high-leverage move.
