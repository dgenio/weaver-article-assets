A senior engineer's guide to is_incremental(), the four incremental_strategy options, late-arriving data, and the max(updated_at) trap that silently drops rows.

It's 3 a.m. and your orders model has been running for forty minutes. It's a fact table with 2 billion rows, and every nightly run rebuilds all of it from scratch — re-scanning two years of history to add one day of new orders. The warehouse bill has a hockey-stick shape. The on-call engineer (you) is refreshing the dbt Cloud run page like it owes them money.

This is the exact problem incremental models solve. Instead of CREATE TABLE AS SELECT *, you process only the rows that arrived since the last run and graft them onto the existing table. Done right, a nightly run touches one day of data instead of two years.

Done wrong, you silently drop rows for weeks and nobody notices until finance asks why March revenue dropped 4%.

This article is the version of incremental models I wish someone had handed me earlier: the mechanism, real model code you can paste, the four strategies and when each is correct, and the classic max(updated_at) pattern along with the two bugs it quietly ships with.

The mechanism, in one model

Here's a complete, runnable incremental model. Everything else in this article is a variation on it.

-- models/marts/fct_orders.sql
{{
    config(
        materialized='incremental',
        unique_key='order_id',
        incremental_strategy='merge',
        on_schema_change='append_new_columns'
    )
}}

select
    order_id,
    customer_id,
    order_status,
    order_total,
    updated_at
from {{ ref('stg_orders') }}

{% if is_incremental() %}

  -- only on incremental runs: filter to rows newer than what we have
  where updated_at >= (
      select coalesce(max(updated_at), '1900-01-01') from {{ this }}
  )

{% endif %}

Two pieces of dbt magic are doing the work:

  • {{ this }} is a Jinja reference to the model's own current table in the warehouse — the version from the last run. You're querying yesterday's output to decide what's new.
  • {% if is_incremental() %} wraps the filter so it's applied only on incremental runs. On the very first run (or a full refresh) there's no table to compare against, so the filter is skipped and you build everything.

How is_incremental() actually decides

This is the part people get fuzzy on, and it causes real bugs. Per the dbt docs, is_incremental() returns True only when all three of these are true:

  1. The model is configured with materialized='incremental'.
  2. The target table already exists in the database.
  3. The --full-refresh flag was not passed for this run.

How dbt decides whether to run the incremental branch. is_incremental() is true only when all three conditions hold; otherwise the model does a full build.

The practical consequence: your SQL must be valid in both branches. dbt compiles the full query whether or not the if block runs. A common mistake is referencing a column or CTE inside the incremental branch that doesn't exist on the first full build — it works for weeks, then someone drops the table and the "first run" fails.

Want to force a clean rebuild? That's --full-refresh:

dbt run --select fct_orders --full-refresh

This drops and recreates the table from scratch. Use it after you change the schema, fix a backfill bug, or alter the business logic so historical rows need recomputing. It's your reset button — and a reminder that incremental models trade away automatic recomputation of history. If you change how order_total is calculated, already-loaded rows keep the old value until you full-refresh.

Choosing a strategy

incremental_strategy controls what dbt does with the new rows once it has them. There are four you'll actually use. They are not interchangeable — the difference is the guarantee you get, not the speed.

The four incremental strategies differ in what they do to the target table once the new rows are staged. Pick by guarantee, not by habit.

append — Insert the new rows. That's it. No matching, no dedup. It's the fastest and the most dangerous: re-run the same batch twice and you get duplicates. Use it only for genuinely append-only event streams where you control idempotency upstream, or where exact dedup happens in a downstream model. No unique_key needed.

merge — The default on BigQuery, Snowflake, and Databricks. Issues a SQL MERGE: for each staged row, if a row with the same unique_key exists, update it; otherwise insert. This makes re-running a batch idempotent — run it five times, same result. This is the right default for most fact and dimension tables.

delete+insert — DELETE the target rows whose unique_key matches the incoming batch, then INSERT the staged rows. Functionally similar idempotency to merge, but implemented as two statements. It's the go-to on adapters where MERGE is unsupported or slow — historically Redshift and Postgres lean on this.

insert_overwrite — The partition-aware one. Instead of matching individual keys, it replaces entire partitions that appear in the incoming data. Supported on BigQuery, Spark, and Databricks (among others). This is the strategy that makes time-partitioned tables genuinely cheap, and it deserves its own section below.

A note on portability: not every strategy works on every warehouse. merge, append, and delete+insert are broadly supported; insert_overwrite does partition-replace on BigQuery/Spark/Databricks. Snowflake also accepts insert_overwrite, but it is not partition-aware there — it behaves like a truncate + re-insert of the whole table, so you don't get the cheap per-partition rewrite. Check your adapter's docs. Check the strategy support matrix in the dbt docs for your adapter before you commit to one.

unique_key: the column that makes idempotency possible

For merge and delete+insert, unique_key is what dbt matches on. Two things will burn you:

  1. Nulls. If a unique_key column contains nulls, the match silently fails and you get duplicates or unmatched rows. dbt's own docs flag this explicitly.
  2. Strings vs. lists. Prefer a list of real columns over a concatenated string expression:
{{ config(
    materialized='incremental',
    unique_key=['order_id', 'line_item_id']
) }}

A list builds a proper composite-key join condition. A hand-rolled order_id || '-' || line_item_id string is fragile and harder for the next person to read.

The max(updated_at) from {{ this }} pattern — and its two real bugs

The filter in the first example is the most common incremental pattern in the wild:

{% if is_incremental() %}
  where updated_at >= (select max(updated_at) from {{ this }})
{% endif %}

It's clean and it usually works. But "usually" hides two failure modes that cause silent data loss — the worst kind, because dbt run stays green.

Bug 1: clock skew and the strict >

If you use strict > (greater-than, not >=), any rows whose updated_at exactly equals the previous max are never loaded. More subtly, if multiple source systems write updated_at from their own clocks, a row from a slightly-behind server can carry a timestamp earlier than a max you've already recorded — and it falls outside your window forever.

Two defenses:

{% if is_incremental() %}
  where updated_at >= (
      select coalesce(max(updated_at), '1900-01-01') from {{ this }}
  )
  -- overlap window: re-scan a safety margin to catch skew & late arrivals
  -- (the dedup step below makes the overlap safe)
{% endif %}

Use >=, not >, and add a lookback buffer (re-pull the last few hours or a day). The buffer reprocesses some rows you already have — which is fine only if your strategy is idempotent (merge or delete+insert with a correct unique_key). With append, a lookback buffer manufactures duplicates. This is why strategy and filter design are coupled.

Bug 2: duplicates from the overlap

Once you add a lookback window, the same logical row can appear in both the existing table and the incoming batch. merge/delete+insert handle that by key. But if your source itself can emit the same updated_at key more than once in a batch, dedup before the merge:

with ranked as (
    select
        *,
        row_number() over (
            partition by order_id
            order by updated_at desc, _loaded_at desc
        ) as rn
    from {{ ref('stg_orders') }}

    {% if is_incremental() %}
      where updated_at >= (
          select dateadd('day', -1, coalesce(max(updated_at), '1900-01-01'))
          from {{ this }}
      )
    {% endif %}
)

select * exclude (rn)   -- Snowflake/DuckDB syntax; on BigQuery use SELECT * EXCEPT (rn); list columns elsewhere
from ranked
where rn = 1

The row_number() keeps the latest version of each key. This is the single most important habit in incremental modeling: a high-water-mark filter and a dedup step are a pair, not a choice.

This is also exactly the kind of invariant you want a test guarding. A unique test on order_id (or a dbt_utils.unique_combination_of_columns on a composite key) turns "silent duplicate" into "loud red test." If you haven't set up testing on your incremental models, that's the natural next step — see my companion piece on dbt tests for the suite I run on every fact table.

Late-arriving data

"Late-arriving" means a row whose business event happened in the past but whose record showed up now — a backdated correction, a delayed webhook, an event buffered offline on a mobile device for three days.

If you filter on the event timestamp (event_time), late arrivals fall behind your high-water mark and never load. The fixes:

  • Filter on a load/ingestion timestamp (_loaded_at, the time the row hit your warehouse) rather than the event time, when your goal is "process everything new since last run."
  • Or keep filtering on event_time but widen the lookback window to cover your realistic lateness (e.g. 3 days), and rely on idempotent merge to correct already-loaded rows.

There's no free lunch: a wider window costs more compute per run. You're explicitly buying completeness with money. Pick the window from your data's actual lateness distribution, not a round number that feels safe.

Partitioned incrementals on BigQuery

This is where incremental models go from "nice" to "10x cheaper," because BigQuery bills by bytes scanned. The trick is insert_overwrite with partition_by and dynamic partitions:

{{
    config(
        materialized='incremental',
        incremental_strategy='insert_overwrite',
        partition_by={
            'field': 'order_date',
            'data_type': 'date',
            'granularity': 'day'
        }
    )
}}

select
    order_id,
    customer_id,
    order_total,
    cast(order_timestamp as date) as order_date
from {{ ref('stg_orders') }}

{% if is_incremental() %}
  where cast(order_timestamp as date) in (
      select distinct cast(order_timestamp as date)
      from {{ ref('stg_orders') }}
      where _loaded_at >= timestamp_sub(current_timestamp(), interval 2 day)
  )
{% endif %}

What happens under the hood: dbt computes which partitions (here, which order_date values) appear in the incoming batch, then atomically replaces just those partitions in the target. A nightly run that touches three days of order_date rewrites three partitions and scans roughly three days of source — not the whole table.

The partition math is the whole point. With daily partitions over two years (~730 partitions), reloading the latest day touches on the order of 1/730 of the table's storage and scan cost. That's the difference between a cents-per-run model and a dollars-per-run model, and it compounds every single night.

Two gotchas specific to insert_overwrite:

  • No unique_key needed — the partition is the unit of idempotency. But that also means a partition you don't re-supply is left untouched, even if rows in it changed. Your filter must capture every partition that has new or changed data.
  • If your incoming batch is missing a partition entirely (say, a day with zero orders that should have deleted yesterday's rows), insert_overwrite won't clear it. Partition-replace only overwrites partitions you hand it.

When NOT to use incremental models

Senior move: knowing when to skip the complexity.

  • Small tables. If a full rebuild runs in seconds, incremental logic is pure overhead and a maintenance liability. A table materialization is simpler and always correct.
  • Logic that frequently rewrites history. If your business rules routinely change values for old rows, you'll be running --full-refresh constantly — at which point incremental buys you nothing but bugs.
  • Heavy window functions across all history. If the model computes something like a running lifetime total per customer that depends on all prior rows, you can't just process "new" rows in isolation. These need careful design or a full rebuild.
  • You can't define a reliable high-water mark. No trustworthy updated_at, no reliable load timestamp, mutable rows with no version column? Incremental will silently drift. Fix the source contract first.

Incremental models are a performance optimization, and like all optimizations they add complexity. Reach for them when a full refresh genuinely hurts — not by default.

Takeaways

  • is_incremental() is true only when the model is incremental, the table already exists, and --full-refresh wasn't passed. Your SQL must compile in both branches.
  • Match the strategy to the guarantee: append (fast, dangerous, no dedup), merge (idempotent upsert, the sane default on BigQuery/Snowflake/Databricks), delete+insert (idempotent where merge is weak), insert_overwrite (partition-replace, cheapest for time-partitioned tables).
  • The high-water mark and a dedup step are a pair. Use >= not >, add a lookback buffer, and row_number() to keep the latest row per unique_key.
  • unique_key must be non-null. Nulls silently break matching and create duplicates. Use a list of columns for composite keys.
  • For late-arriving data, filter on load time or widen the lookback window — and only do this with an idempotent strategy.
  • On BigQuery, insert_overwrite + partition_by turns a full-table scan into a few-partition scan. Make sure your filter supplies every changed partition.
  • Guard all of it with tests — a unique test is what turns "silent data loss" into a failing CI run.

If you build incremental models without tests on the unique key, you're flying blind — that's the companion read: dbt tests that catch real bugs.
