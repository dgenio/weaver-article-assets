Spill is the silent tax on every shuffle. Here's how to read it in the Spark UI and tune it away with shuffle partitions, salting, and repartition vs. coalesce.

You kick off a Spark job that should take five minutes. Forty minutes later it's still grinding, your executors are pinned, and the only thing the logs tell you is that something is happening. You open the Spark UI, scroll to the stage detail, and there it is in the task metrics: Spill (Memory) and Spill (Disk), both with uncomfortably large numbers next to them.

That's the job quietly writing intermediate data to local disk because it ran out of memory to hold it. Spill is one of the most common reasons a Spark job is slow without ever throwing an error. It doesn't fail — it just drags, because you've traded fast RAM for slow disk I/O in the middle of a shuffle, often serializing and re-reading gigabytes you never needed to.

This guide is about recognizing spill, understanding why it happens, and the handful of knobs that actually fix it. No magic config that makes everything fast — just the mechanism and the trade-offs.

What "spill" actually means

When Spark executes a wide operation — a groupBy, a join, a sort, a window — each task needs working memory to build hash maps, sort buffers, and aggregation state. That working memory comes from the executor's execution memory pool.

If a task's working set is larger than the execution memory available to it, Spark doesn't just crash (most of the time). It spills: it sorts/serializes what it has, writes it to local disk, frees the memory, and keeps going. Later it reads those spill files back and merges them. The job completes — it's just paying disk I/O and extra serialization for the privilege.

In the Spark UI you'll see two numbers per stage:

  • Spill (Memory) — the in-memory (deserialized) size of the data that was spilled. This is usually the bigger, scarier-looking number.
  • Spill (Disk) — the actual serialized, compressed bytes written to disk.

The disk number is what hits your I/O. The memory number tells you how badly the data overflowed its in-RAM footprint. Both being non-zero across many tasks is your signal.

Where spill happens: a task's data outgrows its execution-memory slice and overflows to disk, twice (once in memory representation, once serialized on disk).

A subtle but important point: execution memory and storage memory (for cached/persisted blocks) share a single unified region in modern Spark. The boundary floats. So aggressively caching DataFrames you don't need can starve execution memory and cause spill. If you're spilling and also caching a lot, un-cache first and re-measure.

The usual root cause: partitions are the wrong size

Spill is almost always a partition-sizing problem in disguise. A Spark stage processes data in partitions, one task per partition. If each partition is too big to fit in a task's slice of execution memory, you spill. So the lever is: how many partitions, and therefore how big is each one.

The single most impactful default here is spark.sql.shuffle.partitions, which controls how many partitions exist after a shuffle (joins, aggregations, repartitions). Its default is 200, and it has been 200 for years regardless of your data size or cluster size.

That default is the source of an enormous amount of pain:

  • Shuffling 2 TB through 200 partitions means each post-shuffle partition is ~10 GB. No task holds 10 GB in memory. Massive spill.
  • Shuffling 200 MB through 200 partitions means each partition is ~1 MB. Now you have 200 near-empty tasks, each with scheduling overhead, producing 200 tiny output files.

Neither is what you want.

The 128MB rule of thumb

A practical target is roughly 128 MB of input per partition for a shuffle stage — the same order of magnitude as a default HDFS/Parquet block. It's not sacred (anywhere from ~100–200 MB is reasonable), but it's a good starting anchor: big enough that per-task overhead is negligible, small enough to comfortably fit in execution memory without spilling.

The math is simple:

target_partitions = ceil(total_shuffle_input_bytes / (128 * 1024 * 1024))

So if a stage shuffles ~64 GB, you want roughly 64 * 1024 / 128 = 512 partitions — not 200, and certainly not the 20 you'd get if you'd manually coalesced upstream.

Partition size is a Goldilocks problem. Too few partitions overflow memory and spill; too many drown the scheduler in tiny tasks. Aim for the middle.

Here's how to set it. You can set the SQL conf globally, or — better — size it per query based on the actual data volume:

from pyspark.sql import SparkSession
import math

spark = SparkSession.builder.getOrCreate()

# Global default — fine as a baseline, but it's one number for every shuffle.
spark.conf.set("spark.sql.shuffle.partitions", 512)

# Better: size it from the data you're about to shuffle.
def target_shuffle_partitions(approx_bytes, target_mb=128):
    return max(1, math.ceil(approx_bytes / (target_mb * 1024 * 1024)))

# e.g. you know (or estimate) the join input is ~64 GB
n = target_shuffle_partitions(64 * 1024**3)   # -> 512
spark.conf.set("spark.sql.shuffle.partitions", n)

Let Spark do it for you: Adaptive Query Execution

Since Spark 3.0, Adaptive Query Execution (AQE) can coalesce shuffle partitions automatically after it sees the real post-shuffle sizes. It's enabled by default in Spark 3.2+. If you're on a modern version, turn these on and let them carry most of the load:

spark.conf.set("spark.sql.adaptive.enabled", True)
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", True)
# The target size AQE aims for per coalesced partition:
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "128m")

AQE solves the "too many tiny partitions" case very well: you can set spark.sql.shuffle.partitions high (say 2000) and AQE will coalesce down to sensible sizes based on actual data. It also has a skew-join feature (spark.sql.adaptive.skewJoin.enabled) that splits oversized partitions. AQE is the first thing to check before you hand-tune anything — but it works off Spark's size estimates, so it's not a substitute for understanding your data. When the problem is skew, not size

Sometimes your average partition is fine but one or two are enormous. This is data skew: one key (or a few) dominates. A classic case is joining on a column where NULL, 0, or "unknown" accounts for 60% of the rows. All those rows hash to the same partition, one task gets buried, it spills hard (or OOMs), and the whole stage waits on that one straggler while everything else sits idle.

You'll spot it in the Spark UI immediately: in the stage's task table, sort by duration or by shuffle read size. If the max is 50x the median, you have skew, not a sizing problem. Bumping shuffle.partitions won't help — the hot key still lands in one partition.

The fix is salting: break the hot key into N sub-keys so its rows spread across N partitions, then undo the split after the join.

from pyspark.sql import functions as F

SALT = 16  # spread each key across 16 partitions

# Big, skewed side: add a random salt bucket to the join key.
facts_salted = facts.withColumn(
    "salt", (F.rand() * SALT).cast("int")
)

# Small / dimension side: replicate each row across all salt buckets
# so every salted fact row still finds its match.
salt_range = spark.range(SALT).withColumnRenamed("id", "salt")
dims_salted = dims.crossJoin(salt_range)

# Join on the original key PLUS the salt.
result = (
    facts_salted.join(dims_salted, on=["join_key", "salt"], how="inner")
                .drop("salt")
)

The trade-off is real: you've inflated the small side by SALTx (the crossJoin), so don't salt blindly. Salt only when you've confirmed skew, and only salt the keys that are actually hot — a common refinement is to salt just the known heavy keys and leave the rest untouched. And before you write any of this by hand, try AQE's skew-join handling; on Spark 3.2+ it resolves a lot of skew automatically by splitting the large partitions for you.

repartition vs. coalesce: pick the right tool

Two operations change partition counts, and people reach for the wrong one constantly.

  • repartition(n) triggers a full shuffle and gives you n roughly even partitions. Use it to increase partitions, or to fix skew/uneven sizing, or to repartition by a column (repartition("country")) before a write. It costs a shuffle, but it produces balanced output.
  • coalesce(n) only decreases partitions and avoids a full shuffle by merging adjacent partitions on the same executor. It's cheap, but it can produce uneven partitions, and if you coalesce too aggressively right before a heavy computation, you've just made each partition huge — hello, spill.
# Increasing partitions or rebalancing: must shuffle.
df_balanced = df.repartition(512)
df_by_key   = df.repartition(512, "customer_id")   # also sets up the join

# Reducing partitions cheaply (e.g. before writing fewer output files):
df_few_files = df.coalesce(10)

The classic spill-by-accident: you coalesce(1) to get a single output file, and that one task now has to hold the entire dataset in memory. It spills, or dies. If you need few output files and you're doing real work, repartition to a sane number and accept the shuffle, or write normally and compact afterward.

A repeatable workflow for killing spill

  1. Confirm it's spill. Open the Spark UI → Stages → the slow stage. Look at the aggregated Spill (Memory) / Spill (Disk) metrics and the per-task table.
  2. Check for skew first. Sort tasks by shuffle-read size. Max ≫ median → skew → salt (or enable AQE skew join). Don't touch partition counts yet.
  3. If it's uniform spill, it's sizing. Estimate the stage's shuffle input bytes and target ~128 MB per partition. Set spark.sql.shuffle.partitions accordingly, or lean on AQE coalescing.
  4. Enable AQE (Spark 3.2+) before hand-tuning. It often fixes both the too-many and too-few cases on its own.
  5. Re-measure. Re-run and confirm the spill metrics dropped. If you cache heavily, un-cache and check whether storage memory was starving execution memory.

When NOT to chase spill

Some spill is fine. A little spill at the tail of a large sort or aggregation is not worth a week of tuning — Spark is designed to spill rather than fail, and that's a feature. Chase it when:

  • spill is large relative to your data (you're writing back a big multiple of the input), or
  • it's on the critical path of a job you run constantly, or
  • it's paired with skew that's causing one straggler task to gate the whole stage.

If a nightly batch job spills modestly and finishes well within its window, leave it alone and go fix something that matters. Over-partitioning to chase the last byte of spill just trades it for scheduler overhead and a pile of tiny files.

Takeaways

  • Spill = memory overflow to disk. It rarely errors; it just makes jobs slow. Find it in the Spark UI under Spill (Memory) and Spill (Disk).
  • spark.sql.shuffle.partitions defaults to 200 regardless of data size. For big shuffles that's far too few; for tiny ones it's too many.
  • Target ~128 MB per partition: partitions ≈ ceil(shuffle_input_bytes / 128MB).
  • Enable AQE (default on in Spark 3.2+) to auto-coalesce partitions and handle a lot of skew — check it before hand-tuning.
  • Skew needs salting, not more partitions. Confirm with max-vs-median task size, then salt only the hot keys.
  • repartition shuffles for even partitions; coalesce cheaply reduces them but can create giant, spill-prone partitions. Never coalesce(1) before heavy work.
  • Don't over-optimize. Tolerable spill on a job that meets its SLA is not a bug.
