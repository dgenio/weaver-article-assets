Single-node columnar OLAP beats a distributed cluster on mid-sized data more often than anyone admits. No shuffle, no JVM, no 4-minute cluster spin-up — and the receipts to prove it.

For batch jobs under roughly 100GB, DuckDB usually beats an Apache Spark cluster because it skips the three costs that dominate a Spark job's wall-clock on mid-sized data: cluster acquisition (minutes to provision executors), JVM startup and serialization, and the network shuffle triggered by any GROUP BY, JOIN, or DISTINCT. DuckDB is an in-process columnar OLAP engine that reads Parquet directly with vectorized execution and morsel-driven parallelism across your cores — and it handles larger-than-memory workloads by spilling to disk. Spark still wins for genuinely huge multi-terabyte joins, horizontal scale-out, streaming, heavy multi-user concurrency, and mature governed platforms (Delta Lake, metastore, lineage). The uncomfortable truth about "big data"

Most jobs that run on a Spark cluster are not big-data jobs. They're mid-data jobs wearing a big-data costume.

A modern cloud box gives you a lot of headroom on one node. You can rent a single instance with 64–128 cores and 256GB–1TB of RAM. Against that, a 40GB Parquet dataset is not "distributed computing" territory — it's "fits comfortably on one machine" territory. When you force it through a distributed engine anyway, you pay for distribution you don't need:

  • Cluster acquisition. Ephemeral clusters take minutes to provision and register executors before a single byte is read.
  • JVM startup and serialization. Every executor boots a JVM. Data crosses serialization boundaries.
  • The shuffle. Any GROUP BY, JOIN, or DISTINCT that spans partitions triggers a shuffle — data is repartitioned and sent across the network between executors. On one node, there is no network.

DuckDB skips all three. It reads Parquet directly with a vectorized, columnar engine and parallelizes across your cores using morsel-driven parallelism — the same academic lineage (MonetDB/X100, Morsel-Driven Parallelism) that modern analytical databases are built on.

The 30-second version

Here's the entire "cluster" for a 40GB aggregation:

import duckdb

# No server, no cluster. This runs in your current process.
con = duckdb.connect()  # in-memory; or duckdb.connect("analytics.db") to persist

result = con.sql("""
    SELECT
        event_type,
        date_trunc('day', ts) AS day,
        count(*)              AS events,
        approx_count_distinct(user_id) AS uniques
    FROM read_parquet('s3://analytics/events/*/*.parquet')
    GROUP BY event_type, day
    ORDER BY day
""").df()   # .df() -> pandas, .pl() -> Polars, .arrow() -> Arrow, .fetchall() -> rows

print(result)

That's it. read_parquet scans the files directly — no load step, no CREATE TABLE, no ingestion job. DuckDB reads multiple Parquet row groups in parallel across your CPU cores by default. The output drops straight into pandas, Polars, or Arrow with zero-copy where possible.

If your data lives in S3, enable the httpfs extension once and point at the bucket:

con.sql("INSTALL httpfs; LOAD httpfs;")
con.sql("""
    CREATE SECRET (
        TYPE s3,
        KEY_ID    'AKIA...',
        SECRET    '...',
        REGION    'eu-west-1'
    );
""")

Why the single node wins: the mechanism

Don't take the speedup on faith — look at where the time goes. The query work is usually the smallest slice of a Spark job on mid-sized data. The expensive parts are structural overhead that simply don't exist on one node.

Where the time actually goes: a Spark job on mid-sized data spends most of its wall-clock on cluster acquisition, JVM startup, and a shuffle stage — none of which DuckDB has. The query itself is the small slice.

Three mechanisms do the heavy lifting:

1. Columnar + vectorized execution. DuckDB stores and processes data in columns, in batches (vectors) of values at a time rather than row-by-row. A SELECT count(*) ... GROUP BY event_type only ever touches the event_type column in your Parquet files — Parquet is columnar on disk, and DuckDB pushes that projection and any filters down into the scan, so unused columns and whole row groups are never read.

2. No shuffle. A distributed GROUP BY has to repartition rows by key across executors over the network — the shuffle stage. On a single node, the aggregation happens in shared memory across threads. There is no exchange operator, no network round-trip, no shuffle spill to worry about.

3. No JVM, no cluster. DuckDB is a C++ library that runs in your process. The "startup cost" is importing a Python package. There's nothing to provision and nothing to tear down.

You can see the projection/filter pushdown yourself — this is the senior-engineer move, reading the plan instead of trusting the vibe:

con.sql("""
    EXPLAIN
    SELECT event_type, count(*)
    FROM read_parquet('s3://analytics/events/*/*.parquet')
    WHERE event_type = 'purchase'
    GROUP BY event_type
""").show()

In the plan you'll see the PARQUET_SCAN node reporting which columns it projects and the filter being applied at scan time — confirmation that DuckDB isn't dragging the whole dataset into memory to filter afterward.

Partition pruning is free if your data is laid out well

If your Parquet is Hive-partitioned (year=2025/month=11/day=01/...), filters on the partition keys prune entire directories before any file is opened:

con.sql("""
    SELECT user_id, event_type, ts
    FROM read_parquet(
        's3://analytics/events/year=2025/month=11/day=01/*.parquet',
        hive_partitioning = true
    )
    WHERE event_type = 'purchase'
""").df()

One gotcha worth flagging: in some versions, reading Hive-partitioned data from object storage with a broad glob can enumerate all matching files first and then apply the partition filter, rather than touching only matching paths. Narrow your glob to the partitions you want when you can, and keep an eye on the file-listing step — it's the most common surprise when moving cloud workloads to DuckDB.

"But it won't fit in memory"

This is the objection that keeps people on Spark, and it's mostly outdated. DuckDB supports larger-than-memory workloads by spilling intermediates to disk. By default the memory_limit is set to about 80% of physical RAM; when an operator's intermediates grow past that, DuckDB adaptively spills to a temporary directory. Out-of-core support covers grouping, joining, sorting, and windowing — the operators that actually blow up.

You control it explicitly:

con.sql("SET memory_limit = '24GB';")
con.sql("SET temp_directory = '/mnt/fast-ssd/duckdb_tmp';")
con.sql("SET max_temp_directory_size = '200GB';")  # bound the spill
con.sql("SET threads = 16;")                        # default: your core count

Point temp_directory at a fast local SSD and a job whose hash tables exceed RAM still completes — it just reads and writes spill files in the background. If spilling is disabled or the temp limit is exceeded, you get a clean out-of-memory error and the query cancels, rather than silent corruption. That's the design boundary: DuckDB will work past memory, not infinitely past it.

A fair, reproducible comparison

If you want to convince a skeptical teammate (or yourself), don't trust a blog's numbers — including mine. Generate data and time both engines on your hardware. Here's a self-contained harness:

import duckdb, time

con = duckdb.connect()
# Generate ~a few GB of synthetic events as partitioned Parquet
con.sql("""
    COPY (
        SELECT
            (random() * 1e6)::BIGINT                          AS user_id,
            ['view','click','purchase'][1 + floor(random()*3)::INT] AS event_type,
            TIMESTAMP '2025-01-01' + (random()*365)::INT * INTERVAL '1 day' AS ts,
            random()                                           AS amount
        FROM range(50_000_000)
    ) TO 'events.parquet' (FORMAT parquet);
""")

t0 = time.time()
con.sql("""
    SELECT event_type, count(*) AS n, avg(amount) AS avg_amount
    FROM 'events.parquet'
    GROUP BY event_type
""").fetchall()
print(f"DuckDB aggregate: {time.time() - t0:.2f}s")

Run the equivalent groupBy().agg() in PySpark against the same file, and include the cluster start time in your stopwatch — because in production, you pay that every run. For mid-sized data, the gap is usually not subtle. The mechanism in the diagram is why.

When Spark still wins (and I'm not being polite)

DuckDB is a single-node engine. Past a point, one node is the wrong tool, and pretending otherwise is how you get paged at 3 a.m.

A simple decision boundary: dataset size and workload shape decide the engine. DuckDB owns the mid-data quadrant most teams actually live in.

Reach for Spark when:

  • The data is genuinely huge. Multi-terabyte joins where even one input doesn't fit on the largest node you can rent. This is what distributed shuffle is for.
  • You need horizontal scale-out. Throughput that no single box delivers, or jobs that must fan out across many machines on a fixed wall-clock budget.
  • Streaming. Continuous, stateful processing belongs to Spark Structured Streaming or Flink. DuckDB is a batch/interactive engine.
  • The ecosystem is already there. A mature Spark platform — Delta Lake ACID transactions, a metastore, lineage, existing tested pipelines, a team fluent in it. "Technically faster on one node" rarely justifies rewriting a working, governed platform.
  • Heavy concurrent multi-user workloads. DuckDB is embedded and single-writer; it's not a shared cluster serving hundreds of concurrent analysts.

The honest framing: size your data per job, not your ambitions. Most teams' median job is far smaller than their largest, and they provision for the largest. The mid-data quadrant — the one most jobs actually live in — is where a single node quietly wins.

If you're already rethinking single-node tooling, the same logic applies one layer down to your in-memory dataframes — I made the parallel case in Polars vs pandas, where the column-oriented, multi-threaded engine beats the row-oriented default for the same structural reasons.

Takeaways

  • Under ~100GB, default to a single node. A 64–128 core box with hundreds of GB of RAM handles far more than most "big data" jobs need.
  • The savings are structural, not magic. No cluster spin-up, no JVM, and crucially no shuffle — DuckDB removes the overhead that dominates a Spark job's wall-clock on mid-sized data.
  • Read Parquet directly. read_parquet(...) with projection/filter pushdown and Hive partition pruning means you scan only the columns and files you need. Verify with EXPLAIN.
  • Larger-than-memory is supported. Set memory_limit, temp_directory, and max_temp_directory_size, point spill at a fast SSD, and out-of-core grouping/join/sort/window jobs complete.
  • Keep Spark for truly huge, streaming, multi-user, and mature-platform workloads. Use the decision quadrant, not habit.
  • Benchmark on your own data — including cluster start time in the stopwatch, because you pay it every run.

I'm not anti-Spark. I'm anti-paying-for-a-cluster-to-GROUP BY-40GB. Try moving one mid-sized job to DuckDB this week, put EXPLAIN in front of it, and time it honestly against your current pipeline.

FAQ

When should I use DuckDB instead of Apache Spark? Default to DuckDB for batch or interactive jobs under roughly 100GB. A modern single cloud box (64–128 cores, 256GB–1TB RAM) handles far more than most "big data" jobs need, and DuckDB removes the cluster spin-up, JVM, and shuffle overhead that dominate a Spark job on mid-sized data.

Why is DuckDB faster than Spark on mid-sized data? Three mechanisms: columnar plus vectorized execution (it only reads the columns and row groups a query touches, via projection and filter pushdown), no shuffle (aggregation happens in shared memory across threads instead of repartitioning rows over the network), and no JVM or cluster (it's a C++ library running in your process, so startup is just importing a Python package).

Can DuckDB handle datasets larger than RAM? Yes. DuckDB supports larger-than-memory workloads by spilling intermediates to disk. The default memory_limit is about 80% of physical RAM, and out-of-core support covers grouping, joining, sorting, and windowing. Point temp_directory at a fast SSD and set max_temp_directory_size to bound the spill; if the limit is exceeded you get a clean out-of-memory error, not silent corruption.

Does DuckDB read Parquet from S3 directly? Yes. read_parquet() scans files directly with no load step or CREATE TABLE, reading multiple row groups in parallel across your cores. For S3, enable the httpfs extension once (INSTALL httpfs; LOAD httpfs;) and configure credentials via CREATE SECRET. Hive-partitioned data also gets partition pruning, though a broad glob can sometimes enumerate all files before filtering — narrow your glob when you can.

When does Spark still win over DuckDB? Reach for Spark when data is genuinely huge (multi-terabyte joins where even one input won't fit on the largest node), when you need horizontal scale-out across many machines, for continuous streaming (Spark Structured Streaming or Flink), for heavy concurrent multi-user workloads (DuckDB is embedded and single-writer), and when a mature governed platform already exists (Delta Lake ACID, metastore, lineage, tested pipelines).
