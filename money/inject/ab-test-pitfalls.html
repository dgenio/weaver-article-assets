Peeking, multiple comparisons, Simpson's paradox, sample-ratio mismatch, novelty effects, and underpowered tests — the mechanism behind each, plus runnable checks in Python.

You ship a button-color change. Three days in, the dashboard lights up green: +5.2% conversion, p = 0.04. You call the win in standup, the PM updates the roadmap, and the feature rolls to 100%.

Six weeks later the metric is flat. The lift evaporated. Nobody can explain it, so everyone quietly agrees the effect "regressed" and moves on.

It didn't regress. It was never there. Your A/B test lied to you — and it lied in one of a handful of completely predictable ways. The good news: each failure mode has a name, a mechanism, and a check you can run in a few lines of code. This article walks through six of them. None of the math is exotic; all of it is the kind of thing that silently ships losers as winners.

I'll show the mechanism behind each pitfall, not a parade of made-up benchmark numbers. Where I cite a figure, it's either a documented statistical fact or an illustrative simulation you can rerun yourself.

Pitfall 1: Peeking (optional stopping)

This is the most common and most expensive mistake, because it feels like diligence.

Here's the trap. A p-value threshold of 0.05 guarantees a 5% false-positive rate if you look exactly once, at a sample size you fixed in advance. The instant you check the test repeatedly and stop "as soon as it's significant," you've broken that guarantee. Each peek is another roll of the dice. Under the null hypothesis (no real difference), the test statistic wanders like a random walk; given enough chances, it will cross the 1.96 line by chance alone.

You can see it in a quick simulation. Generate two identical groups (no real effect), then check at multiple points and stop the first time p < 0.05:

import numpy as np, math

def false_positive_rate(n_peeks, n_max=10_000, trials=4000, seed=0):
    rng = np.random.default_rng(seed)
    checkpoints = np.linspace(n_max // n_peeks, n_max, n_peeks).astype(int)
    false_positives = 0
    for _ in range(trials):
        a = rng.binomial(1, 0.10, n_max)   # both arms: TRUE rate = 10%
        b = rng.binomial(1, 0.10, n_max)   # there is NO real difference
        for cp in checkpoints:
            xa, xb = a[:cp], b[:cp]
            p = (xa.sum() + xb.sum()) / (2 * cp)
            se = math.sqrt(p * (1 - p) * 2 / cp)
            if se and abs((xa.mean() - xb.mean()) / se) > 1.96:
                false_positives += 1
                break  # stop the moment it's "significant"
    return false_positives / trials

for k in (1, 5, 10):
    print(f"{k:2d} look(s): false-positive rate = {false_positive_rate(k):.1%}")

Run it and you'll get roughly:

 1 look(s): false-positive rate = 5.1%
 5 look(s): false-positive rate = 14.3%
10 look(s): false-positive rate = 20.2%

One look gives you the 5% you signed up for. Ten looks quadruples it. With continuous monitoring the false-positive rate eventually approaches 100% — wait long enough and almost any null test will cross the line at some point.

How peeking inflates the false-positive rate. Each time you check an ongoing test and stop the moment p < 0.05, you give the random walk another chance to cross the significance line. One look ≈ 5% false positives; repeated looks push it far higher.

The fix. Pick one of these, not none of them:

  • Fix the sample size in advance and only judge the test when it's reached (see Pitfall 6 for how to compute it). Boring, robust, free.
  • Use a sequential test designed for continuous monitoring — group sequential boundaries (O'Brien–Fleming, Pocock) or an always-valid approach like mSPRT / confidence sequences. These spend your alpha across looks on purpose, so peeking is allowed by construction. Several commercial platforms ("sequential testing," "always-valid p-values") implement exactly this.

What you cannot do is eyeball a fixed-horizon p-value every morning and stop when you like the number.

Pitfall 2: Multiple comparisons

You test one variant but track ten metrics. Or you run one experiment with five variants. Or you slice the result across eight segments looking for "where it worked." Every one of those is multiple comparisons, and each additional test you run at α = 0.05 adds another ~5% shot at a false positive.

The family-wise error rate — the probability of at least one false positive across m independent tests — is 1 − (1 − α)^m:

for m in (1, 5, 10, 20):
    fwer = 1 - (1 - 0.05) ** m
    print(f"{m:2d} tests: P(at least one false positive) = {fwer:.0%}")
# 1 tests: 5%   5 tests: 23%   10 tests: 40%   20 tests: 64%

Ten metrics and you're more likely than not to find something "significant" in a test where nothing is real. This is how "the variant didn't move the primary metric, but it did lift 7-day retention!" gets reported as a win.

The fix. Decide what counts before you peek at results:

  • Declare one primary metric up front. Everything else is a secondary/guardrail metric that can generate hypotheses but not decisions.
  • When you genuinely must test many hypotheses, correct for it. Bonferroni (test each at α/m) is dead simple and conservative. Benjamini–Hochberg controls the false discovery rate and is far less brutal when you have many tests:
from statsmodels.stats.multitest import multipletests
pvals = [0.001, 0.013, 0.021, 0.04, 0.31, 0.55]
reject, p_adj, *_ = multipletests(pvals, alpha=0.05, method="fdr_bh")
print(list(zip(reject, p_adj.round(3))))

(pip install statsmodels if you don't have it; for Bonferroni pass method="bonferroni".)

Pitfall 3: Simpson's paradox (segmentation)

The variant wins overall. You drill into segments to celebrate, and discover it loses on mobile, loses on desktop, loses in every country. How can it lose everywhere yet win overall?

Because the aggregate rate is a weighted average of segment rates, and the weights can flip the conclusion. If your variant happens to be served disproportionately to a high-converting segment, it inherits that segment's high rate without being any better.

Simpson's paradox: B beats A inside every segment, yet A wins overall because the segments are mixed in different proportions. The aggregate is a weighted average of the segments, and the weights — not the rates — flip the sign.

import pandas as pd
# B converts BETTER in every segment, yet A wins overall
data = pd.DataFrame([
    ("Mobile",  "A",  2,  100),   # 2.0%
    ("Mobile",  "B", 30, 1000),   # 3.0%  B better
    ("Desktop", "A", 90,  900),   # 10.0%
    ("Desktop", "B", 12,  100),   # 12.0% B better
], columns=["segment", "arm", "conv", "n"])

overall = data.groupby("arm").apply(
    lambda d: d.conv.sum() / d.n.sum(), include_groups=False)
print(overall)         # A ~9.2%,  B ~3.8%  -> A "wins"
print((data.assign(rate=lambda d: d.conv / d.n)
           .pivot(index="segment", columns="arm", values="rate")))
# every segment: B > A

A converts at ~9.2% overall, B at ~3.8% — yet B is better in every segment. The trap is the mix: B's traffic was mostly low-converting mobile, A's was mostly high-converting desktop.

The fix. Simpson's paradox in an experiment is almost always a symptom of broken randomization or biased assignment — proper randomization makes segment mixes equal in expectation, so the paradox shouldn't appear. So:

  1. Check the segment mix between arms first. If the variant has 60% mobile and control has 40% mobile, your assignment is broken (this is also Pitfall 4). Fix the plumbing before you trust any number.
  2. If mixes are genuinely equal, report segment-level results alongside the aggregate, and use a model (e.g., logistic regression with a segment term, or CUPED-style adjustment) rather than a raw pooled mean.

Pitfall 4: Sample-ratio mismatch (SRM)

You designed a 50/50 split. You got 50.4 / 49.6. Sounds fine. It is not necessarily fine.

SRM is the single best "is my experiment trustworthy at all?" check, and almost nobody runs it. If the ratio of users between arms differs from what you intended by more than chance allows, something in your assignment, logging, or filtering is broken — a redirect that drops slow connections, a bot filter that hits one arm harder, an SDK that fails to fire for one variant. When the populations differ, every downstream metric is suspect, because you're no longer comparing like with like.

The test is a one-line chi-square goodness-of-fit against your intended ratio:

from scipy.stats import chisquare
def srm_check(counts, expected_ratio=(0.5, 0.5)):
    n = sum(counts)
    expected = [n * r for r in expected_ratio]
    chi2, p = chisquare(f_obs=counts, f_exp=expected)
    flag = "SRM! investigate before trusting results" if p < 0.001 else "ok"
    return round(chi2, 2), round(p, 6), flag

print(srm_check([10_100, 9_900]))   # tiny skew -> (2.0, 0.157, 'ok')
print(srm_check([10_500, 9_500]))   # 52.5/47.5 -> (50.0, 0.0, 'SRM!')

A 52.5/47.5 split on 20,000 users — which doesn't look alarming at a glance — gives a p-value below 0.000001. That is not noise; that is a bug. The convention is to alarm at p < 0.001 (you're checking it on every test, so use a strict threshold to avoid false alarms).

The fix. When SRM fires, stop analyzing the metric and go debug the pipeline. Common culprits: redirect-based variants (the redirect itself causes drop-off), assignment happening before a filter that's correlated with the variant, bot/duplicate filtering applied unevenly, or logging that races. Do not "adjust for it" statistically — find the leak.

Pitfall 5: Novelty and primacy effects

Your new design shows a beautiful +8% in week one. By week three it's +1%. The effect was real — it just wasn't durable.

Two opposite biases live here:

  • Novelty effect: existing users click the shiny new thing because it's new, not because it's better. The lift decays as novelty wears off.
  • Primacy effect: existing users are confused by the change and underperform at first, then recover — making a good change look flat or negative early on.

Both come from the same root: your experiment population is dominated by existing users reacting to change itself, not to the steady-state experience a new user would have.

The fix. You don't need fancy math — you need to look at the effect over time and by user tenure:

import pandas as pd
# daily lift of variant vs control; is it stable or decaying?
def lift_by_day(df):  # df cols: day, arm, converted
    daily = (df.groupby(["day", "arm"]).converted.mean()
               .unstack("arm"))
    daily["lift_%"] = (daily["B"] / daily["A"] - 1) * 100
    return daily.round(3)
# A decaying 'lift_%' column (8 -> 5 -> 2 -> 1) is the novelty fingerprint.

Then, concretely:

  • Segment by new vs. returning users. The clean read on a permanent change is its effect on users who first saw it as new — they have no "old experience" to react to. If the effect holds for new users but not returning ones, you're looking at novelty/primacy.
  • Run the test long enough for the curve to flatten (typically at least one to two full business cycles — often a couple of weeks), and judge the stabilized lift, not the week-one spike.

Pitfall 6: Ignoring variance / underpowered tests

This is the quietest killer. An underpowered test isn't just "might miss a real effect" — when an underpowered test does hit significance, the estimated effect is inflated (the "winner's curse"). You only cross the line on a lucky draw, so the lift you report is systematically too big. That's a major reason wins don't replicate.

Power is just: given the effect size you care about, how likely are you to detect it? You should compute the required sample size before launching. For a two-proportion test:

import math
from statsmodels.stats.proportion import proportion_effectsize
from statsmodels.stats.power import NormalIndPower

def per_arm_n(baseline, mde_abs, alpha=0.05, power=0.8):
    es = proportion_effectsize(baseline, baseline + mde_abs)
    n = NormalIndPower().solve_power(
        effect_size=es, alpha=alpha, power=power, alternative="two-sided")
    return math.ceil(n)

print(per_arm_n(0.10, 0.010))  # detect 10.0%->11.0%  -> ~14,700 / arm
print(per_arm_n(0.10, 0.005))  # detect 10.0%->10.5%  -> ~57,800 / arm

Detecting a 1-percentage-point lift on a 10% baseline needs roughly 15,000 users per arm; halving the effect you want to catch quadruples the sample. If your traffic can't reach that in a reasonable window, you do not have a powered test — you have a coin flip with a dashboard.

The fix.

  • Run a power calculation before every test. If the required n is unreachable, either widen the minimum detectable effect you'll commit to, or don't run the test.
  • Reduce variance instead of just adding users. This is where CUPED earns its keep: by adjusting each user's metric with their pre-experiment behavior, it strips out variance you can predict, often cutting required sample size by a meaningful chunk. Lower variance is mathematically identical to more power — it's the cheapest power you'll ever buy.
  • Treat a lucky-looking win from a tiny sample with suspicion. A +40% lift on 200 users per arm is far more likely to be noise than a real 40% effect.

The 60-second pre-flight checklist

Before you trust any A/B result, run these in order. The first failure stops the line.

  • SRM check. Chi-square the arm counts against your intended split. p < 0.001 → broken pipeline, fix it before reading anything else.
  • Did you peek? If you stopped early on a fixed-horizon test, the p-value is not valid. Use a sequential method or wait for the pre-set n.
  • One primary metric? If you're celebrating a secondary metric or a segment, correct for multiple comparisons (BH / Bonferroni).
  • Segment mix equal across arms? Unequal mix → Simpson's paradox risk and likely an assignment bug.
  • Is the effect stable over time and for new users? A decaying lift is novelty, not value.
  • Was the test powered? Compute required n in advance. A significant result from an underpowered test is probably inflated.

None of these checks take real effort. What takes effort is unwinding a feature you shipped on a number that was never real — explaining to a roomful of people why the +5.2% turned into nothing. Run the checklist instead.
