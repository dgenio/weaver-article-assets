A human-in-the-loop workflow for generating not_null, unique, accepted_values, and relationships tests from schema and profiling stats — and the three failure modes that will burn you if you trust the output blind.

You inherit a 40-model dbt project. The previous analytics engineer left, and your dbt test suite has exactly nine tests in it — eight not_nulls and one unique on a primary key someone added during an incident. Meanwhile a finance dashboard quietly served duplicated revenue for three weeks because nobody asserted that order_id was unique in the staging model.

Writing tests for 40 models by hand is the kind of work that is simultaneously urgent and the first thing you deprioritize. It is repetitive, it is boring, and the boilerplate-to-insight ratio is brutal. So I did what a lot of us are quietly doing now: I fed the schema and some profiling stats to an LLM and asked it to propose the tests.

It went better than I expected in some places and worse in ways that would have shipped silent bugs if I had trusted it. This is the workflow I landed on, the parts I now let the model own, and the three failure modes that will bite you if you paste its output without reading it.

If you want a refresher on the test types themselves before going further, see the companion piece on dbt tests. This article assumes you know what not_null does and focuses on the generation problem.

The input matters more than the prompt

The single biggest lever is not prompt wording — it is what you put in the context window. An LLM handed only column names will guess from the names. That guessing is exactly where hallucinated tests come from. Handed actual profiling statistics, it stops guessing and starts describing the data you already have.

So before prompting, profile the model. You do not need a fancy tool; a SQL query gets you most of the way. Here is a profiling query you can run against any warehouse table (this is standard ANSI SQL):

-- Profile one column set for an LLM prompt.
-- Run per model; paste the result into the context.
select
    count(*)                                   as row_count,
    count(distinct order_id)                   as distinct_order_id,
    sum(case when order_id is null then 1 else 0 end) as null_order_id,
    count(distinct status)                     as distinct_status,
    min(order_total)                           as min_total,
    max(order_total)                           as max_total,
    sum(case when order_total < 0 then 1 else 0 end)  as negative_totals
from analytics.stg_orders;

And critically, for low-cardinality columns, hand the model the observed domain:

select status, count(*) as n
from analytics.stg_orders
group by 1
order by 2 desc;
-- e.g. completed: 81204, pending: 4410, cancelled: 2087, refunded: 311

That group by result is the highest-value thing you can give an LLM. It is the difference between the model inventing accepted values and the model reporting the ones that actually exist.

A workflow that survives contact with reality

The mistake is treating this as "LLM writes tests, I commit them." The workflow that actually held up treats the LLM as a fast junior who drafts, and treats you as the reviewer who classifies every line by how much trust it deserves.

The human-in-the-loop pipeline: profiling stats and schema go in, the LLM drafts tests, and every test passes through a review gate sorted by trust tier before it reaches schema.yml.

The prompt I settled on is boring on purpose. Boring prompts produce auditable output:

You are drafting dbt v1.x generic tests for one model.

Here is the model's column schema (name, type):
<paste schema>

Here are profiling stats for the table:
<paste row_count, distinct counts, null counts, min/max, negative counts>

Here are the full observed value domains for low-cardinality columns:
<paste the group-by results>

Rules:
- Only propose not_null where the null count is 0.
- Only propose unique where distinct count == row count.
- For accepted_values, use ONLY the values present in the observed domain
  I gave you. Do not add values you think *might* appear.
- Propose relationships tests ONLY as commented-out suggestions, and for
  each one state which two columns you are linking and why.
- For any business rule you cannot verify from the stats, emit it as a
  commented TODO, not an active test.
- Output valid schema.yml. After each test add a one-line `# rationale:`
  comment explaining which stat justifies it.

That last rule — a rationale per test — is what makes the review fast. You are not re-deriving the model's reasoning; you are checking it against the stat it cites.

The output, and how to read it

Here is a representative chunk of what comes back, lightly cleaned. Note the data_tests: key — dbt renamed tests: to data_tests: in v1.8, though tests: still works for backward compatibility:

version: 2

models:
  - name: stg_orders
    columns:
      - name: order_id
        data_tests:
          - not_null      # rationale: null_order_id = 0
          - unique        # rationale: distinct_order_id == row_count

      - name: status
        data_tests:
          - accepted_values:
              values: ['completed', 'pending', 'cancelled', 'refunded']
          # rationale: full observed domain, 4 distinct values

      - name: order_total
        data_tests:
          - not_null      # rationale: 0 nulls observed
          # TODO(human): negative_totals = 0 today, but is a negative
          # order_total ever valid (refunds)? business rule — verify.

      # SUGGESTION (commented, verify before enabling):
      # - name: customer_id
      #   data_tests:
      #     - relationships:
      #         to: ref('stg_customers')
      #         field: customer_id
      #   # rationale: name match only — I cannot confirm this FK from stats

Read that customer_id suggestion carefully, because it is the whole point of the article. The model linked stg_orders.customer_id to stg_customers.customer_id on the strength of the name. It has no idea whether your customers table is filtered to active customers, whether guest checkouts produce a customer_id that doesn't exist there, or whether the real key is account_id. Enable that test blind and your next run fails loudly — or worse, passes because the data happens to align today and breaks in three months when guest checkout ships.

Where it shines and where it bites

After running this across a real project, the tests sorted cleanly into trust tiers.

Where the LLM genuinely earns its keep:

  • not_null and unique boilerplate. Tedious, mechanical, and fully verifiable from the stats. This is 60–70% of a typical suite in my experience and the model nails it when you gate on the null/distinct counts.
  • accepted_values from an observed domain. When you hand it the group by result, it transcribes the domain accurately. This is a genuinely nice win — building these by hand means eyeballing a select distinct anyway, and the model does the YAML formatting for free.
  • Surfacing candidates you forgot. It reliably flags "you have a created_at with no freshness or recency check" — useful prompts even when the exact test is wrong.

Where it is actively dangerous:

  • Hallucinated relationships tests. The model cannot know your foreign keys. It pattern-matches on column names and produces confident, plausible, wrong links. This is the number-one thing to never auto-accept.
  • False confidence on thresholds. Ask for a range test and it will happily assert order_total between 0 and 10000 — a bound it invented. The 10000 is fiction. It looks authoritative precisely because it is specific.
  • Business rules it cannot see. "A refunded order must have a non-null refund_date," "a subscription cannot have end_date < start_date." These live in your domain knowledge and your stakeholders' heads, not in column statistics. The model will sometimes guess them, and a lucky guess is more dangerous than an obvious miss because it teaches you to trust the next one.

Where LLM-generated dbt tests land on a trust-vs-effort grid — the cheap wins are real, the dangerous ones look just as confident.

The pattern underneath all three failures is the same: the LLM cannot distinguish a fact about your data from a property of your business. Statistics describe what is. Tests should encode what must be true. Those overlap, but they are not the same set, and the gap between them is exactly where you have to do the thinking.

Singular tests: draft, then verify the SQL

For logic that doesn't fit a generic test, the model writes a decent first draft of a singular test — a SQL file in tests/ that should return zero rows. Here it is a real time-saver because the boilerplate (the select that finds violations) is fiddly to get right:

-- tests/assert_refunded_orders_have_refund_date.sql
-- LLM draft, human-verified: confirmed with finance that this rule holds.
select
    order_id
from {{ ref('stg_orders') }}
where status = 'refunded'
  and refund_date is null

The model got the shape right. What it could not do was tell me whether the rule is true — I had to ask finance. The division of labor is clean: the LLM writes the SQL, you supply the truth. Run it the normal way:

dbt test --select stg_orders
dbt test --select assert_refunded_orders_have_refund_date

When not to bother

This workflow is not free, and there are cases where I skip it:

  • A handful of models. If you have five models, the profiling-and-reviewing overhead costs more than just writing the tests. The LLM wins at volume.
  • Highly regulated or business-critical logic. If a wrong test creates false confidence in a financial close or a compliance report, the asymmetry is brutal — write those by hand with the stakeholder in the room.
  • No profiling stats available. Without the data, you are back to name-based guessing and every failure mode above gets worse. If you can't profile, don't generate.

Takeaways

  • Feed stats, not just names. The group by and null/distinct counts are what separate description from hallucination. The prompt matters less than the context.
  • Auto-accept only what the stats prove: not_null, unique, and accepted_values drawn from the full observed domain.
  • Never auto-accept relationships tests or invented thresholds. Treat them as commented suggestions until a human confirms the key and the bound.
  • Make the model show its work. A # rationale: per test turns review from re-derivation into a quick check against a cited stat.
  • The LLM writes SQL; you supply the truth. Singular tests for business rules are a draft-plus-verify task, never a generate-and-commit one.
  • Use it for volume, skip it for small or high-stakes suites. The economics only work when there's real boilerplate to absorb.

I treat the LLM as a fast, tireless junior who has never met my stakeholders: great at the mechanical 70%, untrustworthy on the 30% that requires knowing the business. Gate accordingly and it genuinely speeds you up.
