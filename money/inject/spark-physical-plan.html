How to use .explain("formatted") to spot Exchange and SortMergeJoin, force a broadcast, and let AQE clean up the rest.

You wrote a join. It's two DataFrames, one key, nothing exotic. On your laptop with sample data it returns instantly. On the cluster, against real volume, the same join sits at 199/200 tasks for eleven minutes and then one straggler task runs for another twenty. The Spark UI shows gigabytes of "Shuffle Read." You add more executors. It gets marginally worse.

This is the single most common Spark performance trap, and you cannot reason your way out of it from the DataFrame code alone. The code says what you want. The physical plan says how Spark will actually do it — and that's where the hidden shuffle lives. Once you can read the plan, slow joins stop being mysterious. You look at the plan, you see the Exchange, you decide whether it's necessary, and you remove it if it isn't.

This article is a working session on exactly that: call .explain("formatted"), find the shuffle, tell SortMergeJoin from BroadcastHashJoin, and fix a needless shuffle with broadcast() and the right config. No fabricated benchmark numbers — just the plan output and the mechanism behind it.

First: print the plan

Every DataFrame has an .explain() method. The argument matters. The default dumps a dense tree; "formatted" gives you a numbered node list with details, which is far easier to read.

from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.getOrCreate()

orders = spark.read.parquet("s3://warehouse/orders")        # large fact table
customers = spark.read.parquet("s3://warehouse/customers")  # dimension table

joined = (
    orders.join(customers, on="customer_id", how="inner")
          .groupBy("country")
          .agg(F.sum("amount").alias("revenue"))
)

joined.explain("formatted")

.explain() is free — it does not run the job. It prints the optimized physical plan Spark would execute. Get in the habit of running it before any join you suspect, and definitely before you submit a long job to a shared cluster.

Reading the plan bottom-up

Spark plans read like a tree printed upside down: leaves (file scans) at the bottom, the final operator at the top. Data flows up. Here is the shape of a plan for the join above when both sides are treated as large:

== Physical Plan ==
* HashAggregate (10)
+- Exchange (9)                          <-- shuffle for the groupBy
   +- * HashAggregate (8)
      +- * Project (7)
         +- * SortMergeJoin Inner (6)     <-- the join strategy
            :- * Sort (4)
            :  +- Exchange (3)            <-- shuffle of orders (BIG)
            :     +- * Filter (2)
            :        +- Scan parquet orders (1)
            +- * Sort (5)
               +- Exchange  ...           <-- shuffle of customers
                  +- Scan parquet customers ...

Three things to internalize:

  • Exchange means shuffle. Every Exchange node is a point where data is written to disk, sent across the network, and re-read, partitioned by some key. Shuffles are the most expensive thing in most Spark jobs. The (3) node here shuffles the entire orders table across the network.
  • The leading * (a star/asterisk) means whole-stage codegen — Spark fused those operators into one compiled loop. Its absence isn't necessarily bad, but Exchange deliberately breaks codegen because it's a stage boundary.
  • The join operator names the strategy. Here it's SortMergeJoin, which requires both inputs sorted and co-partitioned on the join key — hence the two Exchange + Sort pairs feeding it.

In "formatted" mode you also get a numbered detail block below the tree. The one to read is the Exchange detail:

(3) Exchange
Input [3]: [customer_id#10, amount#11, ...]
Arguments: hashpartitioning(customer_id#10, 200), ENSURE_REQUIREMENTS, ...

hashpartitioning(customer_id, 200) tells you Spark is repartitioning by customer_id into 200 partitions (the default spark.sql.shuffle.partitions). That number is why your job had exactly 200 tasks.

How Spark decides between SortMergeJoin (shuffle both sides) and BroadcastHashJoin (ship the small side to every executor).

SortMergeJoin vs BroadcastHashJoin

These are the two join strategies you'll see most, and the difference is the whole game.

SortMergeJoin is Spark's default for joining two large tables. To match rows on a key, every row with a given key must end up on the same executor. Spark guarantees that by shuffling both sides — repartitioning each by the join key — then sorting and merging. Correct and scalable, but you pay for two full shuffles. That's the Exchange on each branch above.

BroadcastHashJoin avoids the shuffle entirely when one side is small enough. Spark collects the small table to the driver, broadcasts a copy to every executor, and each executor builds a hash table in memory. The large table is then joined in place — it never moves across the network. In the plan, the small side feeds a BroadcastExchange instead of a hashpartitioning Exchange, and the join node reads:

* BroadcastHashJoin [customer_id#10], [customer_id#42], Inner, BuildRight
+- ...big side, NO Exchange...
+- BroadcastExchange HashedRelationBroadcastMode(...)
   +- ...small side...

When you see BroadcastHashJoin and your big table has no Exchange above its scan, you've eliminated the expensive shuffle. That's the win.

The trade-off: the broadcast table must fit comfortably in the memory of the driver and every executor. Broadcast a table that's too big and you'll get driver OOMs or spark.driver.maxResultSize errors. Broadcast is for genuinely small dimension tables, not "medium" ones.

Make Spark broadcast: the threshold and the explicit hint

Spark broadcasts automatically when it estimates one side is smaller than spark.sql.autoBroadcastJoinThreshold, which defaults to 10MB (10485760 bytes). Two ways to get the broadcast you want:

1. Raise the threshold if your dimension table is, say, 60MB and you know it fits:

spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 64 * 1024 * 1024)  # 64MB

2. Force it with an explicit hint — the more reliable option, because Spark's size estimate is often wrong. Reading from Parquet, Spark may not have accurate row counts until runtime, so it guesses the table is huge and falls back to SortMergeJoin even when the table is tiny:

from pyspark.sql.functions import broadcast

joined = (
    orders.join(broadcast(customers), on="customer_id", how="inner")
          .groupBy("country")
          .agg(F.sum("amount").alias("revenue"))
)

joined.explain("formatted")

The SQL equivalent is a hint comment:

SELECT /*+ BROADCAST(c) */ ...
FROM orders o JOIN customers c ON o.customer_id = c.customer_id

Re-run .explain("formatted") and confirm the join node now says BroadcastHashJoin and the orders scan has no Exchange above it. Always verify in the plan — a hint is a request, and Spark will silently ignore it if the table genuinely exceeds the broadcast size limits.

To disable auto-broadcast entirely (useful when debugging, or when broadcasts are causing OOMs), set the threshold to -1.

The before/after: fixing a needless shuffle

Here's the realistic situation. orders is hundreds of GB; customers is a few MB. But because both are Parquet and Spark's stats are stale, the optimizer estimates customers as large and picks SortMergeJoin. You shuffle hundreds of GB of orders across the network for no reason — the small side could have been broadcast.

Before — the plan you don't want, with orders going through an Exchange:

* SortMergeJoin Inner (6)
:- * Sort (4)
:  +- Exchange hashpartitioning(customer_id, 200)   <-- shuffling ALL of orders
:     +- Scan parquet orders (1)
+- * Sort (5)
   +- Exchange hashpartitioning(customer_id, 200)
      +- Scan parquet customers ...

After — wrap the small side in broadcast():

* BroadcastHashJoin [customer_id], [customer_id], Inner, BuildRight (4)
:- * Filter (2)
:  +- Scan parquet orders (1)                        <-- NO Exchange, stays put
+- BroadcastExchange HashedRelationBroadcastMode (3)
   +- Scan parquet customers ...

The mechanism behind the speedup is concrete, not magic: the big table's Exchange is gone, so you no longer write hundreds of GB to shuffle storage and re-read it across the network. The only data that moves is one small copy of customers. The number of shuffle stages for the join drops from one (or effectively two passes of data) to zero. If your job was previously bottlenecked on shuffle I/O — which is what "stuck at 199/200 tasks" usually means — this is the fix.

If after broadcasting you're still seeing slowness, the bottleneck may have moved downstream to spilling during the aggregation or sort. That's a different diagnosis with its own plan signatures — see the companion piece on reading and fixing Spark spills for how memory pressure shows up.

Let AQE clean up what's left

Adaptive Query Execution (AQE) is on by default since Spark 3.2 (spark.sql.adaptive.enabled=true). It re-optimizes the plan at runtime using the actual sizes measured after each shuffle completes — information the optimizer simply doesn't have up front. Two AQE behaviors matter most for joins:

Coalescing shuffle partitions. You set spark.sql.shuffle.partitions=200 and your data only needs 12. Without AQE you get 200 tasks, most processing almost nothing — pure scheduling overhead. AQE coalesces small adjacent post-shuffle partitions into fewer, fuller ones automatically (spark.sql.adaptive.coalescePartitions.enabled). You stop hand-tuning the partition count for every job.

Skew join handling. If one join key is wildly over-represented — a null customer_id, or one mega-account that's 40% of orders — that key's partition becomes a straggler task while the other 199 finish. That's the classic "199 done, 1 running for 20 minutes." AQE detects the skewed partition at runtime and splits it into several sub-partitions that run in parallel (spark.sql.adaptive.skewJoin.enabled).

spark.conf.set("spark.sql.adaptive.enabled", True)             # default on in 3.2+
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", True)
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", True)

What AQE does at runtime once the first shuffle's real sizes are known.

One catch when reading plans with AQE on: .explain() shows the plan before runtime adaptation, and you'll see an AdaptiveSparkPlan wrapper node at the top, often marked isFinalPlan=false. To inspect what AQE actually did, look at the SQL tab in the Spark UI after the query runs, or call .explain() on the result of an action. The pre-execution plan is still the right place to confirm your broadcast landed, because that decision can happen before runtime — but don't expect the pre-run plan to reflect coalescing or skew splits.

When NOT to reach for broadcast

Reading the plan is always worth it. The fixes have limits worth naming:

  • Both sides genuinely large? Broadcast is off the table — neither fits in executor memory. SortMergeJoin with a shuffle is the correct plan. Your levers there are partition count, AQE skew handling, and pre-bucketing the tables on the join key so the shuffle can be skipped.
  • Broadcasting something "kind of big" to dodge a shuffle is a classic foot-gun: you trade a shuffle for driver/executor OOMs and cluster-wide memory pressure. Keep broadcasts to real dimension tables.
  • Raising autoBroadcastJoinThreshold globally affects every join in the session. Prefer per-join broadcast() hints so you don't accidentally broadcast a medium table somewhere else.
  • A shuffle isn't always the enemy. A groupBy/aggregate or a window over a large dataset needs its shuffle. The goal is removing the needless ones, not all of them.

Takeaways

  • Run .explain("formatted") before you trust a join. It's free, it doesn't execute the job, and it tells you the truth the DataFrame code hides.
  • Exchange = shuffle. Find it. Ask whether it's necessary. An Exchange over your biggest table is the first thing to question.
  • SortMergeJoin shuffles both sides; BroadcastHashJoin ships the small side and shuffles nothing. The join node name tells you which you got.
  • Force the broadcast with broadcast(df) rather than trusting Spark's size estimate from Parquet — then re-read the plan to confirm the big table's Exchange is gone.
  • autoBroadcastJoinThreshold defaults to 10MB; set it to -1 to disable, raise it (carefully) for known-small tables.
  • AQE (default-on in Spark 3.2+) coalesces tiny partitions and splits skewed ones at runtime — but .explain() shows the pre-adaptation plan, so verify runtime behavior in the Spark UI.

The discipline that pays off: never tune a Spark join by changing executor counts and hoping. Read the plan, name the strategy, kill the shuffle you can't justify.
