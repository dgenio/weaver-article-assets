We would never ship application code with zero tests and just hope the asserts hold in production. What a dbt test actually is

This is the one mental model that makes everything else click: a dbt test is a SQL query that returns the rows that violate a rule. Zero rows returned means the test passes. One or more rows means it fails.

That's it. A not_null test on order_id compiles, roughly, to:

select order_id
from analytics.fct_orders
where order_id is null

If that query returns nothing, the column has no nulls and the test passes. If it returns 14 rows, you have 14 nulls and the test fails with a count. Once you internalize "tests are queries for the bad rows," writing your own custom checks stops feeling like magic.

Generic tests: the four you'll use constantly

Generic tests are reusable tests you attach to columns or models in a schema.yml (often named after the model, e.g. _models.yml). dbt ships four out of the box, and they cover most of your day-to-day needs.

version: 2

models:
  - name: fct_orders
    columns:
      - name: order_id
        description: "Primary key. One row per order."
        data_tests:
          - unique
          - not_null

      - name: status
        data_tests:
          - accepted_values:
              values: ['placed', 'shipped', 'completed', 'returned']

      - name: customer_id
        data_tests:
          - not_null
          - relationships:
              to: ref('dim_customers')
              field: customer_id
  • unique — no duplicate values. This is the one that would have caught my Monday incident.
  • not_null — no nulls. Put it on every key and every column a downstream calculation depends on.
  • accepted_values — the column only contains values from a known set. Perfect for status enums, country codes, plan tiers. It catches the day someone introduces a STATUS = 'cancelled_v2' upstream.
  • relationships — referential integrity. Every customer_id in fct_orders must exist in dim_customers. This is your defense against silently-dropped joins and orphaned foreign keys.

Note on syntax: dbt now prefers data_tests: over the older tests: key. tests: still works and you'll see it everywhere in the wild, but use data_tests: in new code.

Run them with dbt test, or — much better — with dbt build, which runs each model and then immediately runs its tests, stopping a broken model from feeding its children.

Singular tests: when the rule is too specific to be generic

Sometimes a rule is a one-off business invariant that doesn't generalize. For that, dbt has singular tests: plain .sql files in your tests/ directory. Same contract as always — write a query that returns the rows you don't want to exist.

tests/assert_order_total_is_positive.sql:

-- Fails if any completed order has a non-positive total.
select
    order_id,
    order_total
from {{ ref('fct_orders') }}
where status = 'completed'
  and order_total <= 0

If a completed order ever has a total of 0 or less, this returns those rows and the test fails with their count. No macro, no config — just a question phrased as "show me the rows that should be impossible."

Singular tests are also where you encode reconciliation logic: "the sum of line_item revenue must equal the order header revenue," "no two active subscriptions for the same user." If you can write the violating-rows query, you can test it.

dbt_utils: the tests the built-ins are missing

The dbt_utils package (from dbt Labs) adds the generic tests you'll reach for once the basics are in place. Add it to packages.yml and run dbt deps:

packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.1.0", "<2.0.0"]

Three that earn their keep:

models:
  - name: fct_orders
    data_tests:
      # Cross-column business rule, evaluated per row.
      - dbt_utils.expression_is_true:
          expression: "order_total = subtotal + tax - discount"

      # The two tables should have the same number of rows.
      - dbt_utils.equal_rowcount:
          compare_model: ref('stg_orders')

    columns:
      - name: order_id
        data_tests:
          # Composite uniqueness across multiple columns.
          - dbt_utils.unique_combination_of_columns:
              combination_of_columns:
                - order_id
                - line_number
  • expression_is_true lets you assert any boolean SQL expression — the most flexible generic test there is, ideal for arithmetic invariants like order_total = subtotal + tax - discount.
  • equal_rowcount (and its sibling fewer_rows_than) catches fan-out and row loss between two models — exactly the failure mode behind my doubled-revenue incident.
  • unique_combination_of_columns is unique for composite grains, e.g. one row per (order_id, line_number).

There's also dbt_utils.recency, which checks that a table has rows newer than a given interval — a lightweight freshness guard for models (as opposed to sources, which have their own freshness mechanism we'll cover below).

Severity: not every failure should page someone

A not_null on a primary key should hard-fail. A check that "we usually get more than 1,000 orders a day" probably shouldn't block a deploy at 3 a.m. just because yesterday was a holiday. dbt lets you tune this with severity.

columns:
  - name: customer_email
    data_tests:
      - not_null:
          config:
            severity: warn   # log a warning, do not fail the build

  - name: daily_order_count
    data_tests:
      - dbt_utils.expression_is_true:
          expression: "daily_order_count > 0"
          config:
            # Warn if a few rows fail; only error past a threshold.
            severity: error
            warn_if: ">0"
            error_if: ">100"
  • severity: error (the default) → the test fails and dbt exits non-zero. In CI, that blocks the merge.
  • severity: warn → the test logs a warning but the build keeps going.
  • warn_if / error_if → threshold expressions on the failing-row count, so you can warn on a trickle and error on a flood.

Use warn for soft data-quality signals and error for invariants that must hold. A useful discipline: a warn should be something you'll actually triage, not noise you'll learn to ignore. If nobody acts on a warning, delete it or promote it.

where and limit: scope the test, save the warehouse

Two config options keep tests cheap and focused.

where filters the rows a test examines. Don't test historical partitions you've already validated and frozen — test today's data:

- name: event_id
  data_tests:
    - not_null:
        config:
          where: "event_date >= dateadd('day', -3, current_date)"

limit caps how many failing rows dbt stores when it materializes failures, which keeps the failures table small on a wide breakage:

- name: order_id
  data_tests:
    - unique:
        config:
          limit: 500
          store_failures: true   # persist failing rows to a table for debugging

store_failures: true writes the violating rows to a table so you can inspect which rows broke instead of just seeing a count — pair it with limit so a million-row failure doesn't write a million-row debug table.

Source freshness: catch the pipeline that quietly stopped

Tests validate content. Freshness validates timeliness — "is this raw table even being updated?" Define it on your sources:

version: 2

sources:
  - name: shop
    database: raw
    schema: shop
    tables:
      - name: orders
        loaded_at_field: _loaded_at
        freshness:
          warn_after: {count: 12, period: hour}
          error_after: {count: 24, period: hour}

loaded_at_field is the column dbt uses to measure age (often an ingestion timestamp). Run dbt source freshness. If the newest row is older than 12 hours you get a warning; past 24 hours, an error. This is how you find out the Fivetran connector died before the stakeholder finds out the numbers are stale. (For data platforms that track table metadata, dbt can use that instead of a column — but an explicit loaded_at_field is the portable default.)

For freshness on derived models rather than raw sources, reach back for dbt_utils.recency — same idea, applied downstream.

Testing in CI: where this all pays off

A test suite that only runs on your laptop is theater. The payoff is gating every pull request. The core idea: on a PR, build and test only what changed plus its descendants, using dbt's state comparison.

# In CI, against a deferred production manifest:
dbt build --select state:modified+ --defer --state ./prod-manifest
  • state:modified+ selects models that changed versus the stored production manifest, plus everything downstream (+). A staging change re-tests every mart built on it.
  • dbt build interleaves runs and tests, so a model that fails its test won't poison its children within the same invocation.
  • --defer lets unchanged upstream models resolve to production, so you don't rebuild the whole warehouse on every PR.

Because failed error-severity tests make dbt exit non-zero, your CI provider marks the check red and the merge is blocked. That's the whole game: bad data becomes a failed build, not a Monday incident. This pairs naturally with incremental models — your CI run should exercise the incremental logic, not just a full refresh.

A minimum suite per layer

You don't need 40 tests on every model. You need the right tests at each layer, with guarantees tightening as data flows toward the people who consume it.

Sources (raw): freshness (warn_after/error_after), plus not_null and unique on the primary key. Catch ingestion problems at the door.

Staging (stg_): unique + not_null on the key, accepted_values on every enum/status column, not_null on columns downstream math depends on. Staging is where you normalize, so it's where you assert the shape is clean.

Intermediate (int_): relationships for foreign-key integrity, dbt_utils.expression_is_true for business rules, and a row-count sanity check (equal_rowcount / fewer_rows_than) across a join to catch fan-out.

Marts (fct_ / dim_): unique + not_null on the grain key, relationships to every dimension, expression_is_true on financial invariants (totals reconcile), and severity: warn on soft volume checks. Marts feed dashboards, so the grain guarantee is non-negotiable.

A pragmatic minimum test suite mapped to each model layer. Tighter guarantees accumulate as data flows from raw sources to marts.

When not to test (and other honest trade-offs)

  • Don't test what you can't act on. A test that fails and gets muted every run is worse than no test — it trains the team to ignore red.
  • Tests cost warehouse compute. Every test is a query. On large tables, scope with where, lean on incremental logic, and don't put a full-table unique on a billion-row event log if a recent-partition where clause gives you the same confidence.
  • Generic tests aren't free uniqueness constraints. dbt tests run after the data lands; they detect bad data, they don't prevent it from being written. For hard guarantees, combine them with warehouse constraints or dbt's contracts feature.
  • Don't mistake green tests for correct logic. Tests check the invariants you thought to write. They won't catch a wrong JOIN that still produces unique, non-null keys. Code review and a few targeted singular tests still matter.

A reasonable target is the suite above: a handful of high-signal tests per model, all running in dbt build, gated in CI. That's the 20% of effort that prevents 80% of the Monday incidents.

Takeaways

  • A dbt test is just a query for the bad rows. Zero rows = pass. Internalize this and custom tests become trivial.
  • Start with the four built-ins — unique, not_null, accepted_values, relationships — on every key and enum.
  • Add dbt_utils for expression_is_true, equal_rowcount, and composite uniqueness; add tests/*.sql singular tests for one-off business invariants.
  • Tune severity (warn vs error, warn_if/error_if) so only real problems block deploys, and prune warnings nobody triages.
  • Scope with where and limit, and use store_failures to debug which rows broke.
  • Validate timeliness with source freshness, not just content.
  • Gate it in CI with dbt build --select state:modified+ so bad data fails the build instead of the dashboard.
