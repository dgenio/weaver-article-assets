Running totals, dedup, sessionization, and "compare this row to the previous one" — all without a single self-join. A practical, copy-paste guide.

SQL window functions compute an aggregate across rows related to the current row without collapsing them — the entire feature lives in the OVER keyword, where PARTITION BY groups rows, ORDER BY sequences them, and a ROWS/RANGE frame slices which rows the current row can see. They replace slow, error-prone self-joins for running totals (SUM() OVER), ranking (ROW_NUMBER/RANK/DENSE_RANK), row-to-row comparisons (LAG/LEAD), deduplication, and sessionization in a single pass. The examples run as-is on SQLite 3.25+ and port to PostgreSQL, BigQuery, Snowflake, and MySQL 8+ with minor syntax tweaks. You have a table of orders. Product asks a deceptively simple question: "For each customer, what was their second-largest order, and how much bigger was it than the one before it?"

If your instinct is to reach for a self-join — join the table to itself, filter, group, MAX(), then join again to get the previous value — stop. That query will be slow, unreadable, and almost certainly wrong on ties. This is the exact moment that separates engineers who get by in SQL from engineers who are fluent in it.

The fluent answer is a window function. One pass over the data, no self-join, and it reads almost like the English sentence Product asked for.

Window functions are, in my experience, the single most underused feature in everyday SQL — and the one I most reliably see come up in senior data engineering interviews. Let's fix that. Everything below runs as-is on SQLite 3.25+, and the same logic ports to PostgreSQL, BigQuery, Snowflake, and MySQL 8+ with minor syntax tweaks (noted inline) — chiefly Snowflake's lack of the WINDOW clause and engine-specific timestamp functions.

The sample table

We'll use one small events table for every example so you can paste and run as you read.

CREATE TABLE events (
  user_id   INTEGER,
  event_id  INTEGER,
  amount    REAL,
  ts        TEXT      -- timestamp, kept as text for portability
);

INSERT INTO events VALUES
 (1, 101, 12.0, '2026-01-01 09:00'),
 (1, 102, 18.0, '2026-01-01 09:05'),
 (1, 103,  5.0, '2026-01-01 10:30'),
 (1, 104,  9.0, '2026-01-01 10:33'),
 (2, 201, 40.0, '2026-01-01 11:00'),
 (2, 202, 40.0, '2026-01-01 11:02'),  -- deliberate tie
 (2, 203, 25.0, '2026-01-01 12:45'),
 (3, 301,  7.0, '2026-01-01 08:00');

The mental model: GROUP BY collapses, OVER preserves

A window function computes an aggregate across a set of rows related to the current row — but, unlike GROUP BY, it does not collapse those rows. You get the aggregate and keep every original row.

The whole feature lives in one keyword: OVER. Inside the parentheses you can declare up to three things:

  • PARTITION BY — split the rows into groups (like GROUP BY, but the rows survive).
  • ORDER BY — order the rows within each partition (this is what makes "running" and "previous row" possible).
  • a frame (ROWS/RANGE BETWEEN ...) — which slice of the partition the current row can see.

GROUP BY collapses rows into one summary per group; a window function keeps every row and attaches the aggregate alongside it. Same PARTITION, different output shape.

That's it. Everything else is which function you put in front of OVER.

Ranking: ROW_NUMBER vs RANK vs DENSE_RANK

These three look interchangeable until you hit a tie — and ties are where juniors ship bugs. Run this:

SELECT user_id, event_id, amount,
  ROW_NUMBER() OVER w AS rn,
  RANK()       OVER w AS rnk,
  DENSE_RANK() OVER w AS drnk
FROM events
WINDOW w AS (PARTITION BY user_id ORDER BY amount DESC)
ORDER BY user_id, amount DESC;

(That WINDOW w AS (...) clause lets you name a window once and reuse it — handy when three functions share the same definition. Supported in PostgreSQL, MySQL 8+, SQLite, and BigQuery; Snowflake doesn't support the named WINDOW clause, so inline the OVER (...) instead.)

For user 2, who has two orders of exactly 40.0, the three functions diverge:

amount   ROW_NUMBER   RANK   DENSE_RANK
------   ----------   ----   ----------
40.0     1            1      1
40.0     2            1      1
25.0     3            3      2
  • ROW_NUMBER is arbitrary-but-unique — the two tied rows get 1 and 2, but which one gets 1 is undefined unless you add a tiebreaker to the ORDER BY.
  • RANK gives ties the same rank, then skips (1, 1, 3).
  • DENSE_RANK gives ties the same rank, then doesn't skip (1, 1, 2).

Rule of thumb: use ROW_NUMBER when you need exactly one row per group (dedup, "latest record"), RANK/DENSE_RANK when ties genuinely should share a position (leaderboards). And when you use ROW_NUMBER for dedup, always add a deterministic tiebreaker — ORDER BY amount DESC, event_id — or your "pick one" logic is a coin flip that flips differently on every run.

Running totals with SUM() OVER

Add ORDER BY inside OVER and an aggregate becomes a running aggregate. This is the running-total query, no self-join in sight:

SELECT user_id, event_id, amount,
  SUM(amount) OVER (
    PARTITION BY user_id
    ORDER BY ts
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM events
ORDER BY user_id, ts;

For user 1 this produces 12 -> 30 -> 35 -> 44: each row sees itself plus everything before it in time. Swap SUM for AVG, COUNT, MAX, or MIN and you get moving aggregates for free.

The frame is the slice of the partition each row can see. ROWS counts physical rows; RANGE groups rows that tie on the ORDER BY value — which is why duplicate keys behave differently.

"Compare to the previous row" with LAG and LEAD

LAG reaches backward, LEAD reaches forward — within the partition, along the ORDER BY. This is how you compute deltas, gaps, and "time since last event" without correlated subqueries.

SELECT user_id, event_id, ts,
  LAG(ts)  OVER (PARTITION BY user_id ORDER BY ts) AS prev_ts,
  amount - LAG(amount, 1, 0) OVER (PARTITION BY user_id ORDER BY ts) AS delta_vs_prev
FROM events
ORDER BY user_id, ts;

Two things worth knowing: the first row of each partition has no predecessor, so LAG returns NULL unless you pass a default — LAG(amount, 1, 0) means "1 row back, or 0 if none." And LAG/LEAD are how you answer the Product question from the intro: rank with ROW_NUMBER, then LAG to get the order before it.

Frames: ROWS vs RANGE (the gotcha seniors know cold)

You saw ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW above. The alternative is RANGE, and the difference is subtle until it corrupts a number in production.

  • ROWS counts physical rows: "the 3 rows before me and me."
  • RANGE counts logical peers: rows that tie on the ORDER BY value are treated as a single unit.

Here's the proof, using user 2's two tied 40.0 rows:

SELECT user_id, event_id, amount,
  SUM(amount) OVER (PARTITION BY user_id ORDER BY amount
                    ROWS  BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS rows_sum,
  SUM(amount) OVER (PARTITION BY user_id ORDER BY amount
                    RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS range_sum
FROM events WHERE user_id = 2
ORDER BY amount;

amount   rows_sum   range_sum
------   --------   ---------
25.0     25         25
40.0     65         **105**
40.0     105        **105**

With ROWS, the two 40 rows accumulate one at a time (65, then 105). With RANGE, both tied rows see the same frame end and both report 105. Neither is wrong — but if you wanted a row-by-row running total and you let the engine default to RANGE, your numbers will silently double-count peers.

The most important default to memorize: if you write ORDER BY inside OVER but omit the frame entirely, the SQL standard defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. For most running totals you actually want ROWS. Be explicit.

FIRST_VALUE / LAST_VALUE — and why LAST_VALUE lies by default

FIRST_VALUE and LAST_VALUE grab a value from the edge of the frame. FIRST_VALUE usually does what you expect. LAST_VALUE is the most common window-function bug I see:

SELECT user_id, ts, amount,
  FIRST_VALUE(amount) OVER (PARTITION BY user_id ORDER BY ts) AS first_amt,
  LAST_VALUE(amount)  OVER (PARTITION BY user_id ORDER BY ts) AS last_amt_WRONG,
  LAST_VALUE(amount)  OVER (PARTITION BY user_id ORDER BY ts
       ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS last_amt_right
FROM events WHERE user_id = 1
ORDER BY ts;

Because the default frame ends at CURRENT ROW, last_amt_WRONG just returns the current row's value on every row — useless. To get the partition's actual last value you must extend the frame to UNBOUNDED FOLLOWING. This single gotcha trips up almost everyone the first time.

Two patterns you'll use constantly

Deduplication — keep the latest row per key. This is the cleanest dedup in SQL, and it generalizes to "keep the first/best/most-recent of anything":

WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC, event_id DESC) AS rn
  FROM events
)
SELECT user_id, event_id, ts
FROM ranked
WHERE rn = 1;

Sessionization — group events into sessions with a 30-minute inactivity timeout. This is the classic "gaps and islands" problem, and it's a two-step window pattern: flag each row that starts a new session (gap > 30 min or no predecessor), then take a running sum of those flags to assign a session number.

WITH gaps AS (
  SELECT user_id, event_id, ts,
    (julianday(ts) - julianday(
        LAG(ts) OVER (PARTITION BY user_id ORDER BY ts))) * 24 * 60 AS gap_min
  FROM events
),
flagged AS (
  SELECT *,
    CASE WHEN gap_min IS NULL OR gap_min > 30 THEN 1 ELSE 0 END AS is_new_session
  FROM gaps
)
SELECT user_id, event_id, ts,
  SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY ts
                            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS session_seq
FROM flagged
ORDER BY user_id, ts;

For user 1, the 85-minute jump from 09:05 to 10:30 flips is_new_session to 1, so the running sum bumps session_seq from 1 to 2 — splitting their four events into two sessions. (The julianday() math is SQLite-specific for timestamp diffs; on PostgreSQL use EXTRACT(EPOCH FROM ts - LAG(ts) OVER ...)/60, on BigQuery use TIMESTAMP_DIFF.) The window logic is identical everywhere.

When NOT to use window functions

Fluency includes knowing the limits:

  • You can't filter on a window function in WHERE. Window functions are computed after WHERE/GROUP BY, so WHERE ROW_NUMBER() OVER (...) = 1 is a syntax error. Wrap it in a CTE or subquery and filter the outer query — that's why the dedup pattern uses a WITH.
  • They don't reduce row count. If you genuinely want one aggregated row per group, plain GROUP BY is simpler and usually cheaper. Don't window-then-DISTINCT to fake an aggregation.
  • Ordered windows aren't free. An ORDER BY inside OVER generally forces a sort. On large tables, partitioning on a column that matches an existing index (or clustering key) matters. On columnar warehouses, an unbounded PARTITION BY with high cardinality can blow up a query the same way a bad join can — see the partitioning and shuffle traps in bigquery-antipatterns for how this plays out on a warehouse bill.
  • Watch the frame default. As shown above, the implicit RANGE default has bitten more analysts than I can count. If you mean physical rows, say ROWS.

Takeaways

  • OVER is the whole feature. PARTITION BY groups, ORDER BY sequences, the frame slices. Memorize those three roles.
  • Ties decide your function. ROW_NUMBER (unique), RANK (skips), DENSE_RANK (no skip). For dedup, use ROW_NUMBER with a deterministic tiebreaker.
  • Running totals = aggregate + ORDER BY + explicit ROWS frame. Don't trust the default frame.
  • LAG/LEAD kill self-joins for deltas, gaps, and previous/next-row comparisons.
  • LAST_VALUE needs UNBOUNDED FOLLOWING or it lies.
  • Dedup and sessionization are just ROW_NUMBER and a running sum of flags — two patterns that cover a huge share of real analytics work.
  • You can't WHERE on a window — wrap it in a CTE and filter outside.

Window functions aren't an advanced curiosity; they're table stakes for anyone writing analytics SQL professionally. FAQ

What is the difference between GROUP BY and a window function (OVER)? GROUP BY collapses rows into one summary per group, while a window function computes the aggregate across a related set of rows but keeps every original row, attaching the result alongside it. If you genuinely want one aggregated row per group, plain GROUP BY is simpler and usually cheaper.

What is the difference between ROW_NUMBER, RANK, and DENSE_RANK? They diverge on ties: ROW_NUMBER is arbitrary-but-unique (tied rows get 1, 2), RANK gives ties the same rank then skips (1, 1, 3), and DENSE_RANK gives ties the same rank without skipping (1, 1, 2). Use ROW_NUMBER with a deterministic tiebreaker for dedup or "latest record," and RANK/DENSE_RANK for leaderboards where ties should share a position.

What is the difference between ROWS and RANGE frames? ROWS counts physical rows ("the 3 before me and me"), while RANGE treats rows that tie on the ORDER BY value as a single unit, so tied rows see the same frame and report the same total. This matters because if you write ORDER BY inside OVER but omit the frame, the SQL standard defaults to RANGE, which can silently double-count peers when you wanted a row-by-row running total — be explicit with ROWS.

Why does LAST_VALUE return the wrong result by default? Because the default frame ends at CURRENT ROW, LAST_VALUE just returns the current row's value on every row, which is useless. To get the partition's actual last value you must extend the frame to ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.

Why can't I filter on a window function in a WHERE clause? Window functions are computed after WHERE/GROUP BY, so WHERE ROW_NUMBER() OVER (...) = 1 is a syntax error. Wrap the window function in a CTE (WITH) or subquery and filter the outer query instead — that is exactly why the deduplication pattern uses a WITH block.
