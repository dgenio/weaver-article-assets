On-demand vs. slots, partitioning, clustering, and the one preview button that stops $60 mistakes before they run.

You can cut a BigQuery bill in half without rewriting a single query because on-demand pricing charges for bytes scanned ($6.25/TB in the US, first 1 TB free monthly), not rows returned — so the fix is laying tables out to scan less. The highest-payoff moves: dry-run every query (free) with a --maximum_bytes_billed cap, partition big tables by date (a one-day query can scan ~1/730th of the data), add require_partition_filter = TRUE, cluster by the columns you filter on, and stop using SELECT *. Layer in materialized views, table/partition expiration, long-term storage, and BI Engine for dashboards. Last quarter a teammate pinged me on Slack: "Why is our BigQuery bill up 40% and nobody shipped anything new?"

Nobody had shipped anything new. That was the problem. A scheduled dashboard had quietly been running SELECT * against a 2 TB events table every 15 minutes, and a new analyst had cloned it three times. On BigQuery's on-demand model — where you pay for bytes scanned, not rows returned — that's a money fire that produces zero new insight.

The good news: almost every BigQuery cost problem is a bytes-scanned problem, and you can fix the biggest ones without rewriting your business logic. You change how tables are laid out and how queries touch them. Same SQL shape, a fraction of the bill.

Here's the playbook I actually use, roughly in order of effort-to-payoff.

First, understand what you're paying for

BigQuery has two pricing models, and confusing them is where a lot of money leaks.

  • On-demand: you pay per TB of data your queries scan. The US multi-region list price is $6.25 per TB, and the first 1 TB each month is free (always check the current pricing page — rates vary by region and change over time). You don't manage any infrastructure; you just get billed for scans.
  • Capacity (Editions / slots): you buy compute capacity measured in slots. Standard Edition pay-as-you-go is around $0.04 per slot-hour, with discounts for 1- and 3-year commitments. You pay for compute time, not bytes scanned.

Most small-to-medium teams start on on-demand, and that's usually right. The mental switch is this: on-demand punishes big scans; capacity punishes long-running or highly concurrent compute. If your bill is dominated by a few enormous queries scanning huge tables, the techniques below attack that directly. If you're a large org running thousands of concurrent queries all day, a slot reservation may be cheaper and more predictable — but don't reach for it before you've cleaned up the scans, or you'll just buy slots to power waste.

Storage is billed separately and is comparatively cheap, but it has its own lever we'll get to.

Lever 0: Dry-run every query before it runs (free, do it today)

This is the single cheapest habit you can build, and it costs nothing. BigQuery can tell you exactly how many bytes a query will scan before you run it.

In the console SQL editor, type your query and look at the top-right validator — it shows something like "This query will process 2.1 TB when run." That little line is your invoice preview.

From the command line, use --dry_run:

bq query --use_legacy_sql=false --dry_run \
'SELECT user_id, event_name
 FROM `my_project.analytics.events`
 WHERE event_date = "2026-06-01"'

It returns the byte estimate without scanning anything or charging you. In Python:

from google.cloud import bigquery

client = bigquery.Client()
job_config = bigquery.QueryJobConfig(dry_run=True, use_query_cache=False)

sql = """
SELECT user_id, event_name
FROM `my_project.analytics.events`
WHERE event_date = "2026-06-01"
"""

job = client.query(sql, job_config=job_config)
gb = job.total_bytes_processed / 1024**3
print(f"This query will scan ~{gb:.2f} GB")

Wire that into CI for your scheduled queries and you'll catch the 2 TB accidents before they bill. Pair it with a maximum bytes billed guardrail so a runaway query fails instead of charging you:

bq query --use_legacy_sql=false --maximum_bytes_billed=100000000000 \
'SELECT ... '   # caps this query at 100 GB; it errors out if it would exceed that

Lever 1: Stop scanning columns you don't read

SELECT * is the most expensive two characters in BigQuery. Because storage is columnar, BigQuery only reads the columns you name. SELECT * reads every column — including the giant JSON blob and the nested array nobody touches.

On-demand pricing bills you for bytes scanned, not rows returned. A query that filters on a partitioned + clustered column reads only the blocks it needs; the same query on an unpartitioned table reads everything.

The fix is boring and effective: name your columns.

-- Don't
SELECT * FROM `my_project.analytics.events` WHERE event_date = "2026-06-01";

-- Do
SELECT user_id, event_name, event_date
FROM `my_project.analytics.events`
WHERE event_date = "2026-06-01";

If you genuinely need to eyeball a wide table, SELECT * ... LIMIT 10 does not save you money on a normal table — BigQuery still scans the full columns to satisfy the query (the LIMIT is applied after). Use the table preview tab in the console instead, which is free. (More on this and other traps in my companion piece on BigQuery antipatterns.)

Lever 2: Partition by date — the single biggest lever

If you remember one thing, remember this: partition your big tables by date.

Partitioning splits a table into segments (typically one per day) based on a column. When a query filters on that column, BigQuery prunes the partitions that can't match and scans only the rest. A query for one day out of two years of history scans roughly 1/730th of the data instead of all of it. That's not a tuning tweak — it's an order-of-magnitude change in bytes scanned.

Create a partitioned table:

CREATE TABLE `my_project.analytics.events`
(
  user_id     STRING,
  event_name  STRING,
  event_date  DATE,
  payload     STRING
)
PARTITION BY event_date;

For an existing table, you can repartition by writing into a new partitioned table:

CREATE TABLE `my_project.analytics.events_partitioned`
PARTITION BY event_date AS
SELECT * FROM `my_project.analytics.events`;

Now the filter that used to scan everything scans one slice:

SELECT user_id, event_name
FROM `my_project.analytics.events_partitioned`
WHERE event_date BETWEEN "2026-06-01" AND "2026-06-07";  -- 7 partitions, not the whole table

Gotcha that trips everyone up: the pruning only works if your filter is on the partition column directly. WHERE event_date = "2026-06-01" prunes. WHERE CAST(event_date AS STRING) = "2026-06-01" does not — wrapping the partition column in a function defeats pruning and you scan everything. Keep the partition column bare on one side of the comparison.

Lever 3: Require a partition filter so nobody forgets

Partitioning helps only when people filter on the partition column. The way to guarantee it is to make the filter mandatory:

CREATE TABLE `my_project.analytics.events`
(
  user_id    STRING,
  event_name STRING,
  event_date DATE
)
PARTITION BY event_date
OPTIONS (require_partition_filter = TRUE);

Now any query without a predicate on event_date simply fails with an error instead of silently scanning two years of data. It turns a "we hope people filter" convention into an enforced rule — exactly the kind of guardrail that prevents the 3 a.m. surprise bill. You can toggle it on an existing table with ALTER TABLE ... SET OPTIONS (require_partition_filter = TRUE).

The trade-off: ad-hoc exploration gets slightly more annoying because every query must include the filter. For high-volume production tables, that annoyance is the point.

Lever 4: Cluster to prune within a partition

Clustering sorts the data inside each partition by columns you choose (up to four). When you filter on a clustering column, BigQuery skips the blocks that can't contain matching rows — pruning within a partition the way partitioning prunes across days.

CREATE TABLE `my_project.analytics.events`
(
  user_id    STRING,
  event_name STRING,
  event_date DATE
)
PARTITION BY event_date
CLUSTER BY event_name, user_id;

A query that filters on both the partition and a clustering column gets two levels of pruning:

SELECT user_id
FROM `my_project.analytics.events`
WHERE event_date = "2026-06-01"     -- partition pruning
  AND event_name = "purchase";       -- cluster pruning

One honest caveat: with clustering, BigQuery often can't tell you the exact bytes up front in the dry-run estimate, because block pruning happens at execution time. The estimate may look like the full partition while the actual billed bytes come in lower. That's expected — don't panic at the preview number on a clustered table.

Clustering shines on high-cardinality filter/aggregation columns (user IDs, event names, country). It does little for columns you never filter on, so cluster by what your WHERE and GROUP BY clauses actually use.

Lever 5: Materialized views for repeated heavy aggregations

If the same expensive GROUP BY runs on a schedule — daily active users, revenue by region — you're paying to recompute the same answer over and over. A materialized view precomputes and incrementally refreshes the result, and BigQuery can even rewrite queries against the base table to use it automatically.

CREATE MATERIALIZED VIEW `my_project.analytics.daily_active_users` AS
SELECT
  event_date,
  COUNT(DISTINCT user_id) AS dau
FROM `my_project.analytics.events`
GROUP BY event_date;

Subsequent queries read the small precomputed result instead of re-scanning the base table. The trade-offs are real: materialized views support a limited subset of SQL (aggregations over a single table, broadly — no arbitrary joins), and BigQuery charges to keep them refreshed as the base table changes. They pay off when the aggregation is expensive and queried far more often than the base data changes. For a one-off report, skip them.

Lever 6: Table expiration and long-term storage

Storage is cheaper than compute, but it still adds up — and old data you forgot to delete is pure waste. Two free wins:

Set a default expiration so scratch and staging tables clean themselves up:

ALTER SCHEMA `my_project.staging`
SET OPTIONS (default_table_expiration_days = 30);

Or per partition, so each day's data ages out automatically:

ALTER TABLE `my_project.analytics.events`
SET OPTIONS (partition_expiration_days = 540);  -- keep ~18 months, drop older partitions

Long-term storage is automatic and free to trigger. Any table or partition not modified for 90 consecutive days automatically drops to roughly half the storage price — no action needed. The catch most people miss: querying a table does not count as modifying it. Reads don't reset the clock; only writes do. So a job that needlessly rewrites a whole table every night is quietly keeping all of it in the pricier "active storage" tier. Append to partitions instead of rewriting the table, and your cold history slides into long-term pricing on its own.

Lever 7: BI Engine for dashboards

If a Looker / Looker Studio / Tableau dashboard hammers BigQuery on every filter click, each interaction can be its own billable scan. BI Engine is an in-memory acceleration layer: you reserve a small amount of memory capacity, BigQuery caches hot data there, and repeat dashboard queries are served from memory — faster and without re-scanning.

You reserve capacity per project (billed per GB-hour while reserved), and queries that fit transparently use it. It's a targeted tool: great for interactive BI on a bounded set of hot tables, pointless for one-off analytical queries. Reserve a small amount, watch the cache-hit behavior, and size up only if it helps.

Decision flow: which cost lever to reach for first, in order of effort-to-payoff.

When NOT to bother

Honesty matters more than a clean narrative:

  • Small tables (a few GB). Partitioning and clustering add management overhead for tables that scan cheaply anyway. Below the per-partition metadata sweet spot, partitioning can even hurt. Don't partition a 200 MB lookup table.
  • You're under the 1 TB/month free tier. If your whole workload scans less than a terabyte a month, you're paying $0 for queries. Spend your effort elsewhere.
  • Reaching for slots first. Buying a capacity commitment to cover sloppy full scans just locks in the waste at a different SKU. Clean up scans first; then evaluate whether reserved capacity beats on-demand for your steady-state load.

Takeaways

  • You pay for bytes scanned, not rows returned. Every lever here is about scanning less.
  • Dry-run everything. The console validator and bq --dry_run show the bill before you pay it. Add --maximum_bytes_billed as a circuit breaker.
  • Partition big tables by date, and add require_partition_filter = TRUE so nobody can forget.
  • Cluster by the columns you filter and group on for a second layer of pruning inside each partition.
  • Name your columns. SELECT * is the most expensive habit in the system.
  • Materialized views for repeated heavy aggregations; table/partition expiration to drop dead data; let long-term storage halve the price of cold data automatically (remember: reads don't reset the 90-day clock).
  • BI Engine for dashboards that re-query on every click.

None of this requires rewriting your analytics. It requires laying tables out so your existing queries touch less data.

If you try one thing this week, make it the dry-run habit plus partitioning your single biggest table — that combination alone usually bends the cost curve.
