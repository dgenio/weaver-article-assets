BigQuery charges by bytes scanned, not rows returned. Here are the seven habits that scan far more than you think — each with a copy-paste fix and the query-plan reason it works.

BigQuery's on-demand model bills you for bytes scanned off storage, not rows returned — so a query returning 10 rows can still scan 4 TiB. The seven habits that quietly double your bill are: SELECT * on wide columnar tables, no partition filter, joining before filtering, missing clustering, exploding self-joins, CTEs recomputed on every reference, and scanning raw data instead of a pre-aggregate. Each has a one-line fix, and the dry-run estimate (bq query --dry_run) is your cost EXPLAIN — if it doesn't move the bytes number, the fix didn't work. Start with partition filters and pre-aggregates, where the savings are biggest. You open the BigQuery billing console on the first of the month and the on-demand line item is up again. Nobody added a new pipeline. Traffic is flat. But the bytes-scanned number keeps climbing, and somewhere a finance partner is drafting an email with the subject line "quick question about cloud spend."

Here's the thing most people miss: in BigQuery's on-demand model, you don't pay for the rows you get back — you pay for the bytes the engine reads off storage to answer your query. A query that returns 10 rows can scan 4 TiB. That gap between "what I asked for" and "what got read" is where almost all wasted spend lives, and it's almost always caused by a handful of repeatable habits.

BigQuery on-demand pricing is driven by bytes scanned from disk, not the rows you get back. The fixes in this article all attack the left side of this equation.

Below are seven of those habits, ranked roughly by how often I see them double a bill in the wild. Each one comes with a runnable fix and — more importantly — the mechanism that makes the fix work, so you can reason about your own queries instead of cargo-culting mine.

One ground rule before we start: check the validator. Every BigQuery client (the Cloud Console editor, the bq CLI, the API) gives you a dry-run estimate of bytes that will be processed before you run anything. In the console it's the little "This query will process X" line in the top right. From the CLI:

bq query --dry_run --use_legacy_sql=false \
  'SELECT user_id FROM `proj.analytics.events` WHERE event_date = "2026-06-01"'

That number is your ground truth for everything below. If a "fix" doesn't move it, the fix didn't work. Treat it like EXPLAIN for cost.

1. SELECT * on a wide columnar table

This is the cheapest mistake to make and the cheapest to fix. BigQuery is a columnar store: it reads only the columns your query references. SELECT * references all of them.

-- Anti-pattern: scans every column, even the 80 you ignore
SELECT *
FROM `proj.analytics.events`
WHERE event_date = "2026-06-01";

If events has 90 columns and you actually use 3, you're scanning roughly 30x the bytes you need. The fix is boring and that's the point:

-- Fix: name the columns you use
SELECT user_id, event_name, event_timestamp
FROM `proj.analytics.events`
WHERE event_date = "2026-06-01";

A common rebuttal: "but I need to preview the data." Use the table preview tab in the console (it's free — it reads metadata, not a query) or LIMIT on a small set of columns. And note that LIMIT does not reduce bytes scanned on its own — BigQuery still reads the columns it needs across the partitions involved, then truncates the result. LIMIT bounds your result, not your cost.

If you genuinely need most columns but not all, SELECT * EXCEPT(big_blob_col, raw_payload) is a clean way to drop the heaviest offenders while keeping the convenience.

2. No partition filter on a partitioned table

Most large event tables are partitioned by a date or timestamp column. Partitioning only saves money if your query lets BigQuery prune — skip partitions it doesn't need. A query with no filter on the partition column reads every partition, which on a multi-year table is the single most expensive thing you can do.

Partition pruning vs. a full scan. With a WHERE filter on the partition column, BigQuery reads only the matching day-partitions; without it, every partition is scanned.

-- Anti-pattern: no filter on the partition column => full-table scan
SELECT user_id, event_name
FROM `proj.analytics.events`
WHERE event_name = "purchase";

-- Fix: filter on the partition column with a constant expression
SELECT user_id, event_name
FROM `proj.analytics.events`
WHERE event_date BETWEEN "2026-06-01" AND "2026-06-07"
  AND event_name = "purchase";

The gotcha that bites people: pruning needs the filter to be on the partitioning column itself, ideally compared to a constant or a simple deterministic expression. Wrapping the column in certain functions, or comparing it to the output of a non-trivial subquery, can defeat pruning so the engine falls back to reading everything. After any change, confirm with the dry-run estimate — if the bytes didn't drop, pruning didn't happen.

You can also make this a hard guardrail at the table level by setting require_partition_filter = true, which makes BigQuery reject any query that omits a partition filter. On shared tables that one setting prevents an entire category of accidental full scans.

3. Joining first, filtering later

SQL is declarative, and BigQuery's optimizer will often push filters down for you. But it can't always, especially across complex joins, views, and UNIONs — and when it can't, you pay to join rows you were about to throw away.

-- Anti-pattern: join the full tables, then filter the joined result
SELECT o.order_id, c.country
FROM `proj.sales.orders` o
JOIN `proj.sales.customers` c USING (customer_id)
WHERE o.order_date BETWEEN "2026-06-01" AND "2026-06-07"
  AND c.country = "PT";

The cleaner version filters each side before the join, so the join probes a much smaller build side:

-- Fix: reduce each input first, then join the small results
WITH recent_orders AS (
  SELECT order_id, customer_id
  FROM `proj.sales.orders`
  WHERE order_date BETWEEN "2026-06-01" AND "2026-06-07"
),
pt_customers AS (
  SELECT customer_id, country
  FROM `proj.sales.customers`
  WHERE country = "PT"
)
SELECT o.order_id, c.country
FROM recent_orders o
JOIN pt_customers c USING (customer_id);

Two honest caveats. First, the partition filter (#2) is doing most of the heavy lifting here — that's what shrinks bytes scanned. Second, on simple queries the optimizer frequently does this push-down automatically, so the rewrite changes nothing. The value of writing it explicitly is on the messy queries — nested views, multiple joins, filters that depend on a join key — where the optimizer gives up. When in doubt, dry-run both forms and keep the cheaper one.

4. Big tables with no clustering

Partitioning prunes by date. Clustering sorts the data within each partition by columns you choose, so filters and joins on those columns can skip blocks of data they don't need. On a large table that's frequently filtered by a high-cardinality column (user_id, country, product_sku), clustering is often the difference between scanning a partition and scanning a sliver of it.

-- Fix: create the table partitioned AND clustered
CREATE TABLE `proj.analytics.events_clustered`
PARTITION BY event_date
CLUSTER BY country, event_name AS
SELECT * FROM `proj.analytics.events`;

Now a query like this can skip clustered blocks that don't match:

SELECT user_id, event_timestamp
FROM `proj.analytics.events_clustered`
WHERE event_date = "2026-06-01"
  AND country = "PT"
  AND event_name = "purchase";

Things to know before you reach for it. Cluster on the columns you actually filter or aggregate by most, and order them from most- to least-frequently filtered — clustering is most effective on the leading columns. The savings are data-dependent: BigQuery does not guarantee an upfront byte estimate for clustered tables the way it does for partition pruning, because how much it can skip depends on how the data is laid out. And clustering doesn't help if your filters never touch the cluster columns. It's a strong default for big, repeatedly-filtered tables — not a magic switch for every table.

5. Exploding self-joins instead of window functions

This is the one that turns a cheap query into a runaway one. People reach for a self-join to answer "compare each row to others in its group" questions — running totals, deduplication, previous-row deltas, top-N-per-group. A self-join on a large table can produce an intermediate result that's quadratic in the group size before any filtering, and you pay to materialize it.

-- Anti-pattern: self-join to compute a per-user running total
SELECT a.user_id, a.order_date, SUM(b.amount) AS running_total
FROM `proj.sales.orders` a
JOIN `proj.sales.orders` b
  ON a.user_id = b.user_id
 AND b.order_date <= a.order_date
GROUP BY a.user_id, a.order_date;

The window-function version expresses the same intent in a single pass over the data:

-- Fix: window function, no self-join
SELECT
  user_id,
  order_date,
  SUM(amount) OVER (
    PARTITION BY user_id
    ORDER BY order_date
    RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM `proj.sales.orders`;

Same logic applies to deduplication (ROW_NUMBER() OVER (PARTITION BY id ORDER BY updated_at DESC) = 1 instead of a NOT EXISTS self-join) and previous-row deltas (LAG() instead of joining the table to itself offset by one). If you want a deeper tour of these, I wrote a companion piece on SQL window functions that covers the frame clauses in detail. The headline: anytime you're tempted to join a table to itself, ask whether a window function does it in one scan first.

6. CTEs that get recomputed every time you reference them

This one surprises experienced engineers. In BigQuery, a WITH clause (CTE) is not guaranteed to be materialized. If you reference the same CTE three times, the engine may re-execute its underlying scan three times — and you pay for each one.

-- Anti-pattern: expensive CTE referenced multiple times
WITH heavy AS (
  SELECT user_id, country, amount
  FROM `proj.sales.orders`
  WHERE order_date BETWEEN "2026-01-01" AND "2026-06-01"  -- scans 6 months
)
SELECT country, SUM(amount) FROM heavy GROUP BY country
UNION ALL
SELECT country, COUNT(DISTINCT user_id) FROM heavy GROUP BY country;

If heavy is recomputed per reference, that six-month scan happens twice. The fix is to materialize once into a temp table and read from it:

-- Fix: materialize the expensive part once
CREATE TEMP TABLE heavy AS
SELECT user_id, country, amount
FROM `proj.sales.orders`
WHERE order_date BETWEEN "2026-01-01" AND "2026-06-01";

SELECT country, SUM(amount) FROM heavy GROUP BY country
UNION ALL
SELECT country, COUNT(DISTINCT user_id) FROM heavy GROUP BY country;

The temp table is scanned once to build, then the two reads hit a much smaller intermediate result. Caveat: if a CTE is referenced only once, leave it as a CTE — it's more readable and there's nothing to save. Only reach for the temp table when the same expensive scan is referenced multiple times. (Worth a dry-run on both: the multi-statement version reports per-statement bytes, so you can see the single build scan replace the repeated ones.)

7. Scanning raw data when a pre-aggregate would do

The most expensive query is the one you run a thousand times a day against raw events to compute the same dashboard number. If a daily KPI reads a billion raw rows every refresh, you're paying to re-derive a number that barely changes within the day.

The fix is to push the aggregation upstream and read the small thing. Two flavors:

Materialized views — BigQuery maintains them incrementally as the base table changes, and the optimizer can transparently rewrite queries against the base table to hit the view:

CREATE MATERIALIZED VIEW `proj.analytics.daily_country_revenue` AS
SELECT
  event_date,
  country,
  SUM(amount) AS revenue,
  COUNT(*) AS orders
FROM `proj.analytics.events`
GROUP BY event_date, country;

Scheduled rollup tables — a scheduled query that writes yesterday's aggregate into a small summary table your dashboards read instead of the raw firehose:

-- Runs daily; appends one small day of pre-aggregated rows
INSERT INTO `proj.analytics.daily_country_revenue_rollup`
SELECT event_date, country, SUM(amount) AS revenue, COUNT(*) AS orders
FROM `proj.analytics.events`
WHERE event_date = CURRENT_DATE() - 1
GROUP BY event_date, country;

Dashboards then scan kilobytes instead of terabytes. The trade-offs are real: materialized views have restrictions on the aggregations they support and add maintenance cost as the base table churns, and rollup tables add a pipeline you have to own and backfill. But for any number that's read far more often than the underlying data changes, pre-aggregating is the highest-leverage cost fix on this list. When NOT to optimize

Senior move: don't micro-optimize a query that runs once a week and scans 200 MB. BigQuery's on-demand pricing has a per-query minimum (a small floor of bytes billed per query), so shaving bytes off already-tiny queries is noise. Spend your effort where bytes × frequency is large — the dashboard query that runs every five minutes against a 4 TiB table, not the ad-hoc analysis you ran twice. Pull up INFORMATION_SCHEMA.JOBS and sort by total_bytes_processed to find them; that's your real backlog.

Also: if your workload is on flat-rate / capacity (slot) pricing rather than on-demand, the cost model changes — you're paying for slot-time, not bytes. Most of these fixes still help (they reduce slot contention and make queries faster), but the "bytes scanned = dollars" framing is specific to on-demand.

Takeaways

  • You pay for bytes scanned, not rows returned. Use the dry-run estimate as your cost EXPLAIN before every change.
  • SELECT * is a columnar tax. Name your columns; use EXCEPT to drop the heavy ones.
  • Always filter the partition column with a constant expression, and set require_partition_filter on shared tables.
  • Filter before you join on messy multi-join queries; verify the optimizer isn't already doing it.
  • Cluster big, repeatedly-filtered tables on the columns you actually filter by — leading column matters most.
  • Replace self-joins with window functions for running totals, dedup, and previous-row logic — one pass instead of a quadratic blow-up.
  • Materialize a CTE only when it's referenced multiple times; otherwise leave it readable.
  • Pre-aggregate anything read far more often than it changes, via materialized views or scheduled rollups.
  • Optimize by bytes × frequency, not by gut feel. INFORMATION_SCHEMA.JOBS tells you where the money actually goes.
