Your agent keeps re-deciding the same tool path on every turn. Compile it once into a typed, LLM-free flow — and watch the data-corruption rate drop from 61–96% to 0%.

ChainWeaver is a small, standalone Python library (alpha, v0.13.0) that compiles an agent's repeated, deterministic tool paths into typed Flow objects executed by a FlowExecutor with zero LLM calls between steps. Each step runs validate(input) → fn → validate(output) → merge with strict Pydantic checks, so no code path can drop, rename, or retype a field — driving inter-step data corruption from a benchmarked 61–96% down to a structural 0%. You keep the model only for picking which flow to run; the deterministic interior runs LLM-free, faster, cheaper, and fully auditable. Your agent works. It also does something quietly wasteful on every single turn.

Watch a typical tool-using loop handle a request. It runs search, then asks the model "what now?", runs extract, asks the model again, runs validate, asks again, runs format. Four tool calls, and four model round-trips wedged between them — even though the path never changes. Given a search result, you always extract. Given facts, you always validate. The model isn't reasoning at those points. It's re-deriving a decision you already know the answer to, at the price of latency, tokens, and one more chance to hallucinate a field.

ChainWeaver is a small Python library that takes those repeated, deterministic tool paths and compiles them into typed flows that execute with no LLM between steps. This is a practical walk-through: the problem, the mechanism, a flow you can run in five minutes, the benchmark numbers (with their caveats), and an honest read on maturity. It's alpha software — version 0.13.0 — and I'll be specific about what that means.

The problem, concretely

An LLM-mediated tool loop inserts a model call between every step so the model can "decide" what to do next:

User request
    │
    ▼
LLM call ──► Tool A
    │
    ▼
LLM call ──► Tool B
    │
    ▼
LLM call ──► Tool C
    │
    ▼
Response

For a path that is genuinely deterministic — where the next tool is fully determined by the previous output — those intermediate calls buy you nothing and cost you three things:

  • Latency. Each round-trip is hundreds of milliseconds of model time.
  • Cost. Every call burns input tokens (the growing transcript), output tokens, and re-sends the tool schemas.
  • Unpredictability. A model re-asked "what next?" can route differently, and — worse — it can quietly mangle the data it passes between tools. Drop a field. Rename user_id to userId. Turn "42" into 42 or 42 into "forty-two". The envelope still looks plausible, so nothing errors. The data is just wrong.

That last point is the expensive one, and it compounds. If each hand-off has even a ~25% chance of structural corruption, a four-step chain is corrupted more often than not, and a ten-step chain is corrupted almost every time. ChainWeaver's own benchmark puts numbers on this; we'll get there.

The idea in one paragraph

ChainWeaver is the difference between an interpreter and a compiler for tool chains. You keep the model where it adds value — picking which flow to run — and you compile the deterministic interior into a named, versioned Flow object. A FlowExecutor runs that flow step by step with strict Pydantic validation at every boundary and zero model calls in between. The agent still decides to invoke search_summarize_flow; the four tool calls inside it no longer round-trip through the model. That single design choice — LLM-free between steps, enforced at the framework level — is the whole point, and the rest of the library (typed I/O, file-serializable flows, schema-drift detection, structured traces) is built around defending it.

Naive LLM-in-the-loop chaining vs. a compiled ChainWeaver flow. Every diamond is a model round-trip that can hallucinate, drop, or retype a field; the compiled path replaces them with Pydantic-validated dict merges.

How it works

Three primitives carry the whole model.

A Tool wraps a plain Python callable together with a Pydantic input schema and output schema:

from pydantic import BaseModel
from chainweaver import Tool

class NumberInput(BaseModel):
    number: int

class ValueOutput(BaseModel):
    value: int

def double_fn(inp: NumberInput) -> dict:
    return {"value": inp.number * 2}

double = Tool(
    name="double",
    description="Takes a number and returns its double.",
    input_schema=NumberInput,
    output_schema=ValueOutput,
    fn=double_fn,
)

A Flow is an ordered list of FlowSteps. Each step names a tool and an input_mapping — how to pull this tool's inputs out of the accumulated execution context:

from chainweaver import Flow, FlowStep

flow = Flow(
    name="double_add_format",
    description="Doubles a number, adds 10, and formats the result.",
    steps=[
        FlowStep(tool_name="double",        input_mapping={"number": "number"}),
        FlowStep(tool_name="add_ten",       input_mapping={"value": "value"}),
        FlowStep(tool_name="format_result", input_mapping={"value": "value"}),
    ],
)

In input_mapping, a string value is a context lookup (a plain key is a top-level lookup; a string starting with / is an RFC-6901 JSON pointer into nested context, e.g. "/user/address/city"), and a non-string value is a literal constant. An optional output_mapping renames and prunes a tool's outputs before they merge.

A FlowExecutor runs it. The per-step execution path is the entire data-integrity story:

validate(input)  →  fn(input)  →  validate(output)  →  context.update(output)

No code path between the two validate calls can add, drop, or rename a field. The intermediate context is a plain Python dict updated by direct assignment — no serialization round-trip, no schema inference, no model in the loop. That property is protected by a hard invariant the project enforces by review: no LLM calls, no network I/O, and no randomness inside executor.py. (It's literally rule #1 in the repo's CLAUDE.md.)

Inside FlowExecutor: each step runs validate(input) -> fn -> validate(output) -> merge into a plain dict context. No code path between the two validations can add, drop, or rename a field.

There's also a discovery side most "just wire it up" libraries lack. ChainAnalyzer inspects a set of tools offline and computes which ones are schema-compatible — every required input field of a consumer must appear, with a matching type, in some producer's output. analyzer.suggest_flows(max_depth=3) then returns ready-to-register Flow objects with input_mapping auto-wired by name. And ChainObserver watches what your agent actually does at runtime and proposes compiled flows for the paths it repeats — ranked, never auto-registered. The intended loop is observe → compile → replace: you don't hand-wire the flow, you compile the path the analyzer or observer surfaces.

Try it in five minutes

Install from PyPI. The [yaml] extra is the common one — you need it for .flow.yaml files and the CLI:

pip install 'chainweaver[yaml]'

The base install pulls only five runtime dependencies (deepdiff, packaging, pydantic, tenacity, typer) and pins no LLM SDK. Here is a complete, runnable three-step flow — adapted directly from examples/simple_linear_flow.py in the repo:

from pydantic import BaseModel
from chainweaver import Tool, Flow, FlowStep, FlowRegistry, FlowExecutor

# --- 1. Schemas ---
class NumberInput(BaseModel):
    number: int

class ValueOutput(BaseModel):
    value: int

class ValueInput(BaseModel):
    value: int

class FormattedOutput(BaseModel):
    result: str

# --- 2. Tool functions ---
def double_fn(inp: NumberInput) -> dict:
    return {"value": inp.number * 2}

def add_ten_fn(inp: ValueInput) -> dict:
    return {"value": inp.value + 10}

def format_result_fn(inp: ValueInput) -> dict:
    return {"result": f"Final value: {inp.value}"}

# --- 3. Tools ---
double = Tool(name="double", description="Doubles a number.",
              input_schema=NumberInput, output_schema=ValueOutput, fn=double_fn)
add_ten = Tool(name="add_ten", description="Adds ten.",
               input_schema=ValueInput, output_schema=ValueOutput, fn=add_ten_fn)
format_result = Tool(name="format_result", description="Formats the value.",
                     input_schema=ValueInput, output_schema=FormattedOutput, fn=format_result_fn)

# --- 4. Flow ---
flow = Flow(
    name="double_add_format",
    description="Doubles a number, adds 10, and formats the result.",
    steps=[
        FlowStep(tool_name="double",        input_mapping={"number": "number"}),
        FlowStep(tool_name="add_ten",       input_mapping={"value": "value"}),
        FlowStep(tool_name="format_result", input_mapping={"value": "value"}),
    ],
)

# --- 5. Register and execute — zero LLM calls ---
registry = FlowRegistry()
registry.register_flow(flow)

executor = FlowExecutor(registry=registry)
executor.register_tool(double)
executor.register_tool(add_ten)
executor.register_tool(format_result)

result = executor.execute_flow("double_add_format", {"number": 5})

print(result.success)       # True
print(result.final_output)  # {'number': 5, 'value': 20, 'result': 'Final value: 20'}

for record in result.execution_log:
    print(record.step_index, record.tool_name, record.outputs)
# 0 double {'value': 10}
# 1 add_ten {'value': 20}
# 2 format_result {'result': 'Final value: 20'}

The structured execution_log — step index, tool name, validated outputs — is what makes a run auditable: you can diff it, store it, or replay it.

Prefer less boilerplate? The @tool decorator introspects type hints to build the input schema for you:

from chainweaver import tool

@tool(description="Doubles a number.")
def double(number: int) -> ValueOutput:
    return {"value": number * 2}

print(double(number=5))  # {'value': 10} — decorated tools stay directly callable

And if you'd rather not touch Python at all, the bundled CLI runs a flow file straight from disk:

chainweaver run examples/double_add_format.flow.yaml \
    --tools examples.simple_linear_flow \
    --input '{"number": 5}'

The CLI also ships validate, check, viz (ASCII / Graphviz DOT / Mermaid), explain, attest (observed-determinism evidence over N inputs × M repeats), record (mine candidate flows from a recorded JSONL trace), and doctor. All subcommands share an exit-code contract (0 success, 1 business error, 2 file/argument error) and most accept --format json wrapped in a versioned envelope for automation.

The benchmark: latency, and the part that actually matters

ChainWeaver ships a reproducible benchmark you regenerate with python benchmarks/report.py. The latest committed benchmark report (generated under ChainWeaver 0.11.0, Python 3.11, priced 2026-05-01; the execution path is unchanged through 0.13.0) reports two things.

Latency — compiled vs. naive, where each naive step pays a simulated model delay:

Steps   LLM delay (ms)   Naive total (ms)   Compiled total (ms)   Speedup
-----   --------------   ----------------   -------------------   -------
2       100              100.34             0.25                  407×
5       200              801.12             0.35                  2306×
10      200              1904.36            102.95                18.5×

Those speedups are eye-catching but they are a proxy: the naive baseline simulates a model round-trip with time.sleep, so the figures isolate round-trip count, not any real provider's wall-clock. Read them as "this many model calls were removed," not "your p99 will drop 2000×."

Correctness is the number I'd actually publish. The benchmark routes the same chain two ways. The naive path runs through a seeded, configurable corruption model whose per-step rates — field hallucination, data loss, type corruption, schema drift, routing variance — are order-of-magnitude estimates loosely informed by published tool-calling reliability studies. The compiled path runs the identical chain through FlowExecutor:

Scenario           Steps   Naive corruption   Compiled corruption
----------------   -----   ----------------   -------------------
numeric_pipeline   4       61.0%              **0.0%**
data_enrichment    4       61.0%              **0.0%**
long_chain         10      96.0%              **0.0%**

The compiled 0.0% is not a measurement that happened to come out clean — it's structural. There is no code path in the validated-merge pipeline that can hallucinate, drop, or rename a field, so corruption events go to zero by construction. The naive rates, meanwhile, behave exactly as compounding predicts: plug the default per-step rates into 1 − (∏ clean_rate)^n and a 4-step chain lands near 69% and a 10-step chain near 95% — right where the seeded model reports 61% and 96%.

On cost, the same report estimates a 10-step flow avoids $0.0607 of LLM spend (priced against anthropic/claude-sonnet-4-6). That's small per run and scales with your assumed tokens-per-call — and the repo is careful to call it an estimate from a dated price snapshot, not billed usage. The compute_cost_report(...) helper will produce one for your own step counts and model, and it raises rather than guessing for a (provider, model) pair it doesn't have a price for. Cost savings are framed as a consequence here, not the headline — and that's the honest framing.

Honest limitations

This is alpha software (Development Status :: 3 - Alpha, version 0.13.0). The test suite is substantial for the stage — roughly 1,600 test functions across ~108 test files, with an 80% coverage floor enforced in CI — but the API can still move before 1.0.

More importantly, ChainWeaver guarantees the envelope, not the contents. A tool whose fn returns {"price_usd": 999999.99} for every request still satisfies a price_usd: float schema — the executor validates the shape, not the truthfulness. Determinism is a property of the flow: if one of your tools calls a clock, an RNG, or a flaky network service, that tool breaks the guarantee, and ChainWeaver doesn't pretend otherwise (use chainweaver attest to gather observed-determinism evidence). And by design it is not an agent framework or a general workflow engine — if every step needs open-ended reasoning to pick the next one, use LangGraph or an SDK tool-use loop; if you need durable, scheduled, long-running jobs, use Prefect, Dagster, or Temporal. ChainWeaver is meant to be called from an agent framework, not to replace one. It's the deterministic-execution layer of the broader Weaver Stack, alongside siblings like contextweaver, agent-kernel, and lessonweaver — but its base install has no hard dependency on any of them and runs fully standalone.

Takeaways

  • Stop paying for decisions you've already made. Repeated, deterministic tool paths don't need a model between every step — compile them once.
  • The real win is data integrity, not speed. validate → fn → validate → merge makes inter-step corruption structurally impossible (0% vs. a benchmarked 61–96%). Latency and cost savings are a bonus.
  • You don't have to hand-wire flows. ChainAnalyzer finds schema-compatible chains offline; ChainObserver proposes flows from real traces. Observe → compile → replace.
  • Typed, file-serializable, auditable. Pydantic at every boundary, YAML/JSON flow files, structured per-step logs, a CLI with a stable exit-code contract.
  • Know the boundary. Keep the model for routing and open-ended reasoning; hand the deterministic interior to ChainWeaver. It complements your agent framework — it doesn't replace it.
  • It's alpha — read the caveats. ~1,600 tests and an 80% coverage gate, but the API may shift before 1.0, and the benchmark uses seeded/simulated models, not live LLM spend.

If you run agents that repeat the same tool path turn after turn, clone the repo, run python examples/naive_vs_compiled.py, and see your own before/after. If the trade-off fits your workload, star ChainWeaver on GitHub, try it on one real flow, and open an issue with what broke — alpha projects get better fastest when early users push on the edges.
