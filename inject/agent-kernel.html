Your agent has 1,000 tools and no idea which ones it's allowed to call. agent-kernel gives every tool call an HMAC capability token, a policy gate, and a tamper-evident audit trail — in-process, in two dependencies.

Your agent works. You wired up an LLM, handed it a pile of MCP tools, and watched it plan, call list_invoices, summarize the result. Demo-day magic.

Then you tried to ship it, and the questions started:

  • The agent has a tool that deletes records. What stops it from calling that tool when a prompt-injected document tells it to?
  • A list_customers tool returns email and phone. Those rows go straight into the model's context. Who decided that was OK?
  • Three weeks from now, someone asks "did the agent touch the billing system on the 4th, and under whose authority?" Can you answer?

Most agent frameworks have no answer, because they were built to make tool-calling work, not to make it safe to operate. The tool registry is a flat list. Any tool the model can name, it can call. Raw tool output flows directly back into the prompt. And the only record of what happened is whatever you happened to log.

This is the gap agent-kernel is built to close. It's a small Python library (on PyPI as weaver-kernel) that sits between your LLM loop and your tools and enforces a simple rule: no token, no execution — and it keeps a tamper-evident record of every call that ran.

It is honest alpha software (Development Status :: 3 - Alpha, currently 0.11.0), but it's not a sketch: 665 tests, two runtime dependencies, strict mypy. This post is a deep-dive into the actual mechanism — the token, the policy gate, the firewall, and the audit chain — with code you can run today.

The idea in one paragraph

agent-kernel borrows capability-based security from operating-system design and applies it to tool calls. Instead of asking "is this agent allowed to do things?" at login and trusting it forever, every single tool invocation must present a capability token: an HMAC-signed, time-bounded credential scoped to exactly one principal and one capability. The kernel runs a policy engine to decide whether to mint that token, routes the call to a driver (MCP, HTTP, or in-memory), passes the result through a context firewall that redacts and bounds it before the LLM ever sees it, and writes an ActionTrace to a hash-chained audit log. Five steps, one in-process call path, no external proxy required.

How it works

The full lifecycle is register → grant → invoke → expand → explain. Here's the shape of it before we dig into each piece.

The request path: every tool call is authorized, signed, executed, firewalled, and audited.

1. Capabilities, not functions

A capability is a registered, auditable action — not a raw Python function. Registering one carries the metadata the kernel needs to reason about safety:

from weaver_kernel import (
    Capability, CapabilityRegistry, ImplementationRef,
    SafetyClass, SensitivityTag,
)

registry = CapabilityRegistry()
registry.register(
    Capability(
        capability_id="billing.invoices.list",
        name="List Invoices",
        description="List recent invoices",
        safety_class=SafetyClass.READ,
        sensitivity=SensitivityTag.PII,
        allowed_fields=["id", "customer_name", "amount", "status"],
        tags=["billing", "invoices", "list"],
        impl=ImplementationRef(driver_id="memory", operation="list_invoices"),
    )
)

Two fields do the heavy lifting. safety_class is one of READ, WRITE, or DESTRUCTIVE — it controls which roles may call the capability at all. sensitivity (PII, PCI, SECRETS, MEMORY) tells the policy engine and the firewall how to treat the data. The allowed_fields projection is the whitelist the firewall enforces: email isn't on that list, so email never reaches the model — even though the underlying driver returns it.

2. The capability token: where confused-deputy attacks go to die

The token is the heart of the system. Here is the actual signing payload from tokens.py:

def _signable_payload(self) -> str:
    payload = {
        "token_id": self.token_id,
        "capability_id": self.capability_id,
        "principal_id": self.principal_id,
        "issued_at": self.issued_at.isoformat(),
        "expires_at": self.expires_at.isoformat(),
        "constraints": self.constraints,
        "audit_id": self.audit_id,
    }
    return json.dumps(payload, sort_keys=True, separators=(",", ":"))

That JSON is signed with HMAC-SHA256 using a secret from WEAVER_KERNEL_SECRET. Verification then runs, in order: revocation check, expiry check, constant-time signature comparison (hmac.compare_digest), then two binding checks:

# 3. Principal binding (confused-deputy prevention)
if token.principal_id != expected_principal_id:
    raise TokenScopeError(...)

# 4. Capability binding
if token.capability_id != expected_capability_id:
    raise TokenScopeError(...)

This is what closes the classic confused-deputy hole. If agent A obtains a token for billing.list_invoices and hands it to agent B, B's invocation arrives with expected_principal_id="B", the binding check fails, and the call is rejected. A token is not a bearer credential — it's bound to who asked and what they asked for. Tamper with any field and the recomputed HMAC won't match: TokenInvalid.

3. The policy engine decides whether a token is even minted

get_token() doesn't just sign — it first runs DefaultPolicyEngine.evaluate(), a deterministic, ordered rule chain. The real defaults, straight from policy.py:

  • WRITE capabilities require the writer or admin role and a justification of at least 15 characters.
  • DESTRUCTIVE requires admin.
  • PII/PCI capabilities require a tenant attribute on the principal, and apply allowed_fields unless the principal has the pii_reader role.
  • SECRETS require admin or secrets_reader plus a justification.
  • Row caps: 50 rows for regular users, 500 for principals with the service role.
  • Rate limits (sliding window, per principal_id:capability_id): 60 READ / 10 WRITE / 2 DESTRUCTIVE per 60-second window; service principals get 10×.

Every decision carries a stable, machine-branchable reason_code (missing_role, missing_tenant_attribute, rate_limited, …) — the docs are explicit that your tests should assert on the code, not the human-readable string. A denied write looks like this:

from weaver_kernel.errors import PolicyDenied

try:
    kernel.get_token(
        CapabilityRequest(capability_id="billing.invoices.create",
                          goal="create an invoice"),
        alice, justification="reader trying a write — should fail",
    )
except PolicyDenied as exc:
    print(exc.reason_code)   # 'missing_role'

4. The context firewall: raw output never reaches the model

This is the invariant the project guards hardest (weaver-spec I-01): the LLM never sees raw driver output by default. The Firewall always transforms a RawResult into a bounded Frame. A Frame carries compact facts, an optional row preview, and a Handle the model can expand later — within budget. Redaction is regex-based and covers emails, phone numbers, card and SSN patterns, Bearer tokens, JWTs, API keys, and connection strings, and it fails closed at the depth limit: nested containers past max_depth are elided as [REDACTED: nested data beyond depth limit] rather than passed through raw (a real bug fixed in 0.11.0, #149).

There's a raw response mode, but non-admin principals can't get it — the firewall silently downgrades them to summary.

5. Tamper-evident audit

Every successful invoke() writes an ActionTrace you can retrieve with kernel.explain(action_id): who, what, when, which driver, under which policy decision. With a durable backend (SQLiteTraceStore or append-only JsonlTraceStore), each record is wrapped in a hash chain:

record_hash = HMAC-SHA256(secret, {seq, prev_hash, trace})

The audit hash chain: each record commits to the previous one, so mutation, insertion, deletion, and reordering are all detectable.

verify_chain() recomputes every link and reports the seq of the first divergent record, catching mutation, interior insertion, deletion, and reordering. The maintainers are refreshingly precise about the limit: because there's no signed head anchor, tail-truncation and whole-log deletion are not detected — that's a documented follow-up, and until then the recommended defense is append-only durable storage. This is tamper-evidence, not non-repudiation. It's rare to see a security library this candid about its own threat model.

Try it in 5 minutes

Two dependencies, no services to stand up.

pip install weaver-kernel

Note the naming: the GitHub repo is dgenio/agent-kernel, the PyPI package is weaver-kernel, and the Python import is weaver_kernel. There is no agent_kernel import anymore.

The following is the project's own README quickstart — it's extracted and executed by CI (tests/test_readme_quickstart.py), so it can't silently drift from the working API. I ran it against 0.11.0 while writing this; the output below is real.

import asyncio, os
os.environ["WEAVER_KERNEL_SECRET"] = "my-secret"

from weaver_kernel import (
    Capability, CapabilityRegistry,
    InMemoryDriver, Kernel, Principal, SafetyClass, StaticRouter,
)
from weaver_kernel.models import CapabilityRequest

# 1. Register a capability
registry = CapabilityRegistry()
registry.register(Capability(
    capability_id="tasks.list",
    name="List Tasks",
    description="List all tasks",
    safety_class=SafetyClass.READ,
    tags=["tasks", "list"],
))

# 2. Wire up a driver
driver = InMemoryDriver()
driver.register_handler("tasks.list", lambda ctx: [{"id": 1, "title": "Buy milk"}])

# 3. Build the kernel
kernel = Kernel(registry=registry, router=StaticRouter(routes={"tasks.list": ["memory"]}))
kernel.register_driver(driver)

async def main():
    principal = Principal(principal_id="alice", roles=["reader"])

    # 4. grant -> invoke -> expand -> explain
    token = kernel.get_token(
        CapabilityRequest(capability_id="tasks.list", goal="list tasks"),
        principal, justification="",
    )
    frame = await kernel.invoke(token, principal=principal, args={})
    print(frame.facts)   # ['Total rows: 1', 'Top keys: id, title', 'id: min=1.00, max=1.00, avg=1.00']

    # principal is required: the handle is bound to the granting principal
    expanded = kernel.expand(
        frame.handle, query={"limit": 1, "fields": ["title"]}, principal=principal
    )
    print(expanded.table_preview)   # [{'title': 'Buy milk'}]

    trace = kernel.explain(frame.action_id)
    print(trace.driver_id)          # 'memory'

asyncio.run(main())

Notice that expand() requires the principal — a handle bound to one principal can't be expanded by another, the same confused-deputy guard applied to data references. When you're ready for a real tool server, swap InMemoryDriver for MCPDriver.from_stdio(...); capabilities, policy, tokens, and firewall behave identically. There's also a weaver-kernel operator CLI (audit list|show|verify|export, plus doctor) for inspecting and verifying a persisted trace store, and zero-dependency adapters that emit Anthropic and OpenAI tool definitions and route tool_use blocks through the kernel.

Honest limitations

The README says it plainly: v0.1 is not production-hardened for real authentication. Specifically:

  • HMAC tokens are signed, not encrypted. Don't put secrets in token fields. The WEAVER_KERNEL_SECRET must be kept secret and rotated if compromised — and because that same secret signs both tokens and the audit chain, the log is only as trustworthy as your secret custody.
  • PII redaction is heuristic (regex). It catches common patterns; it is not a substitute for real data governance. Streaming redaction reassembles secrets split across chunks via a bounded holdback window, but a pattern with internal whitespace split exactly at the boundary, or a single token longer than the buffer, can still slip — a documented memory-vs-safety trade.
  • Audit truncation isn't detectable yet (no signed head anchor), and rate-limit state is in-process only — it resets on restart and isn't distributed.
  • It's alpha, single-digit minor version. The import name changed under it recently (agent_kernel → weaver_kernel in 0.10.0). Expect churn.

None of this is disqualifying for the job it's built for — adding least-privilege authorization and an audit trail to an agent you control — but go in with eyes open.

Where it fits

agent-kernel is the in-process authorization runtime of the broader Weaver stack, and it's deliberately scoped against its siblings:

  • contextweaver decides what context the LLM sees; agent-kernel produces the canonical Frame it selects over.
  • AgentFence enforces a policy gate at the external process boundary; agent-kernel enforces the equivalent gate embedded in your agent host.
  • ChainWeaver runs deterministic tool chains; weaver-spec is the shared contract all of them conform to.

It has no hard dependency on any of them — httpx and pydantic are the whole runtime footprint. Use it standalone today.

Takeaways

  • Treat tool calls like syscalls. A capability token scoped to one principal + one capability + a TTL turns "the agent can call anything it can name" into "the agent can do exactly what it was granted, and not a byte more."
  • The firewall is the under-rated half. Bounding and redacting tool output before it hits the context window defends against both context blowup and prompt injection riding in on raw data.
  • Audit is a feature, not an afterthought. A hash-chained ActionTrace per invocation means "what did the agent do, and under whose authority?" has an answer you can verify.
  • Know the boundaries. Tamper-evidence ≠ non-repudiation; heuristic redaction ≠ governance; alpha ≠ production-hardened. The repo says so itself, which is the right instinct in a security project.

If you're building agents that touch anything that matters — money, customer data, destructive operations — the authorization-and-audit layer is not optional, and you probably shouldn't hand-roll it. agent-kernel is a small, well-tested, brutally honest starting point.
