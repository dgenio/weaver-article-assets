contextweaver puts a deterministic gateway in front of your MCP tools: a bounded ChoiceCard shortlist instead of every schema, and a firewall that swaps huge tool results for compact summaries.

contextweaver is a deterministic context gateway you drop in front of your MCP servers to fight context rot. Instead of dumping every tool schema into the prompt, its Routing Engine returns a bounded shortlist of ChoiceCards (typically 5), and a Context Firewall swaps huge tool results for compact summaries while storing raw bytes as addressable artifacts. Across the six benchmark scenarios in the repo, this cuts prompt tokens 41.6% to 84.3% (average 64.3%), measured with tiktoken.cl100k_base against a naive concat baseline — with no model in the loop, no API key, and identical outputs every run. You wired up an MCP agent. It worked great with five tools. Then you connected three more MCP servers, the catalog hit a hundred tools, and two things broke at once.

First, every request now ships a hundred tool schemas into the prompt before the model has done anything. Second, one db_query came back with a 14 KB JSON dump and your "answer" prompt exploded to 21,000 characters. Your token bill climbed, latency crept up, and — the part nobody warns you about — accuracy got worse, because the right tool is now buried among ninety-nine distractors.

This is context rot, and it is the failure mode contextweaver is built to kill. It is a deterministic context gateway you drop in front of your MCP servers. The problem, concretely

A tool-using agent answers four questions on every step: which tool do I call (route), with what arguments (call), what did it return (interpret), and how do I answer the user (answer). The naive loop pays for everything at every step.

Take the repo's --scenario killer: an internal ops agent with 100 tools and a running conversation, asked to "find unpaid invoices, check the account notes, and draft a reminder." Here is what the naive loop spends versus what contextweaver spends — these are the exact figures the demo prints, in characters (deterministic everywhere):

                            Naive                           contextweaver                   Reduction
-------------------------   -----------------------------   -----------------------------   ---------
Tools in the route prompt   all 100 (6,326 chars)           5 ChoiceCards (491 chars)       **92.2%**
The huge tool result        raw (14,430 chars)              firewalled summary (60 chars)   **99.6%**
The full answer prompt      everything raw (21,332 chars)   compiled (814 chars)            **96.2%**

The token cost is the obvious problem. The quieter one is quality: as distractor tools pile up, the correct tool gets harder to retrieve. Keeping the model-visible surface bounded — five cards, not a hundred schemas — is the lever that actually moves both.

Context rot: as the catalog grows, a naive route prompt balloons while contextweaver stays flat at 5 cards.

The idea in one paragraph

contextweaver is two small, cooperating engines you can adopt together or separately. A Routing Engine turns a flat tool catalog into a bounded DAG and beam-searches it to return a short, LLM-friendly shortlist of ChoiceCards that deliberately strip the full args_schema. A Context Engine runs an eight-stage pipeline that pulls phase-relevant events, keeps tool calls glued to their results via dependency closure, runs a context firewall that stores big tool outputs out-of-band and injects only a summary, then greedily packs the highest-scoring items into a per-phase token budget. The output is a ContextPack: a compiled prompt plus BuildStats telling you exactly what was kept, dropped, and deduplicated.

How it actually works

The two engines feed each other. Here is the shape:

                ┌────────────────────────────┐
  Events ──────>│      Context Engine         │──> ContextPack (prompt)
                │  candidates → closure →     │
                │  sensitivity → firewall →   │
                │  score → dedup → select →   │
                │  render                     │
                └────────────────────────────┘
                           ▲ facts / episodes
                ┌──────────┴─────────────────┐
  Tools ───────>│      Routing Engine         │──> ChoiceCards
                │  Catalog → TreeBuilder →    │
                │  ChoiceGraph → Router       │
                └────────────────────────────┘

Three design choices are worth calling out, because they are what make the savings safe rather than just lossy truncation.

Phase-specific budgets. Each phase gets its own token budget — the defaults are route=2000, call=3000, interpret=4000, answer=6000. The route phase never needs full schemas, so it gets a tight budget and cards only. The answer phase gets more room for relevant history. There is no one-size-fits-all truncation.

Dependency closure. When a selected item has a parent_id, its parent comes along automatically. A tool_result is never separated from the tool_call that produced it, so you do not get the classic "model sees an answer with no idea what question produced it" bug.

The firewall is on for every tool result, not just big ones. Every tool_result passes through the envelope. If it is under the threshold (default 2000 characters) the summary equals the raw text and nothing is saved or added — a correct no-op. When the payload is genuinely large, the raw bytes are stored as an addressable artifact and the prompt gets a short summary plus an ArtifactRef you can drill into later.

The two cooperating engines: a Routing Engine narrows the catalog to bounded ChoiceCards, and a Context Engine firewalls results and compiles a budgeted prompt.

A minimal context build

This is examples/minimal_loop.py, which runs in CI on every push. Ingest events, build a budgeted pack for the answer phase, inspect what happened:

from contextweaver.context.manager import ContextManager
from contextweaver.store.event_log import InMemoryEventLog
from contextweaver.types import ContextItem, ItemKind, Phase

log = InMemoryEventLog()
log.append(ContextItem(id="u1", kind=ItemKind.user_turn,
                       text="How many rows are in the users table?"))
log.append(ContextItem(id="a1", kind=ItemKind.agent_msg,
                       text="I'll check the database for you."))
log.append(ContextItem(id="tc1", kind=ItemKind.tool_call,
                       text='db_query(sql="SELECT COUNT(*) FROM users")', parent_id="u1"))
log.append(ContextItem(id="tr1", kind=ItemKind.tool_result,
                       text="count: 1042", parent_id="tc1"))

mgr = ContextManager(event_log=log)
pack = mgr.build_sync(phase=Phase.answer, query="users table row count")

print(pack.prompt)                       # the compiled, budget-aware context
print(pack.stats.total_candidates)       # how many events were considered
print(pack.stats.included_count)         # how many made the cut
print(pack.stats.dedup_removed)          # near-duplicates dropped

pack.stats is your debugging surface. A completed build always satisfies included_count + dropped_count == total_candidates, and dropped_reasons tells you why anything was evicted.

Route a tool catalog to a shortlist

The routing engine stands alone — point it at a catalog and ask for the top tools (adapted from examples/routing_demo.py):

from contextweaver.routing.cards import cards_for_route, format_card_for_prompt
from contextweaver.routing.catalog import Catalog
from contextweaver.routing.router import Router
from contextweaver.routing.tree import TreeBuilder
from contextweaver.types import SelectableItem

catalog = Catalog()
catalog.register(SelectableItem("send_email", "tool", "send_email",
                 "Send an email notification", tags=["comm", "email"],
                 namespace="comms", side_effects=True))
catalog.register(SelectableItem("db_query", "tool", "query_database",
                 "Execute arbitrary SQL queries", tags=["data", "sql"], namespace="db"))
# ... register the rest of your catalog ...

items = catalog.all()
graph = TreeBuilder().build(items)
router = Router(graph, items=items, beam_width=3)

result = router.route("send a reminder email about unpaid invoices")
for card in cards_for_route(result.candidate_ids, catalog):
    print(format_card_for_prompt(card))   # compact cards — no full schemas

See the firewall fire

examples/before_after.py runs the same eight-event loop twice. The naive version concatenates everything; the contextweaver version routes the large tool_result through the firewall with a 200-char threshold:

if event.kind == ItemKind.tool_result and len(event.text) > 200:
    mgr.ingest_tool_result(
        tool_call_id=event.parent_id or event.id,
        raw_output=event.text,
        tool_name="db_query",
        firewall_threshold=200,
    )
else:
    mgr.ingest(event)

pack = mgr.build_sync(phase=Phase.answer, query="engineering users")

The raw bytes don't vanish — they're addressable via mgr.artifact_store, so you can drill back into the full payload (head / lines / json_keys / rows selectors) when the model actually needs a detail.

Already have an OpenAI / Anthropic / Gemini history? The drop-in is five lines via from_openai_messages(messages, into=mgr) (and from_anthropic_messages / from_gemini_contents).

The benchmark, honestly

Token reduction is the headline, and it holds up across the six committed scenarios in benchmarks/scorecard.md, measured against the naive baseline of concatenating all tool schemas plus full history:

scenario              naive tokens   cw tokens   reduction
-------------------   ------------   ---------   ---------
tiny_payload          1,704          267         84.3%
mixed_payload         2,277          497         78.2%
short_conversation    1,946          496         74.5%
stress_conversation   17,482         6,651       62.0%
large_catalog         2,767          1,514       45.3%
long_conversation     4,365          2,548       41.6%

Now the part the README does not hide: routing recall degrades as the catalog grows. On the bundled TF-IDF backend, recall@5 is 0.56 at 50 tools but drops to about 0.15 at 1,000 tools — noise items compete with true matches. The fix is documented: for catalogs past ~200 tools, switch the scorer to bm25 (a core dep) or an embedding backend via Router(scorer_backend=...). The token savings are environment-independent and deterministic; the recall numbers are an honest signal that retrieval quality, not prompt size, becomes the bottleneck at large scale.

Honest limitations

This is Alpha in package metadata. The core context and routing APIs are deterministic and heavily tested (the repo reports 1,150+ passing tests across the pipeline, routing, firewall, adapters, and sensitivity enforcement), but newer surfaces — gateway commands, some framework adapters, optional extras — are explicitly flagged as experimental and may change before 1.0.

And it is genuinely not for everyone. Skip it if:

  • You have five tools and tiny outputs — building a routing graph and beam search adds latency you don't need.
  • You run single-shot Q&A — there's no history to compact and no later phases to budget.
  • You want LLM-driven, non-deterministic routing — contextweaver tie-breaks by sorted ID on purpose; if you want a model to reason about tool choice freely, tool_choice="auto" or LangGraph fits better.

The smallest scenario where it clearly pays for itself is roughly a 40-tool catalog and a 50-turn transcript.

Takeaways

  • Bound the model-visible surface. Five ChoiceCards beat a hundred schemas on both cost and accuracy.
  • Firewall every tool result. Big payloads go out-of-band as addressable artifacts; small ones no-op cleanly.
  • Budget per phase. Route, call, interpret, and answer each get their own token ceiling — no blunt global truncation.
  • Determinism is a feature. Same inputs, same prompt, every run — which makes debugging and regression testing actually tractable.
  • Read pack.stats. It's the receipt for what the engine kept and dropped, and why.

contextweaver is the context layer of the Weaver Stack — it routes a catalog and firewalls results, then hands the selected capability to ChainWeaver for execution, with agent-kernel owning the boundary and agentfence applying output guardrails. Every piece works standalone; contextweaver has no hard dependency on any sibling.

FAQ

What is contextweaver and what problem does it solve? contextweaver is a deterministic context gateway that sits in front of your MCP servers to kill "context rot" — the failure mode where connecting many MCP servers floods the prompt with hundreds of tool schemas and giant tool results, driving up tokens, latency, and even hurting accuracy because the right tool gets buried among distractors.
