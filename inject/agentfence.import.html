AgentFence: A Local MCP Policy Firewall for AI Agent Tool Calls

Your coding agent can now touch your filesystem, GitHub, and shell. AgentFence is a single Go binary that decides allow, deny, or ask before any of it runs — no cloud, no telemetry.

Your AI coding agent just asked to write a file. The path is .env. The content is OPENAI_API_KEY=sk-....

Did you mean for that to happen? Probably not. But the agent doesn't know that, and the MCP server on the other end will happily oblige. Modern tool-using agents get handed real capabilities — your filesystem, your GitHub account, your shell, your internal APIs — and the gap between "the model decided to call a tool" and "the tool ran" is usually zero. A prompt injection buried in a fetched web page, a hallucinated rm -rf, a secret copied into a log: all of it executes at machine speed, and your only record afterward is whatever the agent felt like printing.

AgentFence is a small answer to that problem: a local policy gate that sits in front of tool calls and decides whether each one is allowed, denied, or sent for approval — before anything executes. It's a single Go binary you run yourself. No SaaS, no account, no telemetry. The policy is yours and the audit log is yours.

This is a practical walkthrough: the problem, how the gate works, and a five-minute hands-on with code copied straight from the repo. AgentFence is early-stage (v0.7.0 at the time of writing), so I'll also be specific about what it does not do.

The problem, concretely

Three failure modes show up over and over with tool-capable agents:

  • Destructive actions done too fast. github.delete_repo, a force-push, rm -rf — irreversible calls fire before a human can react.
  • Prompt injection turning into tool calls. Untrusted content (a web page, an issue body, a file) carries instructions that steer the agent into a call it shouldn't make.
  • Secrets leaking into logs. Arguments containing API keys and tokens get written verbatim into whatever trace the tooling keeps.

There's a wave of hosted "agent security gateways" aiming at this. The catch is the same one you always get with a SaaS control plane: your tool calls — which contain your code, your prompts, and sometimes your secrets — now flow through someone else's cloud. For a security control specifically, that's an awkward trade. AgentFence takes the opposite stance: run the gate locally, keep everything on your machine, and make the whole thing auditable by design.

The idea in one paragraph

Put a policy engine between the agent and its tools. Every tool call is matched against a YAML policy that yields one of three decisions — allow, deny, or ask — plus a human-readable reason. The default is deny: you opt calls in, not out. Constraints can narrow a rule further (which paths, which repos, which URLs, which executables). Sensitive-looking values are redacted before anything is written. And every decision lands in an append-only, hash-chained audit log you can verify offline. That's the whole model — boring on purpose, which is what you want from a security control.

How it works

AgentFence runs in two shapes from the same binary:

  • Batch check mode evaluates a JSONL stream of recorded tool calls offline. It never executes anything — it produces decisions and audit events. This is what you wire into CI.
  • MCP proxy mode (agentfence proxy) is a live gate. It spawns your MCP server as a subprocess, relays JSON-RPC over stdio in both directions, and intercepts every tools/call. Allowed calls are forwarded; denied calls get a JSON-RPC error response (code -32001, "blocked by AgentFence policy") and the server never sees them; ask calls hit an approver. There's also proxy-http for remote MCP servers reached over streamable HTTP/SSE.

Where AgentFence sits: between the agent and its tools, stamping every call and writing an audit trail you own.

Evaluation is deterministic. Rule lookup goes exact match → named group → wildcard → default, and the first match wins. Then any constraints on that rule run (paths, argument values, URLs, shell commands, durable-memory writes), with deny patterns evaluated before allow patterns so a denial short-circuits immediately. Because the default is deny, a call that matches nothing is blocked rather than waved through.

One honest architectural point, stated plainly in the repo: AgentFence is not a sandbox. It enforces policy before a call executes; it does not contain a call that has already been forwarded. Pair it with OS-level isolation for defense in depth.

Try it in 5 minutes

You need Go 1.22+ (or grab a pre-built binary for Linux/macOS/Windows from the Releases page). Build from source:

go build -o agentfence ./cmd/agentfence

Run the built-in demo — it evaluates four bundled tool calls against the example policy:

./agentfence demo

You get a decision line plus a redacted JSONL audit event for each call. The interesting ones:

call_002 filesystem.write -> deny (path ".env" denied by pattern ".env")
{"schema_version":"2","seq":2,"call_id":"call_002","tool":"filesystem.write","decision":"deny","reason":"path \".env\" denied by pattern \".env\"","arguments":{"content":"OPENAI_[REDACTED:generic_secret_assignment]","path":".env"}}
call_004 github.delete_repo -> deny (tool github.delete_repo matched explicit policy rule)

Two things to notice. The .env write is denied by a path constraint, and the secret in its content was redacted to [REDACTED:generic_secret_assignment] before it ever touched the log. The delete_repo call is denied outright. (Timestamps and session IDs are real at runtime — UUIDs and RFC3339Nano values — shown abbreviated here.)

Now run the same check yourself against the example files:

./agentfence check --policy examples/policy.yaml --call examples/tool-calls.jsonl

The input is just JSONL — one tool call per line, exactly the shape an agent emits:

{"id":"call_001","tool":"filesystem.read","arguments":{"path":"README.md"}}
{"id":"call_002","tool":"filesystem.write","arguments":{"path":".env","content":"OPENAI_API_KEY=sk-demo-secret"}}
{"id":"call_003","tool":"github.create_issue","arguments":{"repo":"dgenio/agentfence","title":"Demo issue","body":"Created by an agent"}}
{"id":"call_004","tool":"github.delete_repo","arguments":{"repo":"dgenio/agentfence"}}

The policy

Here's a minimal policy. Note the deny-by-default and the three decision types:

version: "0.1"
defaults:
  decision: deny
tools:
  filesystem.read:
    decision: allow
  filesystem.write:
    decision: ask
  github.create_issue:
    decision: ask
  github.delete_repo:
    decision: deny

Rule lookup precedence: the first match wins, and unmatched calls fall through to a deny-by-default.

The real examples/policy.yaml goes further with constraints. A few examples straight from the repo:

tools:
  filesystem.write:
    decision: ask
    constraints:
      paths:
        allow: ["./src/**", "./docs/**"]
        deny: [".github/workflows/**", ".env", "**/secrets/**"]

  github.create_issue:
    decision: ask
    constraints:
      args:
        repo:
          allow: ["dgenio/*", "myorg/*"]
          deny: ["dgenio/private-*"]

  shell.exec:
    decision: ask
    constraints:
      command:
        allow_executables: ["git", "go", "make", "npm"]
        deny_patterns: ["rm -rf*", "curl * | bash"]

redaction:
  enabled: true
  patterns:
    - name: openai_api_key
      regex: "sk-[A-Za-z0-9_-]{20,}"
    - name: github_token
      regex: "gh[pousr]_[A-Za-z0-9_]{20,}"

Validate, explain, test

Three commands make policies maintainable instead of magic. validate lints the file and rejects unknown fields:

./agentfence validate --policy examples/policy.yaml

explain shows you why a single call got its decision, with a trace:

./agentfence explain --policy examples/policy.yaml \
  --tool filesystem.write --args '{"path":".env"}'
# decision: deny
# reason:   path ".env" denied by pattern ".env"

And policy test runs declarative fixtures so your policy has a regression suite, the same way your code does.

Wire it into a live agent

For Claude Code or any MCP client, wrap the server command with agentfence proxy in your config:

{
  "mcpServers": {
    "filesystem": {
      "command": "agentfence",
      "args": [
        "proxy",
        "--policy", "/Users/you/.config/agentfence/policy.yaml",
        "--audit-log", "/Users/you/.local/share/agentfence/audit.jsonl",
        "--",
        "npx", "-y", "@modelcontextprotocol/server-filesystem", "/Users/you/work"
      ]
    }
  }
}

From the agent's side it still looks like one MCP server on stdio. AgentFence is invisible until a decision is enforced. On an ask rule with a terminal attached, you get a y/N prompt (read from /dev/tty, never stdin, so it can't collide with the JSON-RPC channel); in CI you pass --no-interactive to fail closed.

The audit trail you own

Write a tamper-evident log and verify it offline:

./agentfence check --policy examples/policy.yaml --call examples/tool-calls.jsonl \
  --audit-log audit.jsonl --tamper-evident
./agentfence audit verify --log audit.jsonl
# OK: 4 event(s) verified

Each event is hash-chained to the previous one, so modifying or deleting an event in the middle is detectable — verify exits non-zero and names the event number that failed. v0.7.0 adds Ed25519 signing (--sign-key) to authenticate the writer, and publishable anchors so a third party can detect silent whole-log deletion. Useful one-liner once you have a log:

jq -c 'select(.decision=="deny")' audit.jsonl

Honest limitations

The repo is refreshingly upfront, so I will be too:

  • It is not a sandbox. Enforcement happens before forwarding; it cannot claw back a call it already allowed. Use OS isolation alongside it.
  • Shell-command constraints are best-effort, not a security boundary. Metacharacters (|, ;, &&, $()) can slip past allow_executables and deny_patterns. The docs say so in bold.
  • Tamper-evidence is detection, not prevention. Hash chaining lets you detect tampering after the fact; it doesn't stop someone with write access from rewriting the whole log. Whole-log deletion is only catchable if you also ship to an append-only --audit-sink or publish an anchor.
  • Taint tracking is a string-provenance heuristic, not full information-flow analysis, and only runs in the proxy paths where tool outputs are observed.
  • It's v0.7.0 — early. That said, "early" here means actively developed and tested: ~27 test files with 340+ test and fuzz functions, fuzz coverage on the parser/glob/redaction code, and a CI gate (make ci) that runs the policy fixtures.

Takeaways

  • AgentFence is a local, no-cloud policy gate for AI agent tool calls — allow, deny, or ask, with a human-readable reason on every decision.
  • It runs as an offline batch checker for CI and a live MCP proxy for real agents, from one Go binary.
  • Deny-by-default plus path/arg/URL/shell/memory constraints means you opt calls in, not out.
  • Redaction scrubs secret-looking values before logging; tamper-evident, optionally signed, audit logs give you a trail you can verify offline.
  • It's honest about being early-stage and not a sandbox — treat it as one layer of defense in depth, not the whole wall.

AgentFence is part of the broader Weaver Stack of small, composable agent-safety tools: agent-kernel (an in-process firewall compiled into your agent), vibeguard (dev-time guardrails), and lessonweaver (which turns recurring deny/ask traces into reviewed policy lessons — AgentFence can export its audit decisions straight into it). Each works standalone; AgentFence has no hard dependency on any of them.
