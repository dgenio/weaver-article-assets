VibeGuard: An Offline Pre-Merge Safety Gate for AI Code

A deterministic, no-API-key gate that catches the secrets, packaging leaks, and "trust all certs" footprints that AI-generated diffs sneak past review.

Your AI assistant just opened a 600-line pull request. The tests are green. The description is tidy. Somewhere in there is a line that says verify=False, added "to make the staging cert error go away."

Do you catch it in review? Honestly — at 600 lines, on a Friday, probably not.

AI coding tools made it cheap to generate code. They did not make it cheap to trust code. The bottleneck moved from typing to reviewing, and human review does not scale to the volume of diffs an agent can produce. So a new failure mode shows up in pull requests: not exotic zero-days, but boring, high-blast-radius mistakes that a tired reviewer waves through.

This is a concrete, practical walkthrough of VibeGuard — an open-source gate built specifically for that failure mode. It is early-stage (more on that below), but it already runs offline, in seconds, with no API key. You can try it on a deliberately-broken example in about five minutes.

The problem, concretely

Here are the categories of mistake that slip through when you accept large AI-generated diffs without line-by-line scrutiny:

  • Secrets in new files. API keys, GitHub tokens, database URLs — committed inside an AI-generated config file you skimmed.
  • Packaging leaks. Source maps, .env files, and test fixtures that end up published to npm or PyPI because the agent "fixed" your files array.
  • Security control bypasses. The classic: an auth check commented out, shell=True added, SSL verification turned off "to make things work."
  • AI footprints. Placeholder credentials, # TODO: implement real auth, trust all certificates, "temporary" bypasses that become permanent.
  • Risky changes with no tests. A huge diff touching auth, crypto, and database writes — and zero new test coverage.

Your existing tools have real gaps here. Secret scanners like gitleaks are great against git history but won't stop a .env that's about to be committed now. SAST tools like Semgrep do deep dataflow but tend to miss "AI footprint" patterns and packaging-manifest hygiene. Dependency scanners catch known CVEs but not a git-URL dependency an agent slipped in. VibeGuard is built to sit in that gap, not to replace any of them.

The idea, in one paragraph

VibeGuard is a deterministic, offline pre-merge safety gate. It reads your diff (or your whole tree), runs a fixed set of pattern-based rules, prints the findings, and — in gate mode — exits non-zero if anything meets your severity threshold. No LLM in the loop, no network calls, no telemetry, no API key. Because it's deterministic, the same diff always produces the same findings, which is exactly what you want from something you'll enforce in CI. Think of it as the check between "AI generated this diff" and "this diff reaches production."

Where VibeGuard sits: between an AI-generated diff and the merge button, as a deterministic offline gate that either passes or exits non-zero.

How it works

VibeGuard ships 18 deterministic rules. Each rule emits stable finding IDs you can target, suppress, or re-severity. A sampling:

Rule              Catches
---------------   -----------------------------------------------------------------------------------------------------
`secrets`         AWS/GitHub/OpenAI/Slack/Stripe tokens, private keys, DB URLs with credentials, committed `.env` files
`packaging`       npm/PyPI manifests that would publish `.env`, tests, or source maps
`dependencies`    git/URL/path deps, typosquatting heuristics, unpinned versions
`risky_diff`      Changes to auth, crypto, eval/exec, shell, network, DB writes, CORS, JWT, certs
`auth`            Commented-out auth, JWT `alg=none`, `verify=False`, allow-all middlewares
`ai_footprints`   Placeholder creds, trust-all-certs, CORS wildcards, temporary bypasses, hallucinated TODOs
`tests`           Source files changed without any corresponding test changes

There's a deliberate design choice worth calling out. The risky_diff rule is a risk signal, not a vulnerability claim. It says "human, please look at this," not "this is exploitable." That's why it's MEDIUM and informational by default — VibeGuard separates "this is an area to review" from "this blocks the merge," so the gate stays trustworthy and you don't train your team to ignore it.

Try it in 5 minutes

The repo ships deliberately-vulnerable example packages with fake secrets, so you can see meaningful findings without pointing the tool at anything sensitive.

pip install vibeguard-gate
git clone https://github.com/dgenio/vibeguard
cd vibeguard
vibeguard scan --path examples/vulnerable-node-package

That scan produces 16 findings across five files — 1 critical, 8 high, 5 medium, 1 low, 1 info — spanning a leaked GitHub token, a committed .env, source-map and packaging leaks, a URL dependency, and several AI footprints. Here's the actual table:

                                    VibeGuard Findings
 Sev        Rule           Path               Title

 HIGH       secrets        .env               Sensitive file committed: .env
 HIGH       sourcemaps     package.json       package.json 'files' includes source maps
 HIGH       packaging      package.json       npm package may publish Environment files (.env)
 HIGH       dependencies   package.json       URL/git/path dependency: axios
 HIGH       ai_footprints  src/server.js:14   AI footprint: Security disabled in code
 HIGH       ai_footprints  src/server.js:31   AI footprint: Trust-all certificates
 HIGH       auth           src/server.js:16   Auth: Auth bypass TODO/FIXME/HACK comment

 ...
  Scanned 5 file(s)  •  16 finding(s)  |  critical: 1  high: 8  medium: 5  low: 1  info: 1

Now the part that matters for CI. scan is informational and always exits 0. gate runs the same rules but exits 1 when findings hit your threshold:

vibeguard gate --path examples/vulnerable-node-package --fail-on high
echo "exit: $?"   # exit: 1

A realistic PR, not a kitchen sink

The kitchen-sink demo is convincing, but real PRs are subtler. The repo's examples/pr-scenarios/ directory has self-contained scenarios that each look like a normal pull request. Take scenario 01 — "fix flaky integration tests against staging." The staging endpoint has a self-signed cert, so the agent disabled verification everywhere:

# PR: "fix flaky integration tests against staging"
import requests
import urllib3

# AI: silence the warnings so the CI logs stay clean
urllib3.disable_warnings(urllib3.exceptions.InsecureRequestWarning)

def fetch_billing(url: str) -> dict:
    # verify=False added to make the staging cert error go away
    resp = requests.get(url, verify=False, timeout=30)
    return resp.json()

Scan it:

vibeguard scan --path examples/pr-scenarios/01-tls-verify-disabled

VibeGuard flags two high findings that block the merge — AUTH-VERIFY-FALSE (TLS validation disabled) and AI-TRUSTALLCERTS (trust-all-certificates footprint) — plus medium risky_diff signals noting the certificate-trust change and the outbound network call on billing logic. A green test suite would never have caught this; that's the entire point. The fix the scenario recommends is to trust the staging CA explicitly (verify="/path/to/staging-ca.pem") rather than turning verification off in shipped code.

Want to know why a finding fired and how to fix it? Every finding ID has a built-in explanation:

vibeguard explain SEC-ENV

scan vs gate: same rules, different exit code. scan is always informational (exit 0); gate fails CI when findings meet the threshold.

Wiring it into a PR

One command writes a complete GitHub Actions workflow — SARIF code-scanning annotations, a self-updating summary comment, and a gate that fails on high-severity findings:

vibeguard setup github-actions

Or keep it minimal and copy/paste:

name: VibeGuard
on: [pull_request]
jobs:
  vibeguard:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install vibeguard-gate
      - run: vibeguard gate --diff --fail-on high

The --diff flag scans only changed files, which keeps gate time roughly constant regardless of repo size.

Is it fast and accurate enough?

VibeGuard ships a reproducible benchmark (make bench, make bench-scenarios, make bench-precision). The headline numbers, from the repo's own report:

  • Throughput is roughly linear at ~100 files/second on the maintainer's machine — about 0.5 s for a 50-file repo, ~4.7 s for 500 files. (These are illustrative single-machine timings; run it on your own hardware for a number you can trust. For PRs, --diff keeps it near-constant.)
  • Precision 1.00 on the labeled corpus — no blocking false positives — with detection recall 1.00 at the actionable HIGH/CRITICAL tier. The clean fixtures produce zero blocking findings, and a CI test fails the build if any benign case ever starts producing an actionable finding. That regression guard is the reason the precision number stays meaningful.

Low false positives are not a nice-to-have for a gate. The first time VibeGuard cries wolf on a clean PR, your team starts adding # noqa-style overrides and the gate is dead. The zero-blocking-false-positive baseline is the feature.

Honest limitations

VibeGuard is alpha (version 0.10.0, Development Status :: 3 - Alpha), backed by a sizable test suite — roughly 869 test functions across 68 files — but young. Set expectations accordingly:

  • It is not a SAST replacement. No deep dataflow analysis. Keep Semgrep, CodeQL, or Bandit for that.
  • It is not a full secret-scanner. It won't sweep your git history for already-leaked credentials — use gitleaks or truffleHog for that.
  • It is not a CVE / dependency-vulnerability scanner. Keep Dependabot, pip-audit, or Snyk.
  • It is pattern-based, so it's bounded by its rules. It is a fast first line of defense, explicitly designed to complement the heavyweight tools and to never replace human review.
  • The benchmark scenario set is small and curated — it demonstrates detection on representative mistakes, not statistical coverage of all vulnerabilities.

The maintainers are upfront about this: the project's stated direction is "trust before breadth." A gate you can enforce in CI has to be boring and predictable first, comprehensive second.

Takeaways

  • AI-generated diffs introduce a specific, repeatable class of mistakes — leaked secrets, packaging leaks, disabled security controls, AI footprints — that slip past green tests and tired reviewers.
  • VibeGuard is a deterministic, offline gate for exactly that class: no LLM, no network, no API key, single-digit seconds.
  • scan is informational (exit 0); gate fails CI (exit 1) when findings hit your --fail-on threshold. Use --diff in PRs.
  • It reports precision 1.00 with zero blocking false positives on its labeled corpus, guarded by a CI regression test — the property that makes a gate survivable in practice.
  • It is alpha and deliberately narrow: a first line of defense that complements Semgrep, gitleaks, and Dependabot rather than replacing any of them.

VibeGuard is the dev-time, pre-merge guardrail in the broader Weaver Stack — alongside agentfence (runtime edge), lessonweaver (turns repeated findings into reviewed lessons), and weaver-spec (the shared contracts). They share serialized contracts but never each other's runtime; VibeGuard works completely standalone.
