A tiny, zero-dependency Python repo that shows the difference between an agent with 9 tools and no guardrails and one that shortlists, runs deterministic flows, checks capabilities, and writes an audit trace.

Picture the incident review. A support automation you shipped last quarter just issued a $149 refund to a customer at 2 a.m. Nobody approved it. Nobody can tell you which policy it consulted, what data it read, or why it decided the refund was warranted. The agent had access to the refund tool, the request looked plausible, so it acted. There is no trace.

This is the default failure mode of enterprise tool-using agents. You wire an LLM to your real tools — CRM, billing, email, ticketing — hand it a broad prompt, and hope. It usually works in the demo. Then it does something irreversible in production and you have no answer for the auditor.

enterprise-agent-control-plane is a small, runnable Python repo that makes this contrast concrete. It ships two versions of the same "Customer Operations Agent" — an unsafe baseline and a governed control plane — and lets you run both in one command to see the difference in behavior, not in slideware. It is Apache-2.0, has zero runtime dependencies, and is honest about what it is: a reference architecture and learning repo, not production security software.

The problem: broad access, no record

The repo models a realistic Customer Ops scenario. There are nine fake in-memory enterprise tools, each tagged with a risk level in catalog.py:

ChoiceCard("crm.search_customer", "low",  "Find customer profile and account status."),
ChoiceCard("billing.get_invoice",  "low",  "Read invoice details."),
ChoiceCard("billing.issue_refund", "high", "Issue a monetary refund to a customer."),
ChoiceCard("support.search_tickets","low", "Find support history."),
ChoiceCard("support.create_task",  "medium","Create follow-up tasks for operations."),
ChoiceCard("email.draft_reply",    "low",  "Draft customer-facing response text."),
ChoiceCard("email.send_reply",     "high", "Send customer-facing response."),
ChoiceCard("docs.search_policy",   "low",  "Find internal policy references."),
ChoiceCard("audit.export_case",    "medium","Export case evidence for review."),

The baseline agent gets all nine, with nothing between it and the high-risk ones. Its entire run_case is three tool calls and a raw dump:

def run_case(self, customer_id, invoice_id):
    customer = self.tools["crm.search_customer"](customer_id)
    invoice  = self.tools["billing.get_invoice"](invoice_id)
    refund_attempt = self.tools["billing.issue_refund"](invoice_id, 149.0, "customer requested")
    return {
        "mode": "unsafe",
        "visible_tools": [c.capability for c in self.catalog],   # all 9
        "raw_outputs": {"customer": customer, "invoice": invoice,
                        "refund_attempt": refund_attempt},
    }

Run it and the refund comes back "status": "issued". No shortlist, no flow, no policy check, no audit trail. That is the entire point — this is what "just give the agent the tools" looks like when you stop hand-waving and write it down.

The idea, in one paragraph

Don't connect intent directly to action. Put a thin control plane in between that does four things in order: shortlist the catalog down to the few capabilities a request actually needs; run the business path as a deterministic flow with no LLM improvising between steps; require a capability token plus a policy decision before any risky action fires; and write an audit trace of every step regardless of outcome. The agent still does the reasoning — it just can't reach past the rails to do something irreversible without leaving a record and clearing a gate.

How it works

The governed agent runs the same case through those four boundaries. Each one is a small adapter that stands in for a fuller sibling library in the dgenio "Weaver" stack — the repo is explicit that these are adapters, not the production libraries.

What the four control-plane modules map to in the dgenio Weaver stack. Each is a small adapter in this repo standing in for a fuller sibling library.

1. Shortlist (contextweaver-style). shortlist_capabilities() in catalog.py scores each tool against the query and returns the top matches, so the agent reasons over a handful of relevant tools instead of the whole catalog:

shortlist = shortlist_capabilities("refund customer reply", self.catalog)

2. Deterministic flow (ChainWeaver-style). The refund path is a fixed sequence, defined as data in flows/refund_review.flow.yaml and executed by ChainWeaverExecutor with no model call between steps:

"refund_review": FlowDefinition(flow_id="refund_review", steps=[
    FlowStep("lookup_customer", "crm.search_customer"),
    FlowStep("lookup_invoice",  "billing.get_invoice"),
    FlowStep("check_policy",     "docs.search_policy"),
    FlowStep("draft_reply",      "email.draft_reply"),
])

Notice what is not in that flow: billing.issue_refund. The deterministic path drafts a reply; it never silently issues money.

3. Capability token + policy gate (agent-kernel + AgentFence-style). Before the refund can fire, the agent must hold a capability token and pass the local policy. The policy returns allow, deny, or ask — and high-risk actions return ask:

token = CapabilityToken(principal=principal, capability="billing.issue_refund")
can_invoke = check_capability(token, "billing.issue_refund")
if not can_invoke:
    decision = PolicyDecision("deny", "Capability token invalid.")
else:
    decision = self.policy.evaluate("billing.issue_refund", principal)
outcome = "blocked" if decision.decision != "allow" else "allowed"

The nuance worth noticing: the token is valid, but the policy still says ask, so the action ends up blocked pending approval. Both the rule and the gate have to agree before anything irreversible happens.

4. Audit trace. Every step — shortlist, flow execution, policy check — is recorded with a UTC timestamp, actor, action, and outcome, then written to disk.

The two paths a Customer Operations Agent can take for the same request. The baseline wires the LLM straight to every tool and acts. The governed path inserts four boundaries between intent and action.

The repo rounds out the stack with a skdr-eval-style offline lane (evals.py compares a keyword_router_v1 scoring 0.72 against a bounded_router_v2 scoring 0.88) and a lessonweaver-style stub for staging reviewed lessons from failures. These let you compare routing changes offline before they touch production.

Try it in 5 minutes

You need Python 3.10+. There are no third-party dependencies to install — the package installs itself in editable mode.

git clone https://github.com/dgenio/enterprise-agent-control-plane
cd enterprise-agent-control-plane

make setup   # python -m pip install -e .
make demo    # python -m apps.demo_cli.main
make test    # 13 unit tests

make demo runs both agents over the same case (customer_id="C-100", invoice_id="INV-9") and prints the contrast. Here is the actual output:

Same request, two outcomes. The baseline issues the refund and tells you nothing. The governed path narrows nine tools to a relevant shortlist, runs a four-step deterministic flow, blocks the refund with a human-readable reason, and leaves a JSON trace behind. Open traces/governed_run_C-100_INV-9.json and you'll see each step logged with a timestamp — including the policy.check event with "outcome": "blocked".

The test suite is fast and real: make test runs 13 tests in well under a second, covering policy decisions, capability-token validation, the flow runner, the catalog shortlist, and error paths.

Honest limitations

The repo is refreshingly clear about its maturity, and so will I be. This is version 0.1.0, an early-stage reference architecture.

  • It is not production security software. The README says so directly. The policy engine is a handful of if statements; the capability "token" is a dataclass with a string comparison; the tools are fake in-memory dicts. Treat it as a teaching scaffold, not a hardening guide.
  • The dgenio libraries are adapters, not integrations. contextweaver, ChainWeaver, agent-kernel, AgentFence, skdr-eval, and lessonweaver appear here as small local stubs that demonstrate the pattern. Wiring in the real libraries is on the roadmap, not done.
  • There is no LLM in the loop yet. The "agent" is deterministic Python. That's a feature for a clear demo, but it means prompt-injection, tool-selection accuracy, and model non-determinism — the hard parts — aren't exercised here.
  • The scenario is a single happy-ish path. One customer, one invoice, one refund. The eval scores (0.72 vs 0.88) come from a tiny hand-written stub, not a real benchmark.

None of that undercuts the value. The point of a reference architecture is to make a good shape legible and runnable before you spend a sprint building the real thing. This one does that in a few hundred lines you can read in a sitting.

Takeaways

  • Put a control plane between intent and action. The four boundaries — shortlist, deterministic flow, capability+policy gate, audit trace — are the reusable shape, independent of any library.
  • Deterministic flows beat free-form tool use for known business paths. If the refund-review steps are always the same four, encode them as data and stop letting the model improvise.
  • "Ask" is a first-class decision. The most useful policy outcome here isn't allow or deny — it's ask, which blocks the irreversible action and routes it to a human while still recording the attempt.
  • Audit everything, including the blocks. A refund that didn't happen is exactly the event your auditor wants to see logged.
  • A small honest demo is worth more than a big vague claim. Zero dependencies, 13 passing tests, one make demo. You can verify every sentence above in five minutes.

Try it and tell the maintainer what's missing

If you're building enterprise agents, clone it, run make demo, and read governed_agent.py end to end — it's under 70 lines and it's the whole thesis. Then open the sibling Weaver-stack repos to see where each adapter is headed: contextweaver (bounded context routing), ChainWeaver (deterministic flows), agent-kernel (capability tokens), AgentFence (policy gates), skdr-eval (offline evaluation), and lessonweaver (reviewed lessons).

If the pattern is useful, star the repo — github.com/dgenio/enterprise-agent-control-plane — and file an issue with the control you wish it modeled next. Reference architectures get better when people argue with them.
