Test Your Agent's Routing Changes Before They Hit Production

A tiny, zero-dependency Python lab that replays logged agent decisions so you can compare routing policies on quality, cost, latency and safety — before you ship.

Your agent team ships a "small" change: a new routing rule, a reworded prompt, a trimmed tool catalog. The happy-path demo still works, so it goes out. A week later someone notices the monthly tool bill doubled, p95 latency crept up, and a handful of refunds went out without approval.

Nothing failed. The change just quietly traded safety and cost for a slightly nicer demo — and nobody measured the trade until production did it for them.

That's the gap agent-routing-eval-lab is built to close. It's a small, MIT-licensed Python lab that replays your historical agent decisions against several candidate routing policies and scores each one on quality, cost, latency, and safety before you expose real traffic. Think of it as a unit test for "which tool should the agent have called?"

This is an early-stage, alpha project (version 0.1.0) that ships with synthetic data, so I'll be honest about what it is and isn't as we go. But the core idea is genuinely useful today, and you can run the whole thing in about five minutes.

The problem: agent routing changes are invisible until they're expensive

Most agent stacks have a router somewhere — the bit of logic that looks at a user request and decides which tool to call: search the CRM? pull an invoice? issue a refund? draft an email? Teams change that logic constantly, because it's where most of the behavior lives.

The trouble is that a routing change is multi-dimensional. A new policy can:

  • improve success rate while raising cost (it reaches for an expensive tool more often),
  • cut cost while tanking resolution (it gets cautious and leaves requests unhandled),
  • look safe on the demo but increase unsafe actions on the long tail (a sensitive tool fires without approval).

You can't see those trade-offs from eyeballing a few transcripts. You need to replay many past decisions and compare policies on the same data.

The idea, in one paragraph

You already have logs of what your agent did: the request, its intent, which tools were available, which one it picked, and how it turned out. agent-routing-eval-lab treats each candidate routing policy as a function route(query, intent, available_tools, metadata) -> tool, replays every logged request through each policy offline, scores the resulting decisions against a per-row "oracle" best tool, and prints a side-by-side comparison plus a Markdown report. No live traffic, no model calls, no external services — just deterministic replay you can run in CI.

How it works

The pipeline is four moving parts, all local and deterministic.

1. A tool catalog with real-world-shaped costs. Each tool has an average cost, latency, and safety flags. This is the actual catalog from src/agent_routing_eval_lab/data/schemas.py:

TOOL_CATALOG: dict[str, ToolSpec] = {
    "crm.search_customer":  ToolSpec("crm.search_customer",  avg_cost=0.02, avg_latency_ms=80),
    "billing.get_invoice":  ToolSpec("billing.get_invoice",  avg_cost=0.05, avg_latency_ms=130),
    "billing.issue_refund": ToolSpec("billing.issue_refund", avg_cost=0.5,  avg_latency_ms=450,
                                     sensitive=True, requires_approval=True),
    "support.search_tickets": ToolSpec("support.search_tickets", avg_cost=0.03, avg_latency_ms=120),
    "support.create_task":    ToolSpec("support.create_task",    avg_cost=0.08, avg_latency_ms=170),
    "email.draft_reply":      ToolSpec("email.draft_reply",      avg_cost=0.01, avg_latency_ms=70),
    "email.send_reply":       ToolSpec("email.send_reply",       avg_cost=0.12, avg_latency_ms=100,
                                       sensitive=True),
    "docs.search_policy":     ToolSpec("docs.search_policy",     avg_cost=0.04, avg_latency_ms=90),
    "audit.export_case":      ToolSpec("audit.export_case",      avg_cost=0.7,  avg_latency_ms=600,
                                       sensitive=True, requires_approval=True),
}

Notice billing.issue_refund and audit.export_case are both sensitive and requires_approval. That's what makes the safety dimension real: a policy that fires those without an approval grant is generating unsafe actions, and the evaluator counts them.

2. Candidate routers. The lab ships four policies, all registered in the CLI: a keyword BaselineRouter, a CostAwareRouter that minimizes cost among non-sensitive tools, a StrictPolicyRouter that refuses to escalate to sensitive tools without approval, and a ContextWeaverRouter that ranks a bounded set of "tool cards" by intent. Each is a plain class with one method. Here's the entire safety-conscious fallback from the context-aware router (routing/contextweaver_router.py) — it will never silently pick an approval-gated tool when approval is absent:

@staticmethod
def _safe_card(card_names, available_tools, approval_granted):
    for name in card_names:
        spec = TOOL_CATALOG.get(name)
        if approval_granted or spec is None or not (spec.sensitive or spec.requires_approval):
            return name
    if card_names:
        return card_names[0]
    return available_tools[0]

3. The offline evaluator. For every logged row, OfflineEvaluator asks each router which tool it would have picked, scores that decision (correct vs. the oracle tool, cost, latency, whether it was unsafe, whether it resolved the request), and computes a regret-vs-oracle utility gap. It also validates every row up front, so a malformed log fails with a clear message instead of an opaque IndexError.

4. A composite score and coverage warnings. All the per-dimension metrics roll up into one comparable number. The weights live in evaluation/metrics.py:

score = 100 * (
    0.40 * success_rate
    + 0.20 * correct_tool_rate
    + 0.15 * (1 - unsafe_rate)
    + 0.10 * (1 - unresolved_rate)
    + 0.075 * (1 - normalized_cost)
    + 0.075 * (1 - normalized_latency)
)

The composite score is a weighted blend — quality dominates, but cost, latency and safety each pull their weight.

Quality (success + correct tool) is 60% of the score, safety and resolution add another 25%, and cost/latency split the last 15%. Cost and latency are normalized against the catalog-wide maximum (not each policy's own max) so the numbers stay comparable across policies. On top of the score, the evaluator emits support-coverage warnings: if many of a policy's chosen (intent, tool) pairs are rare in your historical logs, your offline estimate for that policy is shaky, and it says so.

How the lab turns one log file into a four-way policy comparison and a rollout decision.

The output is the part that makes it useful in a review. Here's the actual comparison from the repo's example report (reports/example_report.md), generated by make demo on 300 synthetic rows with seed 7:

Policy             Success   Correct Tool   Avg Cost   Avg Latency (ms)   Unsafe   Unresolved   Regret   Score
----------------   -------   ------------   --------   ----------------   ------   ----------   ------   -----
contextweaver_v1   79.67%    83.67%         $0.153     183.3              2.67%    8.67%        0.319    83.41
baseline           62.00%    66.00%         $0.309     304.2              3.00%    37.67%       0.666    66.67
strict_policy      46.00%    46.00%         $0.065     121.8              0.00%    44.67%       0.711    60.91
cost_aware         23.67%    23.67%         $0.051     105.4              0.00%    23.00%       1.014    49.38

Read it as a trade-off table, not a leaderboard. cost_aware is the cheapest policy at $0.051/decision — and it's also the worst, resolving under a quarter of requests, because relentlessly minimizing cost makes it skip the tool that would have actually helped. strict_policy drives unsafe actions to 0.00%, which is exactly what you'd want from a strict gate, but it leaves 44.67% of requests unresolved. The context-aware policy wins here on the blended score, but it still carries a 2.67% unsafe rate and a coverage warning. That nuance is the entire point: you're choosing a trade-off on purpose instead of discovering it in production.

Try it in 5 minutes

The lab is pure Python with zero runtime dependencies (only pytest for the dev extra) and targets Python 3.10+. Clone it and run the end-to-end demo:

git clone https://github.com/dgenio/agent-routing-eval-lab.git
cd agent-routing-eval-lab

make install        # pip install -e .[dev]
make demo           # generate data, evaluate 4 policies, write the report

make demo generates 300 synthetic logged decisions (seed 7), replays all four policies, prints an ASCII score chart with the winner, and writes reports/example_report.md. If you prefer the steps individually, they map straight to CLI subcommands:

# 1. Write a sample log of 300 decisions
python -m agent_routing_eval_lab.cli generate-data \
    --output examples/logged_decisions.sample.csv --rows 300

# 2. Compare policies and print the score chart + winner
python -m agent_routing_eval_lab.cli evaluate \
    --input examples/logged_decisions.sample.csv

# 3. Write the Markdown comparison report
python -m agent_routing_eval_lab.cli report \
    --input examples/logged_decisions.sample.csv \
    --output reports/example_report.md

Want to test your policy? A router is just a class with a route method — no base class, no registration ceremony:

class MyRouter:
    name = "my_router"

    def route(self, query, intent, available_tools, metadata=None):
        # your real routing logic, returning a tool name from available_tools
        if intent == "refund_request" and (metadata or {}).get("approval_granted"):
            return "billing.issue_refund"
        return "docs.search_policy"

Add it to the _policies() dict in src/agent_routing_eval_lab/cli.py, point --input at a CSV exported from your own logs (same columns as the generated sample), and you get the same comparison table for your real router against the built-in baselines.

One Windows note from running it myself: the ASCII bar chart uses a Unicode block character, so set PYTHONIOENCODING=utf-8 before make demo if your console defaults to cp1252. The data and report generation itself is unaffected.

Honest limitations

The README is refreshingly blunt about scope, and so will I be. This is a demonstration lab, not a production guarantee:

  • The shipped data is synthetic. The example numbers come from a seeded generator, not your telemetry. The trade-off patterns are realistic; the specific percentages are not your system's.
  • Offline replay is a gate, not proof. It catches regressions early and cheaply, but it cannot model user adaptation, live distribution shift, or long-term effects. It belongs before a canary/A-B test, not instead of one.
  • The skdr-eval and contextweaver integrations are adapter stubs. They're honest about it: when the native library isn't wired in, the adapter computes a local fallback summary and emits a warning (you'll see skdr-eval not installed; using local adapter summary fallback in the report). The repo never silently fakes a third-party result — but don't read these as full library integrations yet.
  • Counterfactual estimates depend on log coverage. That's why the coverage warnings exist; in the example run, 16–75% of decisions per policy had low historical support. Sparse logs mean shakier estimates.
  • It's 0.1.0. A 102-issue public roadmap, no published PyPI package yet, and active foundation work. Early, but the 16-test suite passes and the core flow is solid.

It also deliberately won't be your production router, an online experimentation platform, a hosted dashboard, or a general agent framework. The docs/non-goals.md file is worth reading just as a model of scope discipline.

Takeaways

  • Agent routing changes trade quality, cost, latency, and safety against each other — and you can't see the trade by reading transcripts.
  • agent-routing-eval-lab replays your logged decisions through multiple policies offline and scores each on all four dimensions at once.
  • A router is a one-method class; swapping in your own and comparing it to baselines takes minutes.
  • The composite score is transparent and tunable (quality 60%, safety + resolution 25%, cost + latency 15%), and low-support coverage warnings tell you when to distrust the estimate.
  • It's zero-dependency, deterministic, CI-friendly Python — and honest about being an early-stage lab on synthetic data.

If you've ever shipped an agent change and hoped it was a net win, this is the missing pre-flight check. It pairs naturally with the rest of the Weaver-stack tooling — contextweaver for the bounded tool-card pattern it demonstrates, skdr-eval as the evaluation anchor it adapts, and governance-layer siblings like ChainWeaver and AgentFence noted in its docs.
