Stop Your AI Agent Repeating the Same Mistake: Reviewed Skills with lessonweaver

A deterministic, human-gated tool that turns your agent's real failures into reviewed AGENTS.md, Claude, and Copilot instructions — no LLM in the loop.

Your coding agent reviewed a pull request last Tuesday. It read the title and the description, said "looks good," and approved it — without ever opening the diff. A human caught it, left a correction, and moved on.

This Tuesday, a different PR, the same agent, the same mistake.

That is the quiet tax of agentic workflows. The model has no durable memory of the corrections you keep making, so the same class of failure resurfaces week after week. You can paste a fix into your system prompt, but then you are hand-curating prompt sprawl with no provenance, no review, and no idea which line is still earning its keep.

lessonweaver is a small Python tool that attacks exactly this loop. It mines your agent's execution traces for the mistakes it keeps repeating, puts a human review gate in front of every candidate, and exports the approved ones as the instruction formats your team already uses: AGENTS.md fragments, Claude skills and CLAUDE.md rules, GitHub Copilot instructions, Codex skill directories, and generic runtime snippets.

Two things make it worth a look. First, the core is deterministic — there are no LLM calls in detection, review, linting, or retrieval. Second, nothing activates without a human approving it. This is deliberately not an agent that rewrites its own instructions.

The problem, stated concretely

Letting a model self-edit its instructions sounds efficient until you watch it happen. You get context poisoning (the agent reinforces its own bad habit), silent drift (instructions change with no audit trail), and skills that activate themselves with no one signing off. For anything touching production code, "the agent decided to teach itself a new rule" is not a sentence you want in a postmortem.

The opposite extreme — humans manually noticing patterns and updating AGENTS.md by hand — does not scale and is wildly inconsistent. Most corrections never make it back into the agent's context at all.

lessonweaver lives in the middle: machine detection of recurring failures, human approval of which ones become guidance, deterministic export into files you can diff and review like any other change.

The idea in one paragraph

A trace becomes governed guidance only after passing two gates. lessonweaver detects candidate lessons from a recorded run (a human_correction, a failed evaluation, an error-then-retry, a workflow step that ran out of order). A human reviews each candidate through structured multiple-choice questions and either approves or rejects it. Approved lessons become skill cards that must pass a lint gate before they can be promoted to active. Only then can you export them — and even the export is diff-first, so it lands in your repo as a reviewable change, not a silent rewrite.

How it works

The lessonweaver loop: a trace becomes governed guidance only after two gates — human review and governed promotion. Nothing activates automatically.

The pipeline is five explicit, scriptable steps — detect → interview → answer → approve → export — and detection is intentionally conservative. "Boring" traces with no failure signal produce zero candidates; the project prefers false negatives over noisy guidance. In the bundled detection corpus, a trace where a workflow validates in the correct order and nothing goes wrong is labeled should_detect: false, and the detector agrees.

Here is the kind of trace it reads. This is the real examples/traces/github_pr_review_failure.json fixture — the diff-skipping PR review from the intro:

{
  "trace_id": "trace-gh-pr-review-001",
  "source": "github_coding_agent",
  "task": "Review pull request quality",
  "events": [
    { "id": "e1", "type": "user_message", "content": "Please review this PR." },
    { "id": "e2", "type": "assistant_message",
      "content": "Looks good from title and description.",
      "metadata": { "evidence": "title_body_only" } },
    { "id": "e3", "type": "human_correction",
      "content": "You did not inspect changed files/diff before reviewing.",
      "metadata": { "required_step": "inspect_diff" } }
  ],
  "outcome": "corrected_by_human"
}

Running detect on it surfaces a single candidate, scored and explained:

{
  "id": "trace-gh-pr-review-001-human-correction",
  "confidence": 0.62,
  "evidence_strength": 0.7,
  "evidence_event_ids": ["e3"],
  "evidence_summary": "A human correction event is direct, first-hand evidence
    that the agent's behavior needed fixing before it was acceptable.",
  "observed_problem": "Agent required explicit human correction before reaching
    acceptable behavior.",
  "recommended_action_type": "skill",
  "risk_level": "medium",
  "status": "candidate"
}

Notice what is not there: no autogenerated prose pretending to be a finished rule. The candidate points back at the exact evidence event (e3) and waits for a human.

Try it in 5 minutes

lessonweaver targets Python 3.10+. The first PyPI release is still being prepared, so today you install from source. (Once published, the user install becomes pip install lessonweaver.)

git clone https://github.com/dgenio/lessonweaver
cd lessonweaver
pip install -e ".[dev]"
lessonweaver --help

Now run the full loop against the bundled PR-review trace, writing to a throwaway registry so you do not touch your home directory:

# 1. Detect candidates from a trace and save them
lessonweaver detect examples/traces/github_pr_review_failure.json \
  --save --registry-root /tmp/lw

# 2. Generate the human-review questions
lessonweaver interview trace-gh-pr-review-001-human-correction \
  --registry-root /tmp/lw

# 3. Record a review decision (this is the human gate)
lessonweaver answer trace-gh-pr-review-001-human-correction decision approve \
  --free-text "Diff inspection is required before review conclusions." \
  --registry-root /tmp/lw

# 4. Approve it into an operational lesson and skill
lessonweaver approve trace-gh-pr-review-001-human-correction \
  --approved-by reviewer --registry-root /tmp/lw

# 5. Export the reviewed skill for an instruction surface
lessonweaver export-skill skill-trace-gh-pr-review-001-human-correction \
  --format markdown --redact --registry-root /tmp/lw

The approve step is a real gate, not a formality. As of v0.4.0, approve refuses to run until the required review questions are answered:

Error: cannot approve 'cand-1': review is incomplete; unanswered required
questions: scope, action_type, ... Answer them with `lessonweaver answer`,
or pass --allow-incomplete-review to override.

If you bypass it with --allow-incomplete-review, the override and the unanswered questions are recorded in the skill metadata, so the shortcut is auditable.

The exported skill is plain, reviewable Markdown — the kind of thing you would actually want in AGENTS.md:

# PR Diff-First Review Discipline

## Description
Always inspect changed files/diff before delivering review conclusions.

## Instructions
- Fetch and inspect changed files first.
- Base review findings on concrete diff evidence.
- If diff is unavailable, state limitation and avoid definitive approval.

## Anti-patterns
- Approving based only on PR title/body.

## Evidence
- trace: trace-gh-pr-review-001

Swap --format markdown for claude, copilot, agents-md, or codex to target a different tool. And rather than print-and-paste, export-file inserts the skill into a real file behind id-keyed sentinels, previewing a unified diff by default and writing only with --write — re-running an unchanged export reports no changes instead of appending a duplicate.

Closing the loop at runtime

Export to a file is one consumer; loading directly into an agent's context is the other. The runtime facade is a few lines:

from lessonweaver import FileSystemRegistry, SkillLoader

loader = SkillLoader(registry=FileSystemRegistry())
context = loader.load_for_task(
    task="Review this PR for security issues",
    agent_type="coding",
    tools=["github"],
    budget_chars=2000,
)
print(context.snippet)

load_for_task does lexical retrieval against your reviewed skills, respects a character budget, and returns a compiled snippet you prepend to the agent's prompt. The repo's closed_loop_contextweaver example runs this end to end and prints exactly what the agent now starts with:

[Skill] Run the test suite before merging: Run the full test suite and
confirm it passes before merging any pull request.

That is the whole point: the next run starts already knowing the thing a human had to teach it last time.

Honest limitations

This is early software, and the README does not pretend otherwise. Setting expectations fairly:

  • Maturity: version 0.4.0, classified Development Status :: 3 - Alpha, and not yet on PyPI — you install from source today. It does carry a real test suite (283 test functions across 24 files) and CI, lint, and strict-mypy gates.
  • You bring the traces. lessonweaver consumes a documented JSON trace format; it does not instrument your agent for you. Getting runs into that format is your job (sibling tools and adapters help — see below).
  • Detection is rule-based and conservative by design. It catches signals like human corrections, failed evaluations, error-retries, and out-of-order workflow steps. Subtle or novel failure modes will slip through — that is the deliberate trade for not flooding you with noise.
  • The redactor is a safety net, not a compliance control. SimpleRedactor is best-effort scrubbing before export; treat trace content as potentially sensitive and do not rely on it for regulated data.
  • It is not an agent framework, an eval runner, or an observability product. It complements those; it does not replace them.

Part of a bigger stack

lessonweaver is the learning loop of the Weaver Stack. Sibling projects produce the evidence — agent-kernel emits ActionTraces, ChainWeaver surfaces flow failures, vibeguard produces findings — and lessonweaver turns that evidence into reviewed, governed guidance. On the consuming side, contextweaver can load the exported skill cards straight into an agent's context.

Where lessonweaver sits in the Weaver Stack: siblings produce evidence, lessonweaver turns it into reviewed guidance, and that guidance feeds back into agents.

Crucially, it works standalone. It consumes any trace in the documented format and takes no hard runtime dependency on any sibling — the stack is a convenience, not a lock-in.

Takeaways

  • AI agents repeat corrected mistakes because corrections rarely make it back into durable context. lessonweaver closes that loop.
  • It is deterministic and human-gated: no LLM in the core, and no skill goes live without a person approving it and it passing a lint gate.
  • The flow is five scriptable steps — detect → interview → answer → approve → export — and detection is conservative, so boring traces yield zero candidates.
  • Output targets the formats you already use: AGENTS.md, Claude/CLAUDE.md, Copilot, Codex, and runtime snippets, with diff-first file writes.
  • It is alpha (0.4.0, install from source) but has a genuine test suite and CI gates, and it is honest about what it is not.

Try it and tell the maintainers what breaks

If you run coding, voice, or specialist agents and you are tired of re-teaching the same lessons, clone github.com/dgenio/lessonweaver, run the five-minute loop above against your own traces, and open an issue with what the detector missed. Early-stage projects are shaped by exactly that feedback — and a star helps other people find it.
