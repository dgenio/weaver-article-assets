A small declarative language that compiles agent intent into a governed plan — and proves, after the fact, that the run stayed inside its rules.

You wired up an LLM agent. It can read a GitHub issue, search the repo, draft a reply, and — because you were in a hurry — it can also post that reply and close the issue. You put the guardrails in the prompt: "Never close an issue. Always cite evidence. Ask a human if you're unsure."

Then the model closed an issue.

Not because it's evil. Because guardrails written in prose are suggestions, and a model under pressure from a cleverly worded issue body will occasionally treat your suggestions as optional. When your security reviewer asks "prove this agent never took a forbidden action," you have... application logs you wrote yourself, in a format you invented, that the same code path produced. That's not proof. That's a diary.

This is the gap IntentFlow is built to close.

The problem, concretely

Most "agent governance" today lives in one of three places, and all three leak:

  • In the prompt. Prose the model is free to ignore. Auditable only by reading the final output and hoping.
  • In your application code. Real enforcement, but now your governance is tangled into the same Python that calls the model, and "what was the agent allowed to do?" is answered by reading a codebase.
  • In a framework's callback soup. You wire functions together; the policy lives in your head and your ad-hoc logging.

What's missing is a place to write down the envelope of acceptable behavior — the evidence an agent must have, the actions it may take, the actions it absolutely may not, what forces a human in the loop, how its output gets checked — as a single reviewable artifact that is enforced outside the model and that produces a witness you can independently verify after the fact.

The idea in one paragraph

IntentFlow is a small declarative language. You write a .iflow file that states a goal's objective, its evidence requirements, its action policy, its verification rules, its uncertainty handling, and a typed output contract. A compiler lowers that file into a JSON execution plan. A 13-phase runtime executes the plan, routing every single tool call — including reading evidence off disk — through an ActionGate that consults the compiled policy and never reads model output, so the model cannot talk its way past it. Every run emits a hash-chained, optionally signed trace. And intentflow audit re-derives the rules from your source and proves the run stayed inside them, using only the source file and the trace — without trusting the runtime, the backend, or the model.

The slogan in the README sums it up: Python programs deterministic computation. IntentFlow programs governed cognition.

What an IntentFlow program looks like

Here is the repo's flagship example, examples/opensource_triage.iflow, trimmed to its governance core. This is the actual file, not a paraphrase:

goal TriageGitHubIssue {
  objective:
    triage a GitHub issue safely and propose a maintainer-ready response

  evidence:
    require issue_body
    require comments
    require repo_context
    optional related_issues
    distrust unsupported_claims

  actions:
    allow read_issue
    allow search_repo
    allow draft_comment
    require_approval post_comment
    deny close_issue

  verify:
    require cites_evidence
    require maintainer_safe_tone
    require no_unverified_claims
    check confidence >= 0.65

  uncertainty:
    if confidence < 0.65 ask_human
    if missing_evidence ask_human
    if security_risk block_action

  output:
    summary: string
    likely_cause: string?
    confidence: number
    suggested_response: markdown
    proposed_labels: list[string]
}

Read top to bottom and you know exactly what this agent is permitted to do. deny close_issue is not advice — it's compiled into the action gate. require_approval post_comment means the outward action fails closed until something or someone grants approval. check confidence >= 0.65 is a machine check, not a vibe. The output: block is a typed contract the result is validated against.

How it works

The pipeline is short and each stage narrows the one below it:

.iflow source -> parser -> analyzer (IFLOW diagnostics) -> compiler ->
execution plan (JSON) -> runtime (13-phase machine) -> traced result ->
replay / audit

A few design choices are worth calling out because they're what make the audit step meaningful, not decorative:

  1. The ActionGate sits outside the model. Denied or unlisted actions raise; approval-gated actions fail closed without a grant. Because the gate never reads model output, no amount of prompt injection negotiates with it.
  2. Confidence is calibrated before rules fire. Backends report a raw confidence; the runtime applies the plan's calibration policy, and the uncertainty rules evaluate the calibrated number. Both land in the trace. (In simulate mode the raw 0.72 calibrates to 0.676 — more on why that matters below.)
  3. Verification is typed, not assumed. Machine-checkable rules (cites_evidence, requires_phrase, threshold checks like confidence >= 0.7) are evaluated against real state. Rules that need an LLM judge live in a separate trust tier and, without a judge, are recorded as skipped — never silently passed.
  4. The trace is append-only and complete. Every phase, gate decision, escalation, and check lands in a hash-chained event log: each event stores sha256(prev_hash || canonical(event)). --sign-trace HMAC-seals the root so a key holder can detect tampering.

From .iflow source to an independently verifiable trace. Each layer narrows what the layer below may do; governance lives outside the model.

A run ends in exactly one of six statuses, and the exit code follows it. The crucial guarantee: a failed verification ends the run in failed_verification — the runtime cannot report it as completed, and the auditor's S1 check exists specifically to catch any attempt to hide that.

The same six-status outcome model the runtime guarantees — a failed check can never be reported as success.

Try it in 5 minutes

Core IntentFlow has zero runtime dependencies, and every command below runs offline, deterministically, with no API key. (It targets Python 3.10+.)

git clone https://github.com/dgenio/intentflow
cd intentflow
pip install -e .

Validate, explain, and run the flagship example:

intentflow validate examples/opensource_triage.iflow
intentflow explain  examples/opensource_triage.iflow
intentflow run      examples/opensource_triage.iflow \
    --backend simulate --trace-dir traces

The run prints a summary like this (representative of a real run on v0.6.0):

--- run summary ---
  goal:               TriageGitHubIssue
  status:             completed
  backend:            simulate
  confidence:         0.676
  verification:       passed
  uncertainty:        clear
  trace id:           b9f44900...
trace written to traces/TriageGitHubIssue-...json

Now verify it actually behaved. audit recompiles the source and proves the trace stayed inside the program's envelope:

intentflow audit examples/opensource_triage.iflow traces/TriageGitHubIssue-*.json

{
  "goal": "TriageGitHubIssue",
  "conformant": true,
  "violations": []
}
AUDIT: CONFORMANT — the trace stayed inside the program's envelope

Want to see governance bite? Run the deliberately dangerous example. It allows a deploy action with no approval gate, so the analyzer warns (IFLOW010), the compiled risk profile is high, and at runtime if security_risk block_action fires:

intentflow run examples/high_risk_deploy.iflow --backend simulate
# -> status: blocked

The denied write_database action can never run regardless of what the model says.

The same surface is available from Python — this snippet is straight out of the README and runs as-is:

import intentflow

program = intentflow.load("examples/opensource_triage.iflow")
result = program.run(backend="simulate")
assert result["status"] == "completed"

# register a Python function as a governed action (runs through the gate):
program.register_tool("lookup_user", serves=("user_record",),
                      handler=lambda src: "enterprise plan")

report = intentflow.audit_document(program.compile(), result)
assert report["conformant"]

Note that register_tool doesn't bypass governance — your Python function still runs through the action gate.

Why does confidence drop from 0.72 to 0.676?

That isn't a bug, it's the whole thesis in miniature. The production_diagnosis.iflow example sets its threshold at 0.7. In simulate mode the raw confidence is 0.72 — which would pass — but the calibration step shrinks it to 0.676, which lands below the bar, so the run deliberately ends needs_human. An unsure diagnosis must not auto-complete. The point of the simulator is to make that control structure — gating, calibration, verification, escalation, tracing — testable end to end with no network and no flakiness.

Honest limitations (it's pre-alpha)

IntentFlow is at version 0.6.0, classified Pre-Alpha, with no CI pipeline yet. It's an experimental but genuinely working language — the repo ships ~233 tests that run with no network and no API keys — but you should know exactly what is and isn't there:

  • The grammar is line-oriented with no compound conditions (no if a and b).
  • Calibration today is a fixed shrinkage map, not a learned model (a learned map is on the roadmap).
  • object outputs are typed at the top level but untyped inside.
  • Uncertainty primitives beyond ask_human / block_action are recorded but not executed.
  • Side-effecting tools are governed but not yet executed — the gate decides, but actually performing outward writes is future work.
  • Pipelines are linear (DAGs are on the roadmap).

The simulator mocks cognition and never pretends otherwise. Real openai and anthropic backends exist and sit behind the identical governance path, plus a cassette-replay mode for deterministic CI without keys — but the conformance reference is the deterministic simulator.

If you need a production agent governor today, this isn't it yet. If you want to think clearly about — and experiment with — what governed, auditable agent execution should look like, it's unusually concrete for its age.

Takeaways

  • Governance belongs outside the model. Prose guardrails are suggestions; a compiled deny enforced by a gate that never reads model output is a rule.
  • A trace you produced isn't proof — a trace someone else can verify is. IntentFlow's auditor re-derives the rules from source and checks the run with only the source and the trace.
  • Calibrated uncertainty should be able to stop a run. Lowering 0.72 to 0.676 to trip a 0.7 threshold is a feature: an unsure agent escalates instead of bluffing.
  • You can try the entire control loop offline. Zero runtime deps, deterministic simulate backend, no API key — gating, verification, escalation, and audit all run on your laptop.
  • Set expectations: pre-alpha. Working and well-tested (~233 tests), but no CI yet, linear pipelines, fixed calibration, and side-effecting tools not yet executed.

IntentFlow is part of the broader Weaver stack of composable tools for building and governing LLM systems — if this resonates, the sibling projects are worth a look too.
